# Системные уровни агентных решений. Пример — обработка лидов

Когда строишь агента под **узкую бизнес-задачу**, главное — заложить правильные решения на каждом **системном уровне** (слое): на каком слое стека стоять, где это запускать, как нарезать контекст, как сообщение становится сессией. Промахнёшься на любом — будешь вечно бороться с инструментом вместо того, чтобы им пользоваться.

Сквозной пример здесь — **обработка лидов в Telegram**: в общем чате появляется лид, ты тегаешь бота и командуешь «новый лид», агент заводит запись в CRM и пишет лиду первое письмо; дальше по лиду идёт диалог (показать черновик, поправить, отправить).

Эта заметка — **карта решений** для такого класса задач (агент за чат-гейтвеем), а не введение в LLM-агентов. Механика тулов (in-process / CLI / MCP) — в [tool-packaging.md](tool-packaging.md); внутренности Claude Code и Agent SDK — в [claude-code-internals.md](claude-code-internals.md); REST/RPC — в [api-protocols.md](api-protocols.md); MCP — в [mcp-protocol.md](mcp-protocol.md). Здесь позиция архитектора, который решает, **на каком слое стоять, где это запускать и как сообщение превращается в сессию с узким контекстом**.

Конкретика по примеру Гермеса (Hermes от Nous Research) сверена с его документацией на июнь 2026 — это **быстро меняющийся open-source-проект**, перед стройкой сверяйся с актуальными доками.

## 1. Карта слоёв: API → SDK → агентный фреймворк → харнес

Под «взять Claude/Codex/Гермес» прячутся четыре разных слоя. Их путают, потому что один бинарь часто содержит несколько. Разложение по слоям, с конкретными именами по вендорам:

| Слой | Что это | Anthropic | OpenAI |
|---|---|---|---|
| **Сырой REST API** | один вызов модели по HTTP; петли нет | Claude API (`/v1/messages`) | OpenAI API |
| **Тонкий SDK** (низкоуровневый) | библиотека-обёртка над REST: вызываешь методы вместо ручного HTTP | `anthropic` | `openai` |
| **Агентный фреймворк = SDK высокого уровня** | петля + исполнение тулов + сессии, как библиотека | Claude Agent SDK | OpenAI Agents SDK |
| **Кодинг CLI/TUI харнес** | фреймворк + UI + стор сессий + интеграции, как продукт | Claude Code | Codex |

**Тонкий SDK есть под каждый язык** — одна и та же библиотека в разных рантаймах: Python (`anthropic` / `openai`), TypeScript/JS (`@anthropic-ai/sdk` / `openai`), а также Go, Java/Kotlin, Ruby, C#, PHP. **Агентный фреймворк — это тоже SDK, просто более высокого уровня**: он не «оборачивает один вызов», а даёт готовую петлю агента поверх того же API.

Сырой API — это **не агент**: один вызов, петли нет. Петлю («модель → вызов тулы → результат → модель → … → финальный ответ») добавляет агентный фреймворк. Вот эта петля и делает из вызова модели рантайм. **Тонкий SDK сам по себе — не рантайм, а транспорт-клиент** (важно для §9).

## 2. Рантайм vs харнес

**Рантайм** — исполняющее ядро: петля вокруг LLM. **Харнес** — вся оснастка вокруг рантайма, превращающая его в продукт (UI, стор сессий, авторизация провайдеров, интеграции, права, MCP, доступ к файлам).

Аналогия: рантайм = двигатель, харнес = вся машина (двигатель + кузов + панель + педали). Claude Code, Codex, Гермес, OpenClaw — это **харнесы**; внутри каждого сидит рантайм (петля) плюс обвязка.

Отсюда связь со слоями §1: **агентный фреймворк ≈ рантайм в форме библиотеки**; **харнес = тот же рантайм + оснастка как продукт**. Claude Agent SDK даёт ту же петлю и контекст-менеджмент, что и Claude Code, — программируемо (подробнее в [claude-code-internals.md](claude-code-internals.md)).

«Какой рантайм поставить за роутером» = какое ядро крутит петлю, после того как телеграм/UI/стор-сессий ты снял на себя.

## 3. Где это крутится — и два способа потреблять рантайм

Слой (§1) отвечает «на чём стоять». Независимая ось — **где исполняется петля**:

- **Библиотека в твоём процессе** — фреймворк (Agent SDK) встроен в твой код, петля крутится у тебя.
- **Сервис на твоей ВМ** — харнес запущен тобой как сервер (Гермес с его REST API), петля у тебя, но отдельным процессом, с готовым стором сессий и авторизацией.
- **Managed (на инфре провайдера)** — Anthropic/OpenAI крутят петлю у себя.

Отсюда — **два способа потреблять агентный рантайм**, и это ключ к вопросу «а есть ли у Гермеса SDK?»:

- **(a) встроить как библиотеку/SDK** (OpenAI Agents SDK, Claude Agent SDK) — петля в твоём процессе;
- **(b) звать как сервис по REST** (Hermes API-сервер, Managed Agents) — петля в отдельном сервере, ты ходишь по HTTP.

**Managed Agents** (Anthropic) — пример (b): создаёшь версионируемый конфиг агента, получаешь ID, заводишь **сессии**, ссылающиеся на него; на каждую сессию Anthropic поднимает контейнер-воркспейс и стримит события (SSE). Для лид-бота, где тулы — вызовы CRM/почты по API, контейнер-на-сессию не нужен → оверкилл. Но это важная точка отсчёта для §9: **Managed Agents — это тоже агентный фреймворк за REST**, просто хостится провайдером и заперт на него. *(Сверено с docs.claude.com.)*

Гермес — тоже (b): server-runtime, который ты зовёшь по REST. Встраиваемого «Hermes Agents SDK» (как Agents SDK) нет — см. §9.

## 4. CLI vs TUI vs headless — три слова про одно

Путаница в слове «CLI». Развязка:

- **CLI-инструмент** — программа, запускаемая командой (`codex`, `claude`). Это весь бинарь.
- У него **два режима**:
  - **интерактивный** → показывает **TUI** (Terminal User Interface): полноэкранное приложение в терминале (перерисовка, панели, стриминг, стрелки). Запускается по умолчанию, когда набрал `codex` без аргументов.
  - **headless / неинтерактивный** → `codex exec "…"`, `claude -p "…"`: без TUI, один выстрел, отдал результат и вышел.

TUI — это **одно из лиц** CLI-инструмента, не отдельный продукт. Headless нужен для автоматизации (скрипты, CI/CD, cron). Но он заточен под *один выстрел*, а не под «сервер, держащий сотни адресуемых сессий» — поэтому для лид-бота кодинг-CLI как бэкенд плохо ложится: используешь ~10% его. Как только захочешь управлять сессиями программно — перейдёшь на Agent SDK (§1), то есть на (a) из §3.

## 5. Агент vs воркфлоу — кто управляет потоком

Ось дизайна: **кто управляет потоком — твой код или модель** (формулировка из anthropic.com «Building Effective Agents»).

- **Воркфлоу** — поток жёстко в коде. Маршрут известен заранее; ЛЛМ заполняет содержимое шагов. (prompt chaining, routing, parallelization, orchestrator-workers, evaluator-optimizer.)
- **Агент** — поток решает модель в рантайме. Цикл тривиален (`while not done`); соль — *какую тулу звать* и *пора ли заканчивать* — выбор ЛЛМ на каждой итерации.

«Бесконечный цикл, где ЛЛМ сама решает — позвать тулу или выдать финальный ответ» — это и есть **агент**. Уточнение: цикл с условием выхода — финальный ответ модели **или** страховка (лимит итераций, бюджет, guardrail).

Совет статьи: **не начинай с агента.** Предсказуемый маршрут — бери воркфлоу. Для лид-бота «счастливый путь» (CRM → черновик → гейт → отправка) — **детерминированный воркфлоу** с одним человеческим гейтом; агентская гибкость нужна точечно — понять интент входящего и править черновик («сделай теплее»). Итого: **тонкая ЛЛМ на понимание/правку + детерминированный пайплайн (обычный код) на CRM+отправку**. ЛЛМ решает и пишет контент — код выполняет побочные эффекты.

## 6. Контекст: чат — шина событий, не единый контейнер

**Gateway (гейтвей)** — слой харнеса между мессенджером и рантаймом: принимает входящие и превращает их в ходы сессий, решая по своему ключу, какой сессии принадлежит сообщение (механика выбора — §7). Гермес ключует **по источнику**.

Задача: лиды идут в одну общую ленту, растущую в бесконечность, а диалог ведётся по каждому лиду отдельно. Ключевое решение именно для **этой** задачи: **общий чат нельзя делать единым контейнером на всех лидов** — он должен работать как шина событий, раздающая каждого лида в свою сессию. (Контейнер — это всегда *сессия*, а не «чат» вообще; и сам General играет двойную роль — он и шина, и собственный контейнер. Разбор — в §8, где видно, *когда* и *как* происходит дробление.) Оговорка важна: в другой задаче — один общий ассистент на чат — чат как раз *и был бы* единым контейнером, и копить в нём контекст правильно. Здесь нет: при письме лиду A модель не должна видеть переписку про B, C, D.

Наивная схема «чат = сессия» тут ломается дважды: контекст растёт в бесконечность **и** лиды перекрёстно загрязняют друг друга.

Развязка — одно предложение: **«сессия» у LLM не живой процесс. Модель stateless: на каждый ход ты шлёшь ей весь массив сообщений заново. Значит «сессия» = свёрток контекста, который ты хранишь у себя и переигрываешь модели.**

**Единица сессии — лид, а не чат.** Одна логическая сессия = один лид, материализуется по требованию из внешнего состояния. Паттерн «stateless-агент + внешняя память».

| Слой | Роль | Состояние |
|---|---|---|
| Роутер (точка входа) | на каждое сообщение: *про какого лида?* + *что за интент?*; не копит контекст | stateless по сообщению |
| Состояние по лиду | внешняя память: CRM-запись (источник истины) + статус + текущий черновик + лог писем | durable, ключ `lead_id` |
| Исполнение задачи | эфемерная сессия с **узким** контекстом: только данные лида + плейбук | живёт одну задачу |

**Откуда берётся контекст под лида.** Никогда из прокрутки ленты. Холодный старт: данные лида из сообщения + шаблоны. Follow-up: пересобирается из состояния лида (текущий черновик + CRM + пара прошлых ходов). Факты, дописанные в CRM асинхронно (обогащение на бэкенде), читаются из CRM **в момент вопроса**, а не из замороженной истории, — иначе сессия тихо устаревает и врёт.

**Три вещи, которые нельзя слипать:**

1. **Telegram reply-граф** → для корреляции «про какого лида сообщение». Один lookup `message_id → lead_id`.
2. **Состояние лида в хранилище** (вкл. лог писем) → настоящая память. «Добавь ещё письмо в цепочку» берётся отсюда, не из Telegram.
3. **Контекст LLM на этот ход** → собирается из (2) под текущую задачу.

Маппинг `message_id → lead_id` — крошечная индекс-таблица (два ID на строку), это указатель, а не контекст. Индексируй обе стороны (сообщения бота и обработанные команды) — тогда реплай на твоё же «ок» резолвится в один lookup.

## 7. Как гейтвей выбирает сессию (механика)

**Сессия определяется ДО того, как побежит LLM, и тулы её уже не перевыбирают.** Гейтвей (отдельный процесс) на каждое входящее:

1. получил апдейт от Telegram;
2. **вычислил ключ сессии из метаданных источника** (`chat_id` / `chat_id+user_id` / `chat_id+thread_id`) — детерминированно, без LLM и без анализа текста;
3. достал/создал сессию по этому ключу в сторе;
4. добавил сообщение как user-message в контекст сессии;
5. **запустил петлю агента** — тут впервые бегут LLM и тулы;
6. ответ ушёл обратно в Telegram.

Шаг 2 раньше шага 5. Тулы (шаг 5) могут *что-то сделать* (создать топик, записать в CRM), но **не могут переназначить сессию текущего сообщения** — она зафиксирована на шаге 2.

**У Гермеса две входные двери (платформы), и сессию они выбирают по-разному:**

| Дверь | Кто выбирает сессию |
|---|---|
| **Telegram-платформа** | гейтвей выводит из метаданных источника (ты не управляешь без своего кода) |
| **HTTP API-платформа** (`gateway/platforms/api_server.py`) | **вызывающий называет сессию явно** (session id в запросе) |

Бэкенд-API к рантайму — это и есть **HTTP API-платформа** (та, что в §9 сравнивается с Managed Agents): одна из платформ гейтвея, наравне с Telegram/Discord/…; гейтвей — оркестратор, который роутит между агентами/сессиями. *(Сверено: репо `NousResearch/hermes-agent`, `gateway/platforms`.)*

**Как бот понимает, что обращаются к нему** (группа): прямой `/command`, реплай на его сообщение, `@упоминание`, регекс из `mention_patterns`, либо `require_mention: false` (реагирует на все видимые сообщения). *(Сверено с docs Гермеса.)*

## 8. Топики = нативная сессия на лида (и где живёт твой код)

**Главное: проблему «собрать историю по лиду в одну сессию» Гермес решает топиками — нативно.** Дока дословно: каждый форум-топик = изолированная сессия с ключом `agent:main:telegram:dm:{chat_id}:{thread_id}`, и «each topic gets its own conversation history, model state, tool execution, and session ID». Группа **без** топиков — одна сессия на чат. *(Сверено с docs Гермеса.)*

Значит **`thread_id` — это и есть твой per-lead-ключ, и Гермес роутит по нему сам.** «lead_id как ключ сессии» тебе не нужен — нужен топик на лида.

**Поправка к ранней версии этой заметки:** «кастомного gateway-адаптера, который роутит по `lead_id`» как поддерживаемого механизма **нет**. Плагин-система Гермеса (плагины из `~/.hermes/plugins/`, `.hermes/plugins/`, pip entry points) даёт **кастомные тулы + lifecycle-хуки**, но не кастомные платформы и не управление маршрутизацией сессий. *(Сверено: это явный пробел в доке.)*

**Динамическое создание топика под лида — НЕ встроенная фича.** Дока: топики создаются **на старте из конфига** («On gateway startup, Hermes calls `createForumTopic` for each topic that doesn't have a `thread_id` yet»); шипнутого рантайм-тула «создай топик по ходу диалога» нет. Чтобы делать топик-на-лида динамически, ты добавляешь **кастомный плагин-тул**, который сам дёргает Bot API (`createForumTopic` + `sendMessage`). *(Это плагин-тул = твой код; что он может ходить в Bot API под токеном бота — вывод из «плагины дают кастомные тулы»; документированного примера исходящих Telegram-сендов из плагина не нашёл — проверь в репо.)*

**Где живёт твой код — два места** (третьего, «адаптера маршрутизации», нет):

- **Плагин-тул внутри агента** — для варианта с топиками. Маршрутизацию (telegram → gateway → agent) не трогаешь; добавляешь тулу (создать топик, lookup/запись в CRM) + системный промпт «на „новый лид“ сделай то-то».
- **Внешний сервис перед Гермесом** — владеет ботом сам, резолвит `lead_id`, зовёт Гермес через **HTTP API**, называя сессию явно. Здесь «гейтвей» — это твой сервис, Telegram-платформа Гермеса не используется.

### Bootstrap «новый лид» и желаемый UX

UX «пишу „новый лид“ в General → в новом треде сразу появляется черновик с вопросом „отправляем?“» — **достижим.** Ключ: **черновик не требует, чтобы сессия топика уже стартовала** — бот просто постит в тред.

1. «новый лид» приходит в General → гейтвей кладёт в **General-сессию** (метаданные, §7);
2. агент в General-сессии (у него есть данные лида) **сам пишет черновик**;
3. **плагин-тул**: `createForumTopic` (тред T) → запись в CRM (с черновиком) → `sendMessage(thread_id=T, черновик + «Отправляем такое письмо?»)`;
4. черновик **сразу виден в новом треде** — его постит бот; `session_T` для этого не нужна;
5. `session_T` стартует, когда **ты** ответишь в треде («ок» / «сделай теплее»). Чтобы тот ход знал черновик — он подтягивается из **состояния лида (CRM)**, куда сохранён на шаге 3 (паттерн §6).

Тула не *перевыбрала* сессию первого сообщения (оно осталось в General) — она **создала новую сессию-приёмник (топик) для будущих ходов** и запушила в неё черновик.

### General — это и шина, и контейнер

Тонкость, видная именно здесь (и почему «чат не контейнер» из §6 — упрощение): **General одновременно шина событий и собственный контейнер.** Шина — потому что это точка входа, откуда лиды раздаются в топики. Контейнер — потому что у самого General своя сессия (`chat_id` без `thread_id`), и она копит всё, что в General пишут: команды, bootstrap-ходы, случайные «привет, как жизнь». Лидовый контекст уходит в топики (узкие контейнеры), а General остаётся **catch-all-контейнером**, не ограниченным по лиду, — и тоже растёт в бесконечность, просто медленнее.

Поэтому General надо **вести как отдельный контейнер**, иначе болезнь «бесконечная лента» вернётся в нём самом:

- держи General **near-stateless под диспетчеризацию** — для «нового лида» прошлая история General не нужна (данные в самом сообщении + reply);
- периодически сбрасывай (`/reset` сбрасывает сессию текущего топика, включая General — *сверено с доками*) или полагайся на компакцию;
- если *хочешь* вести в General и общий разговор («напомни про того лида») — это осознанный выбор, и тогда General сознательно работает контейнером тоже.

### Условная логика перед созданием треда

Делается системным промптом + тулами (естественное место для агентской гибкости, §5):

1. «новый лид» → агент зовёт тул `lookup_lead(контакт)` (запрос в CRM);
2. **нет в CRM** → плагин-тул создаёт топик + черновик (выше);
3. **есть в CRM** → агент отвечает в General: «Контакт уже есть (статус X). Точно новый лид / создавать тред?» и ждёт подтверждения.

Ветвление оркестрирует ЛЛМ по промпту; тулы делают CRM-lookup и создание топика.

## 9. Три ценности Гермеса — и почему этого нет у Anthropic и OpenAI

Гермес даёт связку, которой нет в одном продукте у Anthropic/OpenAI. Это **харнес, который**:

1. **подписочный и провайдер-агностичный** — встроенный OAuth для ChatGPT, Anthropic и других; один рантайм поверх разных провайдеров;
2. **self-hosted** — запущен тобой на твоей ВМ;
3. **за REST API** — причём **OpenAI-совместимым** (`/v1/chat/completions`, `/v1/responses`) с опциональной непрерывностью сессии. *(Сверено с hermes-agent.nousresearch.com.)*

Плюс готовый **Telegram-гейтвей**, нативно дробящий лиды по сессиям через треды (§8).

### Два хопа авторизации (важно не путать)

```
[твой код + openai SDK]  --auth#1-->  [Гермес на ВМ]  --auth#2-->  [OpenAI / Anthropic]
   (локальный токен Гермеса)                        (подписочный OAuth, server-side)
```

- **Форма протокола ≠ что за ней.** API Гермеса OpenAI-*совместим по форме*, но за этим конвертом сидит **весь агент** (петля + тулы + память), а не сырая модель: шлёшь один user-message — получаешь целый ход агента.
- **Клиент = тонкий OpenAI SDK** — это **транспорт-труба, а не фреймворк и не рантайм**. Рантайм — Гермес на сервере. Раз REST OpenAI-совместимый, отдельного проприетарного SDK не нужно: ставишь `openai` SDK, `base_url` → твоя ВМ, `api_key` → локальный токен Гермеса.
- **Авторизацию в SDK не «заменяли».** Тонкий SDK auth-агностичен: вешает bearer-токен на запрос к `base_url`. Он аутентифицирует только хоп **клиент → Гермес**. Подписочный OAuth — это **следующий хоп**, внутри Гермеса, server-side; SDK его не видит и не трогает. OpenAI-совместимая граница развязывает две авторизации.

### Есть ли «Hermes SDK» как альтернатива OpenAI Agents SDK

Нет. Из §3: рантайм потребляют либо **(a) встроив фреймворк** (Agents SDK / Claude Agent SDK — петля в твоём процессе), либо **(b) зовя сервер по REST**. Гермес — только (b). Встраиваемого «Hermes Agents SDK» нет. Поэтому альтернатива «звать Гермес по REST» — это не «Hermes SDK», а **встроить *другой* фреймворк и крутить петлю самому** (вариант A из §11).

### Честное уточнение и «почему этого нет у лабораторий»

«Агентный фреймворк за REST» сам по себе — не уникум: **Managed Agents — ровно он** (§3). Уникальна **комбинация и место**: у Managed Agents хостинг и запертость на Anthropic + биллинг по ключу; у Гермеса — self-hosted + любой провайдер + подписка.

Почему этого нет у Anthropic и OpenAI — **бизнес-стимулы, не техника**:

- self-hosted провайдер-агностичный рантайм каннибализирует их продуктовый слой (Claude Code / Codex / Managed Agents) и развязывает тебя от вендора;
- подписка для автоматизации подрывает потокенный биллинг;
- провайдер-агностичность = увод трафика к конкуренту.

Это **не «ни у кого нет»** — это «нет у лабораторий». У третьей стороны конфликта интересов нет, поэтому категорию занимают сторонние харнесы: Гермес от Nous и **OpenClaw**.

## 10. Подписка vs API-ключ: что реально разрешено

«OpenAI разрешает работать на подписке, Anthropic нет» — **directionally верно, но разница не в “можно/нельзя”, а в enforcement** (насколько жёстко ловят). Факты сверены с первоисточниками (внизу).

**Anthropic** — два слоя, не путать:

- **Raw OAuth-токены подписки (Free/Pro/Max) в стороннем инструменте → запрещено.** Нарушение Consumer ToS, **технически блокируется** (client attestation принимает только настоящий бинарь Claude Code). Под раздачу попали OpenClaw, OpenCode, Cline — и поначалу даже собственный Agent SDK Anthropic.
- **Санкционированный путь = месячный кредит Agent SDK.** После отката подписчикам дали кредит на свои/сторонние приложения на Agent SDK: Pro — $20/мес, Max 5x — $100, Max 20x — $200. Per-user, не переносится; кончился — платишь по API-тарифу или запросы встают. Именно это отражает строка Гермеса «Anthropic OAuth: Required Extra Usage Credits to Use Subscription».

**OpenAI** — «Sign in with ChatGPT» поддержан для Codex под **интерактив**; для программного/headless OpenAI **рекомендует API-ключ** («We recommend API key authentication for programmatic Codex CLI workflows, such as CI/CD jobs»). ChatGPT-подписка «not intended for programmatic use», сторонние тулзы на её OAuth — не endorsed, но **жёсткого технического блока, как у Anthropic, нет**.

| | Подписка в *их* инструменте (интерактив) | Подписка в *твоём* бэкенде (программно) | Enforcement |
|---|---|---|---|
| **OpenAI** | да (Codex) | не endorsed; рекомендуют API-ключ | мягко (ToS) |
| **Anthropic** | да (Claude Code) | только кредит Agent SDK ($20/$100/$200), raw-токены — бан | **жёстко** (attestation) |

**«Могу ли я сам написать такую авторизацию? Что мешает, если Гермес смог?»** Технически — да, OAuth это просто код. Мешает **сам провайдер, намеренно**: у Anthropic самописный клиент с подписочными токенами будет отклонён (attestation) и это бан-риск — «бесплатных Max-токенов для бота» не получить, **и Гермес этого тоже не делает** (его Anthropic-путь — лимитированный кредит). У OpenAI самописный ChatGPT-OAuth заведётся легче (слабый enforcement), но упрётся в fair-use-лимиты под интерактив. Чистый путь без серости — просто API-ключ.

⚠️ Зона турбулентности: запреты были в феврале/апреле 2026, потом смягчение. Апдейт от 15 июня: **планировавшиеся изменения этой политики (ограничения на использование подписки в сторонних приложениях / правки кредитной схемы) приостановлены — пока действует прежний порядок.** **Point-in-time — сверяйся с актуальным ToS перед тем, как строить.**

## 11. Как это собрать: развилки

**Рантайм — два варианта:**

| | A: свой рантайм (Agent SDK, встроенный) | B: Гермес как рантайм (сервер по REST) |
|---|---|---|
| Петля | библиотека в твоём процессе (`tool_runner` крутит за тебя) | сервер Гермеса (готова) |
| Где крутится | твой код | твоя ВМ |
| Биллинг | API-ключ, потокенно | подписка или ключ |
| Стор сессий | твой (твоя БД) | Гермес (SQLite) |
| Контроль / чистота по ToS | высокий, масштабируемо | подписка дешевле, но лимиты + серость + хрупкость |

Внутри **B** — как нарезать сессии по лидам:

- **Топики (нативно)** — `thread_id` = per-lead-сессия из коробки. Динамическое создание топика — плагин-тул (§8). Минимум кода, привычный групповой UX.
- **Одна лента + внешний сервис через HTTP API** — твой сервис резолвит `lead_id` и сам называет сессию. Больше кода, полный контроль, не зависишь от Telegram-модели Гермеса.

**Развилка по зрелости:**

- **Прототип / личное / умеренный объём, подписка дешевле** → B (топики §8). Едешь на подписке, рантайм не пишешь.
- **Продакшн / масштаб / автоматический поток** → A + API-ключ ради предсказуемости, лимитов и чистоты по ToS.

**Несущий принцип:** роутер + состояние + CRM не зависят от рантайма и провайдера → **авторизация это сменная деталь**. Отвалится подписка (ужесточение enforcement, лимиты, бан-риск) — перетыкаешь на API-ключ или другого провайдера без переписывания бота. Проектируй так, чтобы смена auth была переключателем, а не переездом.
