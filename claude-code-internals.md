# Claude Code: как это работает изнутри

Заметки из сессии исследования Claude Code, Agent SDK и мультиагентных систем.

---

## 1. Claude Agent SDK — что это

Claude Agent SDK — фреймворк от Anthropic для построения агентных AI-приложений. Даёт "из коробки":
- Определение субагентов (`AgentDefinition`) с отдельными промптами, моделями, наборами инструментов
- Оркестрацию — главный агент делегирует задачи субагентам через `Task`
- Встроенные инструменты: `Read`, `Write`, `Bash`, `Glob`, `Grep`, `WebSearch`, `WebFetch` и др.
- Подключение MCP-серверов (внешних сервисов)
- Стриминг ответов

Доступен на **Python** (`claude-agent-sdk`) и **TypeScript** (`@anthropic-ai/claude-agent-sdk`).

## 2. Связь Claude Code и Agent SDK

Claude Code — **не потребитель** Agent SDK, а наоборот: **Agent SDK использует Claude Code как движок**.

Доказательство из бинарника Claude Code:
```js
if (process.env.CLAUDE_AGENT_SDK_VERSION)
    T.push(`agent-sdk/${process.env.CLAUDE_AGENT_SDK_VERSION}`);
```

Когда SDK запускается, он поднимает Claude Code как рантайм и передаёт ему переменные окружения.

**Архитектура:**
```
Твой код (Python/TypeScript)
  └── Agent SDK (библиотека-обёртка)
        └── Claude Code (ядро/рантайм — скомпилированный бинарник)
              └── Anthropic API
```

Из документации Anthropic: *"The Claude Agent SDK exposes the same battle-tested infrastructure powering Claude Code as a programmable library."*

## 3. Мультиагентность в Agent SDK (на примере L7)

Проект L7 из курса демонстрирует паттерн оркестратора с тремя субагентами:

| Субагент | Модель | Инструменты | Задача |
|---|---|---|---|
| docs_researcher | Haiku | WebSearch, WebFetch | Официальная документация |
| repo_analyzer | Haiku | WebSearch, Bash | Анализ репозиториев |
| web_researcher | Haiku | WebSearch, WebFetch | Статьи и комьюнити |

Главный агент (Sonnet) координирует их. Субагенты — **не отдельные процессы**, а отдельные **API-вызовы** к Claude с разными промптами. Всё работает в одном Python-процессе.

Субагенты запускаются **параллельно**, когда задачи независимы (это задаётся в промпте главного агента).

### Пример: команда write → review → test

Agent SDK позволяет собрать "команду" из специализированных агентов:
```python
agents = {
    "code_writer": AgentDefinition(
        prompt="You are a Python developer...",
        tools=["Write"], model="sonnet",
    ),
    "code_reviewer": AgentDefinition(
        prompt="You are a senior code reviewer...",
        tools=["Read", "Glob"], model="sonnet",
    ),
    "code_tester": AgentDefinition(
        prompt="You are a QA engineer...",
        tools=["Read", "Write", "Bash"], model="sonnet",
    ),
}
```
Главный агент-оркестратор вызывает их **последовательно** (каждый шаг зависит от предыдущего), передавая результат одного на вход другому. Субагенты не видят друг друга напрямую — весь обмен через оркестратора.

## 4. Субагенты внутри Claude Code CLI

Claude Code (CLI) имеет свой набор встроенных агентов:

| Агент | Назначение |
|---|---|
| Explore | Исследование кодовой базы |
| Plan | Архитектурное планирование |
| Bash | Автономное выполнение цепочки команд |
| general-purpose | Универсальный — поиск, исследование, многоходовые задачи |
| claude-code-guide | Ответы на вопросы про Claude Code/SDK/API |
| statusline-setup | Настройка статусной строки |

Это **инфраструктурные** агенты, а не ролевые (нет writer/reviewer/tester). Claude Code — "универсальный солдат", который делает всё сам в одном контексте.

Создать произвольного агента с кастомной ролью Claude Code **не может** — только использовать предопределённые типы. Но **general-purpose** достаточно гибок: ему можно описать почти любую задачу в промпте.

### Кто решает, запускать ли субагент?

**Claude Code решает сам**, на основе правил из системного промпта:
- Задача потребует больше ~3 запросов поиска → субагент (Explore)
- Несколько независимых подзадач → параллельные субагенты
- Простой поиск файла/функции → тула напрямую, без субагента

**Но пользователь может перевесить:** явная просьба "сделай в субагенте" или "сделай сам" — приоритетнее внутренних правил. Также формулировка влияет: "сравни A, B и C" скорее спровоцирует три параллельных агента, чем "посмотри A, потом B, потом C".

### Task (субагент) vs тула (инструмент)

- **Тула** (Read, Bash, Grep...): Claude сам выполняет одно действие, результат остаётся в его контексте
- **Task**: запускает отдельного агента с **отдельным контекстом**. Агент работает автономно, возвращает компактный результат

Одно и то же действие может быть и тулой, и агентом:
- `Bash` (тула): одна команда, результат в моём контексте
- `Task → Bash` (агент): цепочка команд автономно, мне — только итог

### Когда субагент оправдан, а когда нет

**Да, субагент:**
- Задачи **независимы** друг от друга (параллелизация)
- Задача **объёмная** — много промежуточных данных (защита контекста)
- Нужен **другой фокус** — свежий взгляд без контекста автора
- Нужна **другая модель** — дешёвый Haiku для рутины, дорогой Sonnet для синтеза

**Нет, не субагент:**
- Задача **простая** — один-два вызова инструментов
- Важна **непрерывность** контекста (write→review→fix в одном потоке)
- Передача контекста между агентами **дороже**, чем сама задача

**Быстрый тест:** если результат можно описать одним абзацем — субагент оправдан. Если нужны все детали для следующего шага — лучше сделать самому.

## 5. Три режима работы в Claude Code UI

| Режим | Суть |
|---|---|
| Plan | Запрещено редактировать файлы. Только чтение, исследование, планирование |
| Ask Before Edits | Можно редактировать, но спрашивает перед каждым изменением |
| Edit Automatically | Редактирует без спроса |

Plan Mode — это прежде всего **режим разрешений** ("только чтение"). Можно использовать как страховку, даже если не планируешь — просто чтобы гарантировать, что файлы не будут изменены.

**Не путать** с агентом Plan (субагент для архитектурного планирования в отдельном контексте).

## 6. CLAUDE.md — пользовательские инструкции

Файлы `CLAUDE.md` позволяют влиять на поведение Claude Code:
- `~/.claude/CLAUDE.md` — глобальные инструкции (для всех проектов)
- `проект/CLAUDE.md` — проектные инструкции

Из системного промпта: *"These instructions OVERRIDE any default behavior and you MUST follow them exactly as written."*

CLAUDE.md **приоритетнее** дефолтного поведения. Но не может перезаписать ядро безопасности.

Системный промпт напрямую прочитать как файл нельзя — он не лежит на диске. Но Claude Code может цитировать его содержимое по запросу.

## 7. Extended Thinking — как Claude "думает"

Thinking-блок (сворачиваемый в UI) — внутренний черновик перед ответом.

**Ключевой момент:** это не отдельная система, а тот же процесс генерации текста. Claude — языковая модель, у которой нет невербального мышления. Вся "мысль" — это генерация токенов.

Разговорные конструкции ("actually", "wait", "let me reconsider") — **не мусор**, а функциональные управляющие сигналы:
- Каждый сгенерированный токен становится контекстом для следующего
- "Actually" сдвигает вероятности последующих токенов в сторону пересмотра
- Без таких сигналов модель с большей вероятностью продолжит ошибочную линию рассуждений

Thinking — одновременно output (генерируется) и input (читается как контекст для продолжения).

## 8. Cursor vs Claude Code

Cursor — форк VS Code со своей реализацией. Работает с несколькими LLM-провайдерами через сырые HTTP API. Глубоко интегрирован с редактором (LSP, inline-подсказки).

Claude Code / Agent SDK — фреймворк для агентных приложений, привязан к Claude. Не про IDE-интеграцию, а про оркестрацию агентов с инструментами.

Общее: оба дают модели доступ к файлам и терминалу. Архитектурно — разные вещи.

## 9. Почему в Claude Code нет ролевых агентов (writer/reviewer/tester)?

Осознанное дизайн-решение. Для типичного цикла write→review→fix→test один агент с полным контекстом **эффективнее**:
- Помнит зачем писал код, какие были требования
- Нет потерь при передаче между агентами
- Дешевле — один вызов API вместо нескольких

Отдельные ролевые агенты лучше когда:
- Нужна **изоляция перспектив** — ревьюер не знает "почему автор так написал" и не оправдывает решения
- Нужно **масштабирование** — 10 ревью параллельно
- Есть **кастомные workflow** под конкретную команду/проект

Для кастомных ролей — используй Agent SDK и определяй агентов сам.
