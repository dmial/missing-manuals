# Протоколы API: HTTP, REST, RPC

Заметки о том, как устроено взаимодействие клиент-сервер: от транспортного уровня (HTTP) через архитектурные стили (REST) до вызова удалённых функций (RPC).

---

## 1. HTTP — транспорт

HTTP — это транспортный протокол. Он определяет, **как** клиент и сервер обмениваются сообщениями: формат запросов/ответов, заголовки, коды состояния, методы. REST и RPC — это разные **соглашения** о том, как использовать HTTP.

Аналогия:

- **HTTP** — дорога
- **REST** — правило «грузовики по правой полосе, легковые по левой»
- **JSON-RPC поверх HTTP** — другое правило: «все по одной полосе, но на борту написано кто ты»

### 1.1. Текстовый или бинарный?

Гибридный. **Заголовки — текст, тело — что угодно:**

```http
POST /upload HTTP/1.1              ← текст
Host: example.com                  ← текст
Content-Type: image/jpeg           ← текст (описывает бинарное тело)
Content-Length: 482901             ← текст
                                   ← пустая строка = конец заголовков
ÿØÿà..JFIF.....ÿÛ.C...          ← БИНАРНЫЕ БАЙТЫ JPEG
```

Примеры бинарных тел по HTTP: скачивание файлов (`application/octet-stream`), видео (`video/mp4`), gRPC (`application/grpc`), PDF. Когда YouTube стримит видео — это сотни HTTP GET-запросов, каждый из которых возвращает маленький бинарный сегмент (2-10 секунд видео в нужном качестве).

> HTTP/2 и HTTP/3 и заголовки передают в бинарном формате (сжатые), но логически это тот же HTTP.

### 1.2. Методы HTTP

9 стандартных методов (RFC 9110):

| Метод | Для чего | CRUD |
|---|---|---|
| **GET** | Получить ресурс | Read |
| **POST** | Создать / отправить данные | Create |
| **PUT** | Заменить ресурс целиком | Update (полный) |
| **PATCH** | Частично изменить | Update (частичный) |
| **DELETE** | Удалить | Delete |
| **HEAD** | Как GET, но без тела (только заголовки) | — |
| **OPTIONS** | Узнать возможности сервера (CORS preflight) | — |
| **TRACE** | Эхо для отладки | — |
| **CONNECT** | Установить туннель (HTTPS через прокси) | — |

На практике 95% — GET и POST. REST добавляет PUT/PATCH/DELETE. TRACE и CONNECT ты, скорее всего, никогда не вызовешь сам.

### 1.3. SSE — Server-Sent Events

SSE — это часть спецификации HTML (WHATWG). Простой механизм: сервер держит HTTP-соединение открытым и шлёт текстовые события.

Как выглядит в сниффере:

```http
HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive

data: {"jsonrpc":"2.0","id":1,"result":{"content":[{"type":"text","text":"часть 1"}]}}

data: {"jsonrpc":"2.0","id":2,"result":{"content":[{"type":"text","text":"часть 2"}]}}
```

Каждое сообщение — `data: ...`, разделение — пустая строка. SSE — **однонаправленный** (сервер → клиент), в отличие от WebSocket. Используется для стриминга ChatGPT/Claude, нотификаций в реальном времени, биржевых котировок, CI/CD логов.

### 1.4. SSE vs WebSocket

| | SSE | WebSocket |
|---|---|---|
| Направление | Сервер → клиент | Обе стороны |
| Протокол | Остаётся HTTP на всём протяжении | HTTP только для рукопожатия, потом переключается на бинарный протокол WebSocket |
| Формат | Текст (`data: ...`) | Бинарные фреймы |
| Пример | MCP, стриминг LLM-ответов | Онлайн-игры, чаты, трейдинг |

WebSocket начинается как HTTP (`GET` + заголовок `Upgrade: websocket`), сервер отвечает `101 Switching Protocols`, и дальше это уже **не HTTP** — бинарные фреймы в обе стороны.

### 1.5. base64: бинарные данные в текстовом протоколе

base64 — способ записать бинарные данные (байты JPEG/PNG) в виде ASCII-текста. Каждые 3 байта → 4 символа (из алфавита `A-Z`, `a-z`, `0-9`, `+`, `/`). Размер +33%.

Зачем: JSON не умеет хранить бинарные данные. Это стандартный костыль для текстовых протоколов. В HTTP без JSON-обёртки есть альтернативы:

- **Отдельный URL** — загрузить файл на CDN, передать ссылку (так работают Slack, Telegram)
- **multipart/form-data** — текст и бинарные данные рядом в одном запросе, разделённые boundary-строкой (так работает `<input type="file">` в браузере)

## 2. REST — архитектурный стиль

REST (Representational State Transfer) — **не протокол и не набор методов**, а архитектурный стиль. Его описал Рой Филдинг в диссертации 2000 года.

### 2.1. Почему так называется

Ключевое — **Representational State Transfer** (передача представления состояния):

- **Resource** — любая сущность (пользователь, заказ, статья) идентифицируется по URI (`/users/42`)
- **Representation** — клиент получает не сам ресурс, а его **представление** (JSON, XML, HTML — одна и та же сущность может выглядеть по-разному)
- **State Transfer** — каждый запрос клиента **передаёт серверу всё нужное состояние** для обработки. Сервер не хранит сессию между запросами

Название буквально: «клиент и сервер обмениваются представлениями состояния ресурсов».

### 2.2. Шесть ограничений REST

REST определяет не методы, а **архитектурные ограничения**:

1. **Client-Server** — разделение ответственности
2. **Stateless** — каждый запрос самодостаточен, сервер не помнит предыдущие
3. **Cacheable** — ответы явно помечаются как кешируемые или нет
4. **Uniform Interface** — единообразный интерфейс (URI для ресурсов, CRUD через стандартные методы, self-descriptive messages, HATEOAS)
5. **Layered System** — клиент не знает, общается ли он напрямую с сервером или через прокси/балансировщик
6. **Code on Demand** (опционально) — сервер может отправить клиенту исполняемый код

### 2.3. REST мыслит ресурсами

REST оперирует **существительными** — ресурсами, к которым применяются стандартные HTTP-методы:

```
GET    /emails/42        — получить письмо
POST   /emails           — отправить письмо
DELETE /emails/42        — удалить письмо
```

### 2.4. Распространённое заблуждение

То, что большинство называет «REST API» — на самом деле **HTTP API с JSON**. Настоящий REST по Филдингу включает HATEOAS (ответ содержит ссылки на связанные действия), а этого почти никто не делает. Филдинг сам неоднократно говорил, что большинство «REST API» таковыми не являются.

GET/POST/PUT/DELETE — это HTTP-методы, которые REST *использует*, но REST — это набор архитектурных принципов поверх HTTP, а не сами методы.

## 3. RPC — вызов удалённых функций

### 3.1. Идея

RPC (Remote Procedure Call) — вызов функции, которая выполняется где-то ещё. Выглядит как обычный вызов, но под капотом — сетевой запрос:

```python
result = calculator.add(2, 3)           # локальный вызов
result = remote_calculator.add(2, 3)    # RPC — выполняется на сервере
```

**REST мыслит существительными** (ресурсами): `GET /users/42`. **RPC мыслит глаголами** (вызовами): `call getUser(42)`.

### 3.2. JSON-RPC — конкретная реализация

Это стандарт **формы** (спецификация на одну страницу: jsonrpc.org). Фиксирует структуру JSON-сообщений:

**Запрос:**
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": { "name": "read_inbox", "arguments": {} }
}
```

**Ответ:**
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": { "content": [{"type": "text", "text": "3 письма"}] }
}
```

**Ошибка:**
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": { "code": -32601, "message": "Method not found" }
}
```

Стандарт определяет только конверт (`jsonrpc`, `id`, `method`, `params`, `result`, `error`). Какие `method` бывают и что в `params` — определяет конкретный протокол. MCP определяет `tools/call`, `tools/list`. Ethereum определяет `eth_getBalance`, `eth_sendTransaction`.

### 3.3. JSON-RPC — текстовый. Почему не бинарный?

Осознанный выбор: **простота важнее производительности**. MCP передаёт небольшие сообщения (описания tools, текстовые результаты), не гигабайты. Выигрыш от бинарного формата незаметен, а сложность реализации вырастет.

Для сравнения: gRPC (бинарный, Protocol Buffers) выбирают, когда микросервисы обмениваются миллионами сообщений в секунду. MCP — это десятки вызовов за сессию. JSON читается глазами, парсер есть в каждом языке, порог входа минимальный.

### 3.4. Семейство RPC-технологий

| Технология | Формат данных | Транспорт | Где используется | Статус |
|---|---|---|---|---|
| **JSON-RPC** | JSON (текст) | Любой | MCP, LSP, Ethereum | Актуален |
| **gRPC** | Protobuf (бинарный) | HTTP/2 | Микросервисы, Kubernetes | Очень актуален |
| **tRPC** | JSON (текст) | HTTP | TypeScript full-stack | Популярен |
| **XML-RPC / SOAP** | XML (текст) | HTTP | Энтерпрайз 2000-х | Устарел |

RPC вездесущ, но часто скрыт: LSP (подсветка кода в VSCode) — это JSON-RPC, GraphQL — по сути RPC с единым эндпоинтом. Веб-разработчики часто не сталкиваются с RPC напрямую, потому что REST стал «дефолтом» в вебе. Но в инфраструктуре и тулинге RPC живее всех живых.

### 3.5. Protocol Buffers (Protobuf)

Protobuf — бинарный формат сериализации от Google. Это **не протокол**, а способ упаковки данных — как JSON, только бинарный, компактный и со строгой типизацией.

**Как работает:** сначала описываешь структуру данных в `.proto`-файле, потом генерируешь код для нужного языка:

```protobuf
// user.proto — описание структуры
syntax = "proto3";

message User {
  int32 id = 1;          // = 1, = 2 — номера полей, не значения
  string name = 2;
  string email = 3;
  repeated string roles = 4;  // массив
}

message GetUserRequest {
  int32 id = 1;
}

// Описание сервиса (для gRPC)
service UserService {
  rpc GetUser (GetUserRequest) returns (User);
  rpc ListUsers (Empty) returns (stream User);  // серверный стриминг
}
```

```bash
# Генерация кода из .proto
protoc --python_out=. --grpc_python_out=. user.proto
# Создаёт user_pb2.py (структуры) и user_pb2_grpc.py (клиент/сервер)
```

**Сравнение с JSON:**

```
JSON:     {"id": 42, "name": "Alice", "email": "alice@example.com"}
          → 57 байт, текст, читается глазами

Protobuf: 08 2A 12 05 41 6C 69 63 65 1A 11 61 6C 69 63 65 ...
          → ~30 байт, бинарный, нужен .proto для декодирования
```

| | JSON | Protobuf |
|---|---|---|
| Формат | Текст | Бинарный |
| Размер | Больше (~2x) | Компактный |
| Скорость парсинга | Медленнее | Быстрее (~5-10x) |
| Читаемость | Да | Нет (нужна схема) |
| Типизация | Нет (всё строки/числа) | Строгая (int32, string, enum...) |
| Эволюция схемы | Свободная | Контролируемая (номера полей) |

**Номера полей** (`= 1`, `= 2`) — ключевая деталь. В бинарном формате хранится номер, а не имя. Можно переименовать поле, не ломая совместимость. Нельзя переиспользовать номер удалённого поля.

**Где используется:**
- **gRPC** — и запросы, и ответы сериализуются через Protobuf
- **Kafka / очереди** — компактные сообщения между микросервисами
- **Мобильные приложения** — экономия трафика
- **Внутренние API Google** — исторически, Protobuf создан для этого

**gRPC + Protobuf** — полная связка: Protobuf описывает данные и сервисы (`.proto`), gRPC обеспечивает транспорт (HTTP/2), генератор кода создаёт клиент и сервер на любом языке. Это «JSON-RPC на стероидах» — строгие контракты, стриминг, бинарная эффективность, но за счёт сложности.

### 3.6. gRPC подробнее

gRPC (Google Remote Procedure Call) — фреймворк для RPC, использующий HTTP/2 как транспорт и Protobuf как формат данных.

**Четыре режима вызова:**

```
1. Unary:            клиент ──запрос──► сервер ──ответ──► клиент
                     (как обычный HTTP-запрос)

2. Server streaming: клиент ──запрос──► сервер ══ответ1═► клиент
                                               ══ответ2═►
                                               ══ответ3═►
                     (как SSE)

3. Client streaming: клиент ══запрос1═► сервер ──ответ──► клиент
                            ══запрос2═►
                            ══запрос3═►
                     (загрузка файла чанками)

4. Bidirectional:    клиент ══════════► сервер
                            ◄══════════
                     (как WebSocket, но по HTTP/2)
```

**Почему HTTP/2:** мультиплексирование — один TCP-коннект несёт параллельные запросы без блокировки. В HTTP/1.1 запросы стоят в очереди; в HTTP/2 они идут одновременно по одному соединению.

**На практике:** gRPC — стандарт для общения микросервисов внутри кластера (Kubernetes, облака). Для внешних клиентов (браузер, мобилка) обычно ставят REST/GraphQL-шлюз перед gRPC-сервисами, потому что браузеры не умеют gRPC напрямую (нужен grpc-web прокси).

### 3.7. Connect RPC — gRPC для веба

Connect (connectrpc.com, от компании Buf) решает главную проблему gRPC: **браузеры не умеют gRPC**. gRPC требует HTTP/2 trailers, прямой доступ к фреймам — этого в браузерном `fetch` нет. Раньше обходились через grpc-web прокси (Envoy), но это лишний слой.

Connect — это **отдельный протокол**, который позволяет вызывать те же gRPC-сервисы через обычный HTTP POST с JSON body:

```
gRPC (классический):
  POST /user.v1.UserService/GetUser
  Content-Type: application/grpc
  Body: [бинарный Protobuf с length-prefix фреймингом]
  → требует HTTP/2, trailers, специальный клиент

Connect:
  POST /user.v1.UserService/GetUser
  Content-Type: application/json
  Body: {"id": 42}
  → обычный HTTP/1.1, работает из fetch(), curl, Postman
```

Ключевое: **один и тот же сервер** отвечает на все три протокола — gRPC, gRPC-Web и Connect. Клиент выбирает формат через `Content-Type`:

```
┌──────────────┐     application/grpc       ┌─────────────────────┐
│  Go-сервис   │ ◄──────────────────────── │  gRPC-клиент        │
│  (внутри     │                            │  (микросервис)       │
│   кластера)  │     application/json       │                     │
│              │ ◄──────────────────────── │  Браузер (fetch)     │
│  Один и тот  │                            │                     │
│  же handler  │     application/grpc-web   │                     │
│              │ ◄──────────────────────── │  gRPC-Web клиент     │
└──────────────┘                            └─────────────────────┘
```

**Что даёт Connect:**

| | gRPC | Connect |
|---|---|---|
| Транспорт | HTTP/2 обязателен | HTTP/1.1 достаточно |
| Формат тела | Protobuf (бинарный) | JSON или Protobuf (выбор клиента) |
| Браузер | Не работает напрямую | Работает через `fetch()` |
| curl / Postman | Сложно (бинарный формат) | Просто (обычный JSON POST) |
| Прокси (Envoy) | Не нужен для gRPC-Web | Не нужен вообще |
| Стриминг | Все 4 режима | Все 4 режима (server streaming через SSE-подобный формат) |

**Сервис описывается в том же `.proto`-файле** — Connect не требует отдельного описания API. Buf генерирует и серверный, и клиентский код:

```protobuf
// тот же user.proto, что и для gRPC
service UserService {
  rpc GetUser (GetUserRequest) returns (User);
}
```

```typescript
// Сгенерированный Connect-клиент (TypeScript)
import { createClient } from "@connectrpc/connect";
import { UserService } from "./gen/user_connect";

const client = createClient(UserService, transport);
const user = await client.getUser({ id: 42 });
// Под капотом: POST /user.v1.UserService/GetUser с JSON body
```

**Когда что выбирать:**

| Сценарий | Протокол |
|---|---|
| Микросервис ↔ микросервис (внутри кластера) | gRPC — максимальная производительность |
| Браузер → бэкенд (новый проект) | Connect — gRPC-совместимость + работает из браузера |
| Браузер → существующий gRPC-бэкенд | Connect или gRPC-Web — оба поддерживаются тем же сервером |
| Публичный API для внешних клиентов | REST + OpenAPI — универсальнее, привычнее |

Connect не заменяет REST для публичных API, но **убирает необходимость в REST-шлюзе** перед gRPC-сервисами для внутреннего фронтенда.

## 4. Swagger / OpenAPI — описание REST API

### 4.1. Проблема

REST API не несёт в себе описание своей структуры. Ты получил URL `https://api.example.com` — и что дальше? Какие эндпоинты? Какие параметры? Какие форматы ответов? Без документации — только метод проб и ошибок.

### 4.2. OpenAPI — стандарт описания

OpenAPI (ранее Swagger) — спецификация для **машиночитаемого описания** REST API. Это YAML/JSON-файл, который полностью описывает API: эндпоинты, методы, параметры, типы данных, авторизацию.

```yaml
# openapi.yaml — пример описания API
openapi: 3.0.3
info:
  title: Email API
  version: 1.0.0

paths:
  /emails:
    get:
      summary: List emails
      parameters:
        - name: folder
          in: query
          schema:
            type: string
            enum: [inbox, sent, drafts]
      responses:
        '200':
          description: List of emails
          content:
            application/json:
              schema:
                type: array
                items:
                  $ref: '#/components/schemas/Email'

    post:
      summary: Send email
      requestBody:
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/NewEmail'
      responses:
        '201':
          description: Email sent

  /emails/{id}:
    get:
      summary: Get email by ID
      parameters:
        - name: id
          in: path
          required: true
          schema:
            type: integer

components:
  schemas:
    Email:
      type: object
      properties:
        id:
          type: integer
        subject:
          type: string
        from:
          type: string
        body:
          type: string

    NewEmail:
      type: object
      required: [to, subject, body]
      properties:
        to:
          type: string
        subject:
          type: string
        body:
          type: string
```

### 4.3. Swagger vs OpenAPI — история названий

- **Swagger** — изначальное название проекта (2011, компания SmartBear)
- **OpenAPI** — название спецификации после передачи в Linux Foundation (2016)
- Сейчас: **OpenAPI** — спецификация, **Swagger** — набор инструментов (Swagger UI, Swagger Editor, Swagger Codegen)

На практике «Swagger» и «OpenAPI» часто используют как синонимы.

### 4.4. Что даёт OpenAPI-файл

Из одного YAML-файла генерируется целая экосистема:

| Инструмент | Что делает |
|---|---|
| **Swagger UI** | Интерактивная веб-документация — можно пробовать запросы прямо в браузере |
| **Codegen** | Генерация клиентских SDK на 40+ языках (Python, TypeScript, Go...) |
| **Mock-сервер** | Фейковый сервер для фронтенда, пока бэкенд не готов |
| **Валидация** | Автопроверка запросов/ответов на соответствие схеме |
| **Тесты** | Генерация тестов для API на основе описания |

### 4.5. Связь с другими концепциями

**OpenAPI и JSON Schema** — OpenAPI использует JSON Schema (с расширениями) для описания структуры данных. Те же `type`, `properties`, `required`, `$ref` — что и в JSON Schema для валидации.

**OpenAPI и MCP** — MCP-сервер описывает свои tools через JSON Schema параметров. По сути, это тот же принцип: машиночитаемое описание интерфейса. Разница — OpenAPI описывает HTTP-эндпоинты, а MCP tools описывает JSON-RPC методы.

**OpenAPI и Protobuf** — обе решают проблему «контракта» между клиентом и сервером, но по-разному:

| | OpenAPI | Protobuf (.proto) |
|---|---|---|
| Для чего | REST API (HTTP + JSON) | gRPC (HTTP/2 + бинарный) |
| Формат описания | YAML/JSON | Свой DSL (.proto) |
| Генерация кода | Опциональна | Обязательна |
| Валидация в рантайме | Да (middleware) | Встроена в сериализацию |

## 5. Матрёшка протоколов

Протоколы вкладываются друг в друга. Вот как выглядят разные варианты:

```
REST API:             запрос (JSON)    →  HTTP           →  TCP  →  интернет

gRPC:                 Protobuf         →  HTTP/2         →  TCP  →  интернет

JSON-RPC (локальный): JSON-RPC         →  stdio (stdin/stdout)   →  локальный процесс

JSON-RPC (удалённый): JSON-RPC         →  Streamable HTTP (POST + SSE)  →  TCP  →  интернет
```

Один и тот же HTTP может нести REST-запрос, JSON-RPC вызов или gRPC-фрейм — в зависимости от соглашения, которому следуют клиент и сервер.
