# Web Development: базовые понятия стека

Заметки из сессии разбора архитектуры проекта imaga.ai — как устроена фронтенд-разработка, сборка и деплой.

---

## 1. Что такое Node.js?

**Node.js** — среда выполнения JavaScript вне браузера. Установлена на компьютере разработчика и позволяет запускать JS-программы в терминале.

В проекте imaga.ai Node.js используется только при разработке и сборке:
- Запустить dev-сервер (Vite)
- Собрать HTML/JS/CSS файлы для продакшена
- Установить пакеты (`npm install`)

В продакшене Node.js не работает — Vercel просто раздаёт готовые статические файлы.

## 2. Что такое npm и команда `run`?

**npm** (Node Package Manager) — менеджер пакетов, идёт в комплекте с Node.js.

`run` — встроенная подкоманда npm (не отдельный пакет). `npm run <имя>` берёт скрипт из секции `"scripts"` в `package.json` и выполняет его в shell.

```json
// package.json
"scripts": {
  "dev": "vite",
  "build": "vite build",
  "build:ssg": "..."
}
```

Цепочка вызова:
```
npm run dev → npm смотрит scripts → "dev": "vite" → запускает команду vite
```

Имя скрипта — просто ярлык. Можно назвать как угодно. `dev` — общепринятая конвенция для команды запуска dev-сервера.

При этом `vite` находится не глобально, а в `node_modules/.bin/` — npm автоматически добавляет этот путь при запуске через `run`.

## 3. Что такое Vite?

**Vite** (читается «вит», от фр. «быстрый») — инструмент сборки и dev-сервер для фронтенда. npm-пакет, установленный в проекте.

Что делает:
- Поднимает локальный HTTP-сервер (в нашем проекте на порту 8080)
- Превращает `.tsx` файлы (которые браузер не понимает) в обычный JavaScript на лету
- Поддерживает HMR (Hot Module Replacement) — изменил код, браузер мгновенно обновился без перезагрузки

Конфигурация — в файле `vite.config.ts`:
```ts
server: {
  host: "::",    // слушает все интерфейсы
  port: 8080,    // порт dev-сервера
}
```

Vite — это **не бэкенд-сервер**. Он только отдаёт фронтенд-код браузеру во время разработки. Бэкенд в проекте — Supabase (облачный сервис).

## 4. Как устроен продакшен (нет `npm run prod`)

В продакшене **нет запущенного сервера**. Скриптов типа `npm run prod` нет.

Деплой — статическая генерация:
```
push в main → Vercel запускает build:ssg → генерируются готовые HTML/JS/CSS
→ Vercel раздаёт их через CDN (как обычный файловый хостинг)
```

Сравнение:
| | Dev (локально) | Prod (Vercel) |
|---|---|---|
| Vite | Работает как живой сервер | Один раз собрал файлы и всё |
| Node.js | Нужен для запуска Vite | Не используется |
| Обновления | Мгновенные (HMR) | Требуется пересборка и деплой |

## 5. Что такое Next.js и чем отличается от Vite?

**Next.js** — фреймворк для React от компании Vercel. В проекте imaga.ai **не используется**.

| | Vite | Next.js |
|---|---|---|
| Что это | Сборщик + dev-сервер | Полноценный фреймворк |
| Сервер в проде | Нет (только статика) | Да (Node.js сервер) |
| Рендеринг | В браузере (SPA) или SSG | SSR, SSG, ISR — всё |
| Бэкенд | Нет (нужен внешний) | Встроенный (API routes) |
| Философия | Ты сам собираешь архитектуру | Всё идёт «из коробки» |

Vite — как молоток: простой инструмент. Next.js — как конструктор IKEA: и молоток, и инструкция, и полки.

## 6. Иерархия понятий

```
Node.js   = движок (среда выполнения JS)
  └── npm = менеджер пакетов (встроен в Node.js)
        ├── Vite     = инструмент сборки    ← используем
        ├── React    = UI-библиотека        ← используем
        └── Next.js  = фреймворк            ← НЕ используем
```

Все эти пакеты работают поверх Node.js, но это разные инструменты для разных задач.

## 7. Файл methodology.md — спецификация vs рантайм

На примере Web Tech Advisor: файл `docs/tools/stack-advisor/methodology.md` — это **чистая спецификация**, которая не загружается в рантайме.

В коде слово "methodology" встречается только в комментариях:
```ts
// confidence.ts:64 — "methodology says SSG + unsure → SSG_ISR"
// confidence.ts:76 — "methodology specifies [0.20, 0.95]"
// validate.ts:3   — "validates against 12 reference cases from methodology v1.2"
```

Вся логика из методологии уже захардкожена в TypeScript — таблицы решений, веса, пороги. Удаление `methodology.md` на работу инструмента не повлияет. Файл полезен только как документация для людей.

**Вывод**: спецификация определяет *что* должен делать код, но после реализации код живёт самостоятельно. Спецификация становится справочным документом.
