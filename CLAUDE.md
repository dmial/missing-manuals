# learning-notes — правила работы для агента

Это учебные заметки. После публикации они цитируются как справочник, поэтому фактологическая точность критична.

## Сверка фактов с документацией

**Триггер.** Ты собираешься утверждать конкретный технический факт о внешней системе — API, CLI-инструменте, библиотеке, фреймворке, протоколе, платформе — и текущие файлы проекта этот факт не подтверждают. Примеры: схема запроса/ответа API, семантика CLI-флага, сигнатура функции, ключи в конфиге, поведение в редком сценарии, имена путей на диске.

**Действие.** До того как написать факт в заметку или в ответ пользователю — сверь его с официальной документацией соответствующей системы (через `WebFetch`, специализированного агента или MCP-ресурс, если такой есть). Хеджирование («вероятно», «я думаю», «возможно») НЕ заменяет проверку. Допустимо ровно два режима: либо проверил и пишу уверенно, либо явно говорю «не уверен, надо свериться» и спрашиваю.

**Почему.** Учебные заметки потом цитируются как источник. Один вызов проверки сейчас дешевле, чем мина, заложенная для будущих читателей. Угаданный факт может оказаться верным — а может не оказаться, и без сверки разницу не увидеть.

### Конкретный пример: Claude Code

Когда речь о внутренностях Claude Code — пути на диске (`~/.claude/...`), поведение флагов (`--scope`, `--json` и т.п.), схема `settings.json`, что делает встроенная команда под капотом, поведение hooks / skills / plugins / marketplaces в runtime — сверяйся через агент `claude-code-guide` либо `WebFetch` на `docs.claude.com`.

Прецедент: я угадал семантику `--scope project` без проверки. В тот раз угадал верно, но мог ошибиться — и заметка с дефектом ушла бы в репозиторий.
