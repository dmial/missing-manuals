# Skills, Plugins и Marketplaces в Claude Code

*(The Missing Manual)*

## Пролог. Три концепта — три задачи

 **Есть три концепта**, каждый из которых решает свою задачу:

- **Skill** — единица поведения. Что Claude делает в ответ на запрос пользователя.
- **Plugin** — единица установки и дистрибуции. Что пользователь явно ставит, и чем автор делится с другими как единым пакетом.
- **Marketplace** — канал доставки и единица подписки. Откуда пользователь узнаёт о доступных плагинах и получает обновления.

## Часть 1. Skill — атом поведения

### Что такое скилл

**Роль.** Атом поведения. Пакет инструкций для модели на конкретный класс ситуаций — «когда пользователь делает X, действуй по такой-то схеме».

**Структура.** Папка с файлом `SKILL.md` внутри. Файл содержит:

1. **Frontmatter** (метаданные в заголовке): имя скилла, описание, триггеры активации.
2. **Тело** — инструкция-промпт, которую Claude прочитает, когда триггер сработает.

Минимальный пример:

```markdown
---
name: my-skill
description: Краткое описание для модели — она по нему решит, нужен ли скилл
---

# My Skill

Здесь подробные инструкции, что делать, когда скилл активирован.
```

Упрощенно говоря, скилл — это предзаданный промпт, добавляемый в контекст автоматически в определенный момент.

### Как скилл активируется

Когда пользователь пишет запрос в Claude Code, модель смотрит на описания **всех включённых** скиллов (поле `description` из frontmatter каждого) и решает, какой из них релевантен. Если триггер сработал — содержимое `SKILL.md` загружается в контекст, и Claude следует инструкции.

Это **не магическая команда**, а просто промпт-инжиниринг: автор скилла продумывает, как описать триггер, чтобы скилл вызывался в нужный момент и не вызывался во всех остальных.

### Скилл — больше, чем просто промпт

Скилл может включать не только `SKILL.md`, но и вспомогательные файлы рядом с ним: исполняемые скрипты (Python или shell), которые Claude запускает через Bash; дополнительные справочные документы, на которые `SKILL.md` ссылается для углублённых инструкций; и статические ресурсы — например, изображения, шрифты или данные.

### Что НЕ делает скилл сам по себе

Но скилл всё равно остаётся **пакетом инструкций и ресурсов для модели**, а не интеграцией с Claude Code как с приложением. Он не умеет:

- Перехватывать события Claude Code (например, разрешения).
- Регистрировать слэш-команды.
- Поднимать MCP-серверы.
- Иметь номер версии и канал обновлений.

Hooks, команды и MCP-серверы доступны в Claude Code и без плагина — hooks конфигурируются в `settings.json`, MCP-серверы — в `.mcp.json` / `settings.json`, команды кладутся в `~/.claude/commands/`. Но **связать их со скиллом и собрать в один версионируемый пакет**, который ставится одной командой, можно только на уровень выше — в плагине.

### Как управлять скиллом

Куда положить скилл — определяется двумя независимыми вопросами:

1. **Для кого он?** Только тебе или расходится по людям.
2. **Насколько он привязан к контексту?** Один конкретный репозиторий или универсальный, межпроектный.

Эти две оси дают четыре варианта:

| | Привязан к одному репо | Универсальный (межпроектный) |
|---|---|---|
| **Только себе** | `<repo>/.claude/skills/<skill>/SKILL.md` (через `.gitignore`, если репозиторий общий) | `~/.claude/skills/<skill>/SKILL.md` |
| **Расходится по людям** | `<repo>/.claude/skills/<skill>/SKILL.md` под git — едет вместе с репо при клоне | **Плагин** — см. часть 4 |

**Три из четырёх клеток — это просто папка со `SKILL.md`.** Никаких реестров и регистрационных списков: положил папку — Claude увидит скилл при следующем запросе; удалил — скилла больше нет.

**Правый-нижний угол — особый случай.** Универсальный скилл, который должен расходиться по людям, требует полноценной упаковки: имя автора, номер версии, канал обновлений. Это уже работа плагина — управление переезжает с уровня папок на уровень команд `/plugin ...` или правки `settings.json`, см. часть 4 «Управление плагинами и маркетплейсами».

**Как положить «голый» скилл на место.** Можно вручную: создаёшь папку нужного уровня, кладёшь `SKILL.md`. Удобнее — поручить агенту. Ты говоришь, какой нужен скилл и в какой клетке квадранта он должен жить; агент сам решает, куда именно положить файлы (учитывая твои указания и структуру проекта), а собирать содержимое — frontmatter, тело, bundled resources — ему помогает встроенный скилл `skill-creator`. Промежуточный вариант: написать `SKILL.md` руками, а потом попросить агента проверить результат на корректность по правилам `skill-creator`.

**Это все, что обязательно знать про скиллы**, если не нужен функционал плагинов. Остальное закроет скилл `skill-creator`. Можно идти и пользоваться.

## Часть 2. Plugin — контейнер для скиллов и не только

### Что такое плагин

**Роль.** Единица установки и дистрибуции. Скилл — слишком маленькая единица, чтобы быть самостоятельной: если автору нужны номер версии, явное согласие пользователя на установку или возможность приложить hooks, команды и скрипты вместе со скиллом — всё это упаковывается в плагин.

**Структура.** Папка с подпапкой `.claude-plugin/` и файлом `plugin.json` внутри неё. В `plugin.json` описано имя плагина, версия, автор и всё, что плагин с собой несёт.

```
my-plugin/
├── .claude-plugin/
│   └── plugin.json          ← манифест: имя, версия, hooks, MCP-серверы
├── skills/                  ← один или несколько скиллов
│   ├── skill-one/
│   │   └── SKILL.md
│   └── skill-two/
│       └── SKILL.md
├── commands/                ← пользовательские слэш-команды (опционально)
└── hooks/                   ← shell-скрипты для перехвата событий (опционально)
```

`plugin.json` минимально:

```json
{
  "name": "my-plugin",
  "version": "1.0.0",
  "description": "Что делает плагин",
  "author": { "name": "Автор" }
}
```

С точки зрения пользователя, установившего плагин, структура папки и содержимое `plugin.json` — **подкапотная реализация: эти файлы не предназначены для ручной правки**. Всё, что с плагином можно сделать, делается через `/plugin install`, `/plugin update`, `/plugin uninstall`, `/plugin enable`, `/plugin disable` (см. часть 4). Править руками никто не запрещает, но правки переживут только до следующего `/plugin update` — там новая версия их затрёт.

Для автора плагина это **исходники** — здесь описаны имя, версия и состав пакета.

Скиллы живут в собственном неймспейсе (`plugin-name:skill-name`) и не конфликтуют с личными или проектными скиллами даже при совпадении имени.

### Один плагин может содержать несколько скиллов

Поскольку плагин — это единица дистрибуции, в один плагин логично класть скиллы, которые:

- От одного автора и обновляются синхронно.
- Делят общий контекст (например, набор инструментов одной роли).
- Бессмысленно ставить по одному.

Например, корпоративный «PM-кит» может содержать скиллы `project-brief`, `wbs-generator`, `kickoff-deck`, `meeting-summary` — все они часть одного процесса, ставятся вместе.

И наоборот: если скиллы независимы и предназначены разным аудиториям — лучше разные плагины.

### Что плагин даёт сверх скиллов

Плагин может включать в `plugin.json`:

- **Hooks** — shell-команды, которые Claude Code запускает в моменты определённых событий жизненного цикла сессии: перед/после вызова инструмента (`PreToolUse`/`PostToolUse`), при отправке пользователем промпта (`UserPromptSubmit`), при старте/остановке сессии и других. Через них можно, например, перехватить попытку выхода из плана и потребовать подтверждение.
- **Commands** — пользовательские слэш-команды (заготовленные промпт-шаблоны).
- **MCP servers** — внешние сервисы, которые по протоколу MCP отдают модели свой набор инструментов и ресурсов. Claude Code поднимает их при старте сессии (как локальный процесс) или подключается по сети. Примеры: GitHub MCP добавляет инструменты для работы с PR и issue, Figma MCP — для чтения дизайнов. Плагин может объявить нужные серверы в `plugin.json`, и пользователю не придётся настраивать их руками.

Это та функциональность, которой у голого скилла нет.

Hooks и MCP-серверы — большие темы, заслуживающие отдельного рассказа. В этому документе они вводятся только как одна из вещей, которые плагин умеет приносить с собой.

С командами иначе: по структуре и содержанию они близки к скиллам, поэтому уместно рассказать про них здесь — см. приложение «Команды — это заготовленные промпт-шаблоны».

## Часть 3. Marketplace — каталог плагинов

### Что это

**Роль.** Реестр плагинов и канал подписки. Один источник подписки, через который пользователь получает доступ к набору плагинов от автора.

**Структура.** Папка с подпапкой `.claude-plugin/` и файлом `marketplace.json` внутри неё. В `marketplace.json` перечислены доступные плагины и указано, где их брать.

Минимально:

```
my-marketplace/
└── .claude-plugin/
    └── marketplace.json
```

Содержимое:

```json
{
  "name": "my-marketplace",
  "owner": { "name": "Me" },
  "plugins": [
    { "name": "my-plugin", "source": "./plugins/my-plugin" },
    { "name": "another",   "source": "./plugins/another" }
  ]
}
```

С точки зрения пользователя, подписанного на marketplace, структура папки и содержимое `marketplace.json` — **подкапотная реализация: этот файл не предназначен для ручной правки**. Всё, что с marketplace можно сделать, делается через `/plugin marketplace add`, `/plugin marketplace update`, `/plugin marketplace remove` (см. часть 4); установка плагинов из каталога — через `/plugin install`. Править руками никто не запрещает, но правки переживут только до следующего `/plugin marketplace update` — там новая версия каталога их затрёт.

Для автора marketplace это, наоборот, **исходники** — здесь описаны имя каталога, перечень плагинов и адреса-источники. Параметр `source` (адрес плагина внутри каталога) — единственное, что важно осознанно выбирать самому; разбор форматов и закрепление версий — в части 7.

## Часть 4. Управление плагинами и маркетплейсами

### Команда `/plugin` — точка входа

Управление плагинами и маркетплейсами идёт через группу команд `/plugin ...`.

> `/plugin ...` захардкожены в Claude Code и не являются создаваемыми пользователем слэш-командами, о которых рассказывает приложение «Команды — это заготовленные промпт-шаблоны».

Каждая такая команда под капотом трогает (в разных комбинациях):

- **Внутренние папки и реестры в `~/.claude/plugins/`**: клоны маркетплейсов в `marketplaces/`, runtime-копии плагинов в `cache/`, реестры `known_marketplaces.json` и `installed_plugins.json` — это служебные файлы, не для правки руками.
- **`settings.json`** — общий конфиг Claude Code; в части плагинов хранит поля `extraKnownMarketplaces` (какие сторонние маркетплейсы подключены) и `enabledPlugins` (какие плагины активны). Этот файл, в отличие от папок выше, **предназначен для ручной правки** — но не как способ управлять плагинами оперативно, см. ниже «settings.json как манифест зависимостей».

Иными словами, в `~/.claude/plugins/` живут **материализованные артефакты** — скачанные файлы, кэш, служебные реестры. Это производное состояние: его можно потерять, и оно восстановится по запросу. В `settings.json` лежит **зафиксированное решение** — «знаю такие источники, хочу, чтобы такие плагины были активны». Это первичная декларация, и именно она шарится между машинами и людьми (см. ниже «settings.json как манифест зависимостей».).

> Слово **«extra»** в `extraKnownMarketplaces` значит «сверх дефолтных»: Claude Code знает про официальные маркетплейсы Anthropic из коробки, поэтому записи в `enabledPlugins` вида `something@anthropic-agent-skills` без соответствующего блока в `extraKnownMarketplaces` — нормально, маркетплейс встроенный.

### Подкоманды

**Маркетплейсы:**

- `/plugin marketplace add <source>` — подключить маркетплейс. `<source>` — `owner/repo`, git URL или локальный путь (полный список форматов — в части 7).
- `/plugin marketplace update <name>` — обновить каталог маркетплейса (узнать про новые версии плагинов). Файлы самих плагинов при этом не качаются.
- `/plugin marketplace remove <name>` — удалить маркетплейс. **Каскадно** удаляет все установленные из него плагины — это явно задокументировано. Если нужно «обновить, не удаляя» — используй `update`.

**Плагины:**

- `/plugin install <plugin>@<marketplace>` — установить плагин из подключённого маркетплейса.
- `/plugin update <plugin>` — перезабрать файлы установленной копии (после того, как `marketplace update` показал, что есть новая версия).
- `/plugin uninstall <plugin>` — удалить плагин с диска (маркетплейс не трогает). Алиасы: `remove`, `rm`.
- `/plugin enable <plugin>` / `/plugin disable <plugin>` — переключить активность плагина без удаления файлов. **Разница с `uninstall`:** `disable` оставляет файлы в `cache/`, меняет только флаг в `enabledPlugins`; `uninstall` физически удаляет копию плагина.

**Рантайм:**

- `/reload-plugins` — применить изменения в текущей сессии без перезапуска. На диск ничего не пишет.

**Флаг `--scope user|project|local`** есть у подкоманд `install`, `uninstall`, `enable`, `disable` — он определяет, в какой `settings.json` запишется изменение. По умолчанию — `user`. (См. ниже «Области видимости».)

### Авто-обновления плагинов

**По умолчанию авто-обновление включено для официальных маркетплейсов Anthropic и выключено для всех остальных** (сторонних и твоих собственных). То есть из коробки само обновляется только то, что приходит от Anthropic; плагины из чужих репозиториев и из локальных папок обновляются вручную через `/plugin marketplace update` + `/plugin update`.

Чтобы включить авто-обновление сразу для всех плагинов, включая сторонние, надо задать env-переменную:

```bash
export FORCE_AUTOUPDATE_PLUGINS=1
```

### settings.json как манифест зависимостей

Главная роль `settings.json` для плагинов — не оперативное управление (для этого есть команды `/plugin`), а **декларативный шаринг конфигурации с командой**. Пара полей `extraKnownMarketplaces` (источники) + `enabledPlugins` (состав) устроена ровно как манифест зависимостей в современных пакетных менеджерах: декларация «вот источники, вот что из них должно быть активно», без фиксации точных версий.

Параллели:

| Экосистема | Манифест (≈ settings.json) | Lock |
|---|---|---|
| Node.js / npm | `package.json` | `package-lock.json` |
| iOS / CocoaPods | `Podfile` | `Podfile.lock` |
| Python / Poetry | `pyproject.toml` | `poetry.lock` |
| Ruby / Bundler | `Gemfile` | `Gemfile.lock` |
| Rust / Cargo | `Cargo.toml` | `Cargo.lock` |

`settings.json` соответствует **левой колонке**. Lock-аналога в Claude Code нет — точные версии плагинов фиксируются не пользователем, а автором маркетплейса (через `ref`/`sha`/`version` в `marketplace.json` — см. часть 7).

#### Как шарится `settings.json` — области видимости (scope) 

**Scope** — это область видимости настроек, то есть круг сессий, к которым применяется конкретный файл `settings.json`.

Куда писать настройки — определяется двумя независимыми вопросами:

1. **Для кого они?** Только тебе или расходятся по людям.
2. **Насколько привязаны к контексту?** Один конкретный репозиторий или межпроектные.

Эти две оси дают четыре варианта:

| | Привязан к одному репо | Универсальный (межпроектный) |
|---|---|---|
| **Только себе** | `<repo>/.claude/settings.local.json` — scope `local` (под `.gitignore`, за пределы твоей машины не уходит) | `~/.claude/settings.json` — scope `user` (все твои сессии на этой машине) |
| **Расходится по людям** | `<repo>/.claude/settings.json` — scope `project` (под git, едет с репо ко всем участникам) | `managed-settings.json` в системной директории — scope `managed`, разворачивает IT-админ через MDM |

Подкоманды `install`, `uninstall`, `enable`, `disable` принимают флаг `--scope user|project|local` — он определяет, в какой из трёх «обычных» `settings.json` записывается изменение. **По умолчанию scope — `user` (т.е. правка идёт в `~/.claude/settings.json`)**, даже если команда запущена внутри репозитория. Чтобы записать в project или local, флаг нужно задать явно. Scope `managed` через флаг не указывается — это IT-уровень.

Если одно и то же значение фигурирует сразу в нескольких scope, приоритет такой: **managed > local > project > user**. То есть IT-политика перекрывает всё, далее более узкая запись перекрывает более широкую.

#### Когда срабатывает установка по `settings.json`

Для scope `project` сценарий следующий: когда участник команды получает репозиторий с `.claude/settings.json`, открывает его в Claude Code и даёт **trust к папке**, CLI сравнивает декларацию с фактическим состоянием и при расхождении показывает **интерактивный prompt** с предложением установить недостающие маркетплейсы и плагины. После подтверждения CLI сам выполняет всю цепочку — добавляет маркетплейсы, ставит плагины, активирует их.

> Для scope `managed`, `user` и `local` аналогичное поведение в официальной документации не описано (есть ли prompt, происходит ли тихая установка или поведение различается), надо проверять отдельно.

## Часть 5. Эксперименты и доработки чужих плагинов

Типичные сценарии: 1) эксперименты с чужим плагином без установки, 2) доработка чужого плагина с возможностью подтягивать апдейты автора.

### Разовый эксперимент, без установки  плагина

**Контекст**: нужно запустить плагин на одну сессию, не подключая его marketplace и не оставляя следов в `settings.json`. Типичные ситуации:

- **Попробовать чужой плагин без обязательств** — посмотреть, что внутри, не подключая marketplace и не оставляя записей. Закрыл сессию — чисто.
- **Есть готовый архив** — кто-то прислал `.zip` или ссылку на него; `--plugin-url` запустит без распаковки и без отдельного каталога.

**Рекомендуемая схема:** подключить плагин CLI-флагами при старте сессии.

```bash
claude --plugin-dir /path/to/plugin           # папка или локальный .zip
claude --plugin-url https://.../plugin.zip    # URL .zip-архива
```

Поведение: плагин загружается **только на текущую сессию**, в `enabledPlugins` ничего не пишется. Каждый флаг принимает один путь; для нескольких плагинов флаг повторяется.

> Ограничения: эти флаги принимают только локальную папку, локальный `.zip` или URL архива. GitHub-репозиторий, npm-пакет, git-URL — не поддерживаются; для них нужен полноценный marketplace. Для **постоянной установки** marketplace необходим — это единственный поддерживаемый путь.

### Доработка чужого плагина с GitHub

**Контекст**: чужой marketplace опубликован на GitHub, плагин лежит в той же репе через relative path (`source: "./plugins/X"`) — это стандартная схема публикации. Ты форкаешь репу маркетплейса себе на GitHub, клонируешь свой fork локально и хочешь:

- Дорабатывать конкретный плагин под себя.
- Иметь возможность делиться правками (минимально — backup в облако).
- Уметь подтягивать апдейты автора, но не быть к ним привязанным.

> Подразумевается классическая GitHub-схема: и сам термин «форк», и работа с remote `upstream` ниже не имеют смысла вне GitHub/GitLab. Альтернативные источники плагинов в каталоге (внешние репозитории в `source`, npm) здесь не рассматриваем — для них дев-цикл устроен иначе.

**Рекомендуемая схема:**

1. **Форкнуть на GitHub** — чтобы свои изменения куда-то пушить.
2. **Перенастроить remotes:** `origin` → твой fork, `upstream` → оригинал автора.
3. **Подключить локальную папку как marketplace** (local path) — чтобы Claude Code тянул плагины из твоей локальной копии, без push в fork и `git pull` из fork на каждую итерацию.

Настройка git-remotes:

```bash
git remote rename origin upstream
git remote add origin git@github.com:<твой-username>/<repo>.git
git push -u origin main
```

Подключение склонированной папки как marketplace в Claude Code:

```bash
/plugin marketplace add ./path/to/cloned-repo
/plugin install <plugin-name>@<marketplace-name>
```

> **Важно про имена.** `<marketplace-name>` — это значение поля `name` в `<repo>/.claude-plugin/marketplace.json` (а не имя папки, в которую ты склонировал). `<plugin-name>` — поле `name` в записи плагина в том же `marketplace.json`. Подсмотреть их можно в самом файле или через `/plugin marketplace list` после `add`.

#### Что нужно делать после правок в файлах

 Просто отредактировать файл в склонированном репо и ждать, что Claude Code увидит изменения, **не получится**: `/plugin marketplace add` не даёт hot-reload. Плагин при установке копируется в `~/.claude/plugins/cache/`, и Claude Code потом работает с этой копией, а не с твоей рабочей папкой.

Цикл такой:

```bash
# 1. Правишь файлы плагина в склонированной папке.

# 2. Зафиксировать, что версия изменилась — иначе update не подхватит правки:
#    - если в plugin.json есть "version": "..." — бампни его вручную;
#    - если version нет — закоммить правки (Claude Code возьмёт commit SHA как версию).
git add ... && git commit -m "wip"

# 3. Сказать Claude Code перечитать каталог и пересобрать кэш плагина:
/plugin marketplace update <marketplace-name>
/plugin update <plugin-name>

# 4. Применить изменения в текущей сессии без рестарта:
/reload-plugins
```

Откуда требование менять версию: Claude Code определяет, нужен ли update, сравнивая версию плагина с тем, что лежит в кэше. Версия резолвится по приоритету: `version` в `plugin.json` → `version` в записи маркетплейса → commit SHA git-источника → `unknown` для локальных папок не под git. Если строка версии не поменялась, `/plugin update` скажет `already at the latest version` и ничего не сделает.

Параллельно — можно затягивать изменения от автора с upstream:

```bash
# Подтянуть апдейты автора селективно
git fetch upstream
git log --oneline upstream/main ^main      # что нового
git merge upstream/main                    # либо cherry-pick по коммитам
```

#### Почему для своей разработки лучше локальный путь, а не git

С локальным путем шагов меньше и они проще.

| Шаг | Локальная папка как marketplace | Свой fork (git source) как marketplace |
|---|---|---|
| 1. Правка файлов плагина | edit | edit |
| 2. Зафиксировать новую версию (commit или bump `version` в `plugin.json`) | `git commit -m "..."` | `git commit -m "..."` |
| 3. Отправить в удалённый репозиторий | — *(не нужно)* | `git push origin main` |
| 4. Перечитать каталог marketplace | `/plugin marketplace update <name>` *(чтение файла с диска)* | `/plugin marketplace update <name>` *(внутри — `git pull` репы маркетплейса из fork)* |
| 5. Обновить файлы плагина в кэше | `/plugin update <plugin>` *(копирование из локальной папки в `cache/`)* | `/plugin update <plugin>` *(копирование из локального clone репы маркетплейса в `cache/` — сети уже не нужно, файлы пришли на шаге 4)* |
| 6. Применить в текущей сессии | `/reload-plugins` | `/reload-plugins` |

«Мгновенного» dev-цикла всё равно не получится: шаг 2 (commit или bump) обязателен в обоих случаях, иначе `/plugin update` увидит ту же версию в кэше и не станет ничего обновлять (см. правило резолвинга версии выше).

Git source как marketplace нужен только тогда, когда хочешь, чтобы **другие** ставили твою версию через `/plugin marketplace add <твой-username>/<repo>`.

## Часть 6. Создаем своё. Декомпозиция: что в чём держать

Поскольку у нас три уровня — скилл, плагин и marketplace — нужно понять, где какие границы. Идём снизу вверх: сначала решаем про скиллы, потом про плагины, потом про marketplace.

### Шаг 1. Скилл — упаковывать в плагин или оставить «голым»?

Сначала развилка:

- **Если в комплекте со скиллом идут hooks, MCP-серверы или slash-команды** — нужен плагин, распространяемый через marketplace. Альтернатив нет. Каждый из этих компонентов по отдельности плагина не требует (hooks конфигурируются в `settings.json`, MCP-серверы — в `.mcp.json` / `settings.json`, команды кладутся в `~/.claude/commands/`), но связать их со скиллом в один версионированный пакет можно только так. Дальше — шаги 2 и 3.
- **Если всего этого рядом со скиллом нет** — действует матрица из части 1:

| | Привязан к одному репо | Универсальный (межпроектный) |
|---|---|---|
| **Только себе** | Оставляем голым: `<repo>/.claude/skills/<skill>/SKILL.md` (через `.gitignore`, если репо общий) | Оставляем голым: `~/.claude/skills/<skill>/SKILL.md` |
| **Расходится по людям** | Оставляем голым: `<repo>/.claude/skills/<skill>/SKILL.md` под git — едет вместе с репо при клоне | **Плагин** — дальше шаги 2 и 3 |

Три клетки из четырёх — голый скилл. Плагин нужен только в правом-нижнем углу: межпроектный скилл, который должен расходиться по людям. Там уже не обойтись без имени автора, номера версии (то есть плагина) и канала обновлений (то есть маркетплейса).

### Шаг 2. Несколько скиллов: один плагин или разные?

Когда скилл уже решено упаковывать, вопрос — компоновать ли его с другими.

| Сигнал | Решение |
|---|---|
| Скиллы одной роли, делят общий workflow, ставятся вместе | Один плагин |
| Один автор, общий цикл релизов | Один плагин |
| Бессмысленно ставить по одному (без соседей пользы мало) | Один плагин |
| Скиллы независимы, разные аудитории | Разные плагины |
| Разные циклы релизов и темп изменений | Разные плагины |

**Один скилл на плагин — это нормально?** Да, вполне. Накладные расходы — только тонкий `plugin.json`. Так делают, когда:

- Скилл значимый, имеет собственную ценность и аудиторию.
- Хочется отдельной версии и отдельного канала обновления.
- В комплекте идут hooks, команды или MCP-сервер — это тянет за собой плагин.

Не делай этого «из правильности». Если у тебя пять связанных скиллов одной команды — это один плагин, не пять.

### Шаг 3. Плагины: один marketplace или несколько?

Marketplace — единица **подписки**. Подписавшись на marketplace, пользователь получает доступ ко всему набору плагинов и обновления к ним.

| Сигнал | Решение |
|---|---|
| Все плагины от одной команды/бренда, общая аудитория | Один marketplace |
| Один владелец и одна политика обновлений | Один marketplace |
| Аудитории не пересекаются (внутренние/внешние, разные роли) | Разные marketplace |
| Разные политики доверия и обновлений | Разные marketplace |
| Несвязанные продукты | Разные marketplace |

Главная идея: **marketplace — это «кому подписываться»**. Если внешний разработчик не должен видеть внутренние плагины, они не должны жить в одном каталоге, даже если технически могли бы.

### Юзкейс: компания на 50 человек, несколько департаментов, ~100–200 плагинов

В этом масштабе наивная схема «всё в одном плагине» или «один marketplace на компанию» уже трещит. Разумная декомпозиция:

1. **Marketplace — по аудитории/департаменту**, не «один на компанию» и не «по плагину». Например:
   - `acme-engineering` — для разработчиков (code review, релизы, тестовые скиллы).
   - `acme-pm` — для PM-ов (брифы, статус-апдейты, дорожные карты).
   - `acme-design`, `acme-ops`, `acme-data`...

   Каждый сотрудник подписывается только на нужные. Дизайнер не таскает в контексте описания скиллов SRE.

2. **Плагины внутри marketplace — по роли + workflow.** Не один гигантский `acme-engineering`, а несколько: `code-review`, `incident-response`, `release-tooling`. Это даёт независимый версионинг и возможность поставить только часть.

3. **Один скилл на плагин — только когда обоснованно** (отдельная аудитория, отдельный цикл, hooks/MCP в комплекте). Иначе склеивай в наборы.

4. **Часть скиллов держи проектными**, а не в плагинах. Скилл, полезный только внутри одного репозитория, не тащи в marketplace — он только шумит. Положи в `<repo>/.claude/skills/`, его увидят те, кто работает в этом репо.

Грубый ориентир для 100–200 единиц: **3–7 marketplace по департаментам, 5–15 плагинов в каждом, 1–10 скиллов в плагине** — на этот объём такая раскладка ложится естественно.

## Часть 7. Создаем своё. Разработка плагинов и маркетплейсов

Если ты пишешь плагин для себя или для команды, **не нужен** GitHub-аккаунт и публикация. Достаточно локальной папки.

Большую часть рутины — структуру папок, заполнение `plugin.json` и `marketplace.json`, разводку скиллов, hooks и MCP-серверов — агент сделает сам. Что важно знать **тебе** и чего за тебя никто не решит — это как именно адресовать источники: куда смотрит marketplace, откуда тянутся плагины, и как привязать их к конкретной точке во времени. Этим и определяется, как будет жить и обновляться твоя сборка.

### Параметр `source`: семантика и форматы

Параметр `source` — это адрес-источник. Он есть и у marketplace, и у плагина внутри `marketplace.json`. **Семантика этих двух адресов разная**, перечень форматов — частично пересекающийся.

#### Семантика

| Адрес чего | Где записан | Что CLI находит по адресу | Пользователь правит руками? |
|---|---|---|---|
| **Marketplace** | В `settings.json`, в поле `extraKnownMarketplaces` | Файл `marketplace.json` — реестр доступных плагинов. В папках и git-источниках лежит в подкаталоге `.claude-plugin/`; HTTP(S) URL может указывать прямо на файл | Да. Альтернатива команде `/plugin marketplace add` |
| **Плагин** | В `marketplace.json` соответствующего marketplace, в `plugins[].source` | Папку плагина с `.claude-plugin/plugin.json` и содержимым — `skills/`, `commands/`, `hooks/` | Нет. Записывается автором marketplace при сборке каталога |

Marketplace — это **реестр** (перечень доступных плагинов), плагин — это **пакет** (папка с исполняемым содержимым). CLI сначала идёт за реестром, потом, по записи в реестре, — за пакетом.

**Два `source` — две независимые точки выбора.** Каталог и плагины в нём не обязаны лежать в одном месте: marketplace может быть локальной папкой, а внутри него — записи, ссылающиеся на чужие GitHub-репы (или наоборот). Это даёт ту самую гибкость, ради которой агенту нужны явные указания от тебя — куда именно класть каталог и откуда брать каждый плагин.

#### Форматы

| Тип | Как записан | Marketplace | Plugin |
|---|---|---|---|
| GitHub-репо | строка `owner/repo` или объект `{"source": "github", "repo": "owner/repo"}` | ✓ | ✓ |
| Git URL (https / ssh) | строка `https://...` или объект `{"source": "url", "url": "..."}` | ✓ | ✓ |
| Локальная папка | путь к папке | ✓ | ✓ |
| Прямой URL на `marketplace.json` | `https://example.com/marketplace.json` | ✓ | — |
| Git subdir в монорепо | `{"source": "git-subdir", "url": "...", "path": "tools/plugin"}` | — | ✓ |
| npm-пакет | `{"source": "npm", "package": "@org/plugin"}` | — | ✓ |

Примеры записи `source` для плагина в `plugins[]`:

```json
{ "source": "./plugins/local" }                                      // относительный путь
{ "source": { "source": "github", "repo": "owner/repo" } }           // GitHub
{ "source": { "source": "url", "url": "https://..." } }              // прямой URL
{ "source": { "source": "git-subdir", "url": "...", "path": "..." } } // подпапка git-репо
{ "source": { "source": "npm", "package": "@org/plugin" } }          // npm-пакет
{ "source": { "source": "directory", "path": "/abs/path" } }         // абсолютный локальный путь
```

**Разные плагины в одном marketplace могут иметь разные типы `source`.** Это позволяет, например, корпоративному реестру миксовать «свои» плагины (локальные пути) и «одобренные сторонние» (GitHub-ссылки на конкретные коммиты).

То есть **marketplace — это просто содержимое `marketplace.json`**, способ доставки вторичен. Git и GitHub — конвенция, а не требование.

### Закрепление за веткой, тегом или коммитом

Для git-based источников (`github`, `git-subdir`) по умолчанию берётся **default branch** репозитория (обычно `main` или `master`) и её последний коммит. Чтобы зафиксировать конкретную точку, в объект `source` добавляют:

| Поле | Что закрепляет |
|---|---|
| `ref` | Имя ветки или тега (например, `"v2.0.0"`, `"stable"`) |
| `sha` | Конкретный коммит (40 символов хеша) |
| `version` | Явная версия плагина — Claude Code сравнивает её при `/plugin marketplace update` и пропускает обновление, если значение не изменилось |

Примеры с закреплением:

```json
{
  "name": "my-plugin",
  "source": {
    "source": "github",
    "repo": "owner/repo",
    "ref": "v2.0.0"
  }
}
```

```json
{
  "name": "another-plugin",
  "source": {
    "source": "git-subdir",
    "url": "https://github.com/owner/monorepo.git",
    "path": "packages/another-plugin",
    "sha": "a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0"
  }
}
```

**Как работает обновление плагина.** В Claude Code обновление двухшаговое (см. часть 4): сначала `/plugin marketplace update` перечитывает `marketplace.json` (там лежат `source` каждого плагина), затем `/plugin update <plugin>` реально перезабирает файлы плагина с указанного источника. Что происходит на каждом шаге, зависит от того, как закреплён `source`:

- **`ref` — ветка, `version`/`sha` не заданы.** `marketplace.json` со временем не меняется (всё та же ветка), поэтому из `/plugin marketplace update` о новых коммитах узнать нельзя — это выясняется только во время `/plugin update`, когда Claude Code тянет HEAD ветки и обнаруживает изменения.
- **Задан `sha` или `version`.** Когда автор marketplace меняет значение в `marketplace.json`, после `/plugin marketplace update` Claude Code видит новый pin и может показать «доступно обновление» во вкладке `/plugin` — файлы плагина при этом ещё не скачиваются. Реально pull произойдёт только на следующем шаге, через `/plugin update`. Это работает как `apt update` + `apt upgrade`: индекс отдельно от установки.

## Приложение. Команды — это заготовленные промпт-шаблоны

Слэш-команда — это **заготовленный промпт-шаблон**: markdown-файл с инструкцией для модели, который разворачивается в чат, когда пользователь явно набирает `/<имя>`. По содержанию команда близка к скиллу — там тоже текст «что делать модели в этом случае». Различия — в способе активации и в структуре:

| | Скилл | Команда |
|---|---|---|
| Авто-активация моделью по `description` | Да | Нет |
| Явный вызов пользователем через слэш | Да, через `/<skill-name>` (если `user-invocable: true`, что по умолчанию) | Да, через `/<command-name>` — единственный способ |
| Структура | Папка с `SKILL.md` + опциональные скрипты, ассеты, доп. документы | Просто markdown-файл с промптом (или папка с `COMMAND.md`) |

Скилл — более универсальный механизм: модель может вспомнить про него сама, а пользователь — вызвать руками. Команда — упрощённый формат **только для явного вызова**: без авто-активации и без богатой структуры со скриптами и ассетами.

Дистрибуция у команд устроена так же, как у скиллов, — три точки:

- Личные: `~/.claude/commands/<name>.md`
- Проектные: `<repo>/.claude/commands/<name>.md`
- Из плагинов: `<plugin>/commands/<name>.md`

Когда что выбирать:

- **Скилл** — если хочется, чтобы модель сама вспоминала про него в подходящий момент («когда работаешь с PDF-файлами, используй вот этот workflow»).
- **Команда** — если пользователь и так знает, когда её вызывать, и не нужно ни авто-активации, ни вспомогательных файлов («сделай ревью PR по моему шаблону»).
