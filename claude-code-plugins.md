# Skills, Plugins и Marketplaces в Claude Code

*(The Missing Manual)*

## Пролог. Три концепта — три задачи

 **Есть три концепта**, каждый из которых решает свою задачу:

- **Skill** — единица поведения. Что Claude делает в ответ на запрос пользователя.
- **Plugin** — единица установки и дистрибуции. Что пользователь явно ставит, и чем автор делится с другими как единым пакетом.
- **Marketplace** — канал доставки и единица подписки. Откуда пользователь узнаёт о доступных плагинах и получает обновления.

## Часть 1. Skill — атом поведения

### Что такое скилл

**Роль.** Атом поведения. Пакет инструкций для модели на конкретный класс ситуаций — «когда пользователь делает X, действуй по такой-то схеме».

**Структура.** Папка с файлом `SKILL.md` внутри. Файл содержит:

1. **Frontmatter** (метаданные в заголовке): имя скилла, описание, триггеры активации.
2. **Тело** — инструкция-промпт, которую Claude прочитает, когда триггер сработает.

Минимальный пример:

```markdown
---
name: my-skill
description: Краткое описание для модели — она по нему решит, нужен ли скилл
---

# My Skill

Здесь подробные инструкции, что делать, когда скилл активирован.
```

Упрощенно говоря, скилл — это предзаданный промпт, добавляемый в контекст автоматически в определенный момент.

### Как скилл активируется

Когда пользователь пишет запрос в Claude Code, модель смотрит на описания **всех включённых** скиллов (поле `description` из frontmatter каждого) и решает, какой из них релевантен. Если триггер сработал — содержимое `SKILL.md` загружается в контекст, и Claude следует инструкции.

Это **не магическая команда**, а просто промпт-инжиниринг: автор скилла продумывает, как описать триггер, чтобы скилл вызывался в нужный момент и не вызывался во всех остальных.

### Скилл — больше, чем просто промпт

Скилл может включать не только `SKILL.md`, но и вспомогательные файлы рядом с ним: исполняемые скрипты (Python или shell), которые Claude запускает через Bash; дополнительные справочные документы, на которые `SKILL.md` ссылается для углублённых инструкций; и статические ресурсы — например, изображения, шрифты или данные.

### Что НЕ делает скилл сам по себе

Но скилл всё равно остаётся **пакетом инструкций и ресурсов для модели**, а не интеграцией с Claude Code как с приложением. Он не умеет:

- Перехватывать события Claude Code (например, разрешения).
- Регистрировать слэш-команды.
- Поднимать MCP-серверы.
- Иметь номер версии и канал обновлений.

Hooks, команды и MCP-серверы доступны в Claude Code и без плагина — hooks конфигурируются в `settings.json`, MCP-серверы — в `.mcp.json` / `settings.json`, команды кладутся в `~/.claude/commands/`. Но **связать их со скиллом и собрать в один версионируемый пакет**, который ставится одной командой, можно только на уровень выше — в плагине.

### Как управлять скиллами (без связанных Hooks, команд и MCP-серверов)

Куда положить скилл — определяется двумя независимыми вопросами:

1. **Для кого он?** Только тебе или расходится по людям.
2. **Насколько он привязан к контексту?** Один конкретный репозиторий или универсальный, межпроектный.

Эти две оси дают четыре варианта:

| | Привязан к одному репо | Универсальный (межпроектный) |
|---|---|---|
| **Только себе** | `<repo>/.claude/skills/<skill>/SKILL.md` (через `.gitignore`, если репозиторий общий) | `~/.claude/skills/<skill>/SKILL.md` |
| **Расходится по людям** | `<repo>/.claude/skills/<skill>/SKILL.md` под git — едет вместе с репо при клоне | **Плагин** — см. часть 4 |

**Три из четырёх клеток — это просто папка со `SKILL.md`.** Никаких реестров и регистрационных списков: положил папку — Claude увидит скилл при следующем запросе; удалил — скилла больше нет.

**Правый-нижний угол — особый случай.** Универсальный скилл, который должен расходиться по людям, требует полноценной упаковки: имя автора, номер версии, канал обновлений. Это уже работа плагина — управление переезжает с уровня папок на уровень команд `/plugin ...` или правки `settings.json`, см. часть 4 «Управление плагинами и маркетплейсами».

**Как положить «голый» скилл на место.** Можно вручную: создаёшь папку нужного уровня, кладёшь `SKILL.md`. Удобнее — поручить агенту. Ты говоришь, какой нужен скилл и в какой клетке квадранта он должен жить; агент сам решает, куда именно положить файлы (учитывая твои указания и структуру проекта), а собирать содержимое — frontmatter, тело, bundled resources — ему помогает встроенный скилл `skill-creator`. Промежуточный вариант: написать `SKILL.md` руками, а потом попросить агента проверить результат на корректность по правилам `skill-creator`.

**Это все, что обязательно знать про скиллы**, если не нужен функционал плагинов. Остальное закроет скилл `skill-creator`. Можно идти и пользоваться.

## Часть 2. Plugin — контейнер для скиллов и не только

### Что такое плагин

**Роль.** Единица установки и дистрибуции. Скилл — слишком маленькая единица, чтобы быть самостоятельной: если автору нужны номер версии, явное согласие пользователя на установку или возможность приложить hooks, команды и скрипты вместе со скиллом — всё это упаковывается в плагин.

**Структура.** Папка с подпапкой `.claude-plugin/` и файлом `plugin.json` внутри неё. В `plugin.json` описано имя плагина, версия, автор и всё, что плагин с собой несёт.

```
my-plugin/
├── .claude-plugin/
│   └── plugin.json          ← манифест: имя, версия, hooks, MCP-серверы
├── skills/                  ← один или несколько скиллов
│   ├── skill-one/
│   │   └── SKILL.md
│   └── skill-two/
│       └── SKILL.md
├── commands/                ← пользовательские слэш-команды (опционально)
└── hooks/                   ← shell-скрипты для перехвата событий (опционально)
```

`plugin.json` минимально:

```json
{
  "name": "my-plugin",
  "version": "1.0.0",
  "description": "Что делает плагин",
  "author": { "name": "Автор" }
}
```

С точки зрения пользователя, установившего плагин, структура папки и содержимое `plugin.json` — **подкапотная реализация: эти файлы не предназначены для ручной правки**. Всё, что с плагином можно сделать, делается через `/plugin install`, `/plugin update`, `/plugin uninstall`, `/plugin enable`, `/plugin disable` (см. часть 4). Править руками никто не запрещает, но правки переживут только до следующего `/plugin update` — там новая версия их затрёт.

Для автора плагина это **исходники** — здесь описаны имя, версия и состав пакета. Разбор полей `plugin.json` нужен только ему, и он — в части 6.

Скиллы живут в собственном неймспейсе (`plugin-name:skill-name`) и не конфликтуют с личными или проектными скиллами даже при совпадении имени.

### Один плагин может содержать несколько скиллов

Поскольку плагин — это единица дистрибуции, в один плагин логично класть скиллы, которые:

- От одного автора и обновляются синхронно.
- Делят общий контекст (например, набор инструментов одной роли).
- Бессмысленно ставить по одному.

Например, корпоративный «PM-кит» может содержать скиллы `project-brief`, `wbs-generator`, `kickoff-deck`, `meeting-summary` — все они часть одного процесса, ставятся вместе.

И наоборот: если скиллы независимы и предназначены разным аудиториям — лучше разные плагины.

### Что плагин даёт сверх скиллов

Плагин может включать в `plugin.json`:

- **Hooks** — shell-команды, которые Claude Code запускает в моменты определённых событий жизненного цикла сессии: перед/после вызова инструмента (`PreToolUse`/`PostToolUse`), при отправке пользователем промпта (`UserPromptSubmit`), при старте/остановке сессии и других. Через них можно, например, перехватить попытку выхода из плана и потребовать подтверждение.
- **Commands** — пользовательские слэш-команды (заготовленные промпт-шаблоны).
- **MCP servers** — внешние сервисы, которые по протоколу MCP отдают модели свой набор инструментов и ресурсов. Claude Code поднимает их при старте сессии (как локальный процесс) или подключается по сети. Примеры: GitHub MCP добавляет инструменты для работы с PR и issue, Figma MCP — для чтения дизайнов. Плагин может объявить нужные серверы в `plugin.json`, и пользователю не придётся настраивать их руками.

Это та функциональность, которой у голого скилла нет.

Hooks и MCP-серверы — большие темы, заслуживающие отдельного рассказа. В этому документе они вводятся только как одна из вещей, которые плагин умеет приносить с собой.

С командами иначе: по структуре и содержанию они близки к скиллам, поэтому уместно рассказать про них здесь — см. приложение «Команды — это заготовленные промпт-шаблоны».

## Часть 3. Marketplace — каталог плагинов

### Что это

**Роль.** Реестр плагинов и канал подписки. Один источник подписки, через который пользователь получает доступ к набору плагинов от автора.

**Структура.** Папка с подпапкой `.claude-plugin/` и файлом `marketplace.json` внутри неё. В `marketplace.json` перечислены доступные плагины и указано, где их брать.

Минимально:

```
my-marketplace/
└── .claude-plugin/
    └── marketplace.json
```

Содержимое:

```json
{
  "name": "my-marketplace",
  "owner": { "name": "Me" },
  "plugins": [
    { "name": "my-plugin", "source": "./plugins/my-plugin" },
    { "name": "another",   "source": "./plugins/another" }
  ]
}
```

С точки зрения пользователя, подписанного на marketplace, структура папки и содержимое `marketplace.json` — **подкапотная реализация: этот файл не предназначен для ручной правки**. Всё, что с marketplace можно сделать, делается через `/plugin marketplace add`, `/plugin marketplace update`, `/plugin marketplace remove` (см. часть 4); установка плагинов из каталога — через `/plugin install`. Править руками никто не запрещает, но правки переживут только до следующего `/plugin marketplace update` — там новая версия каталога их затрёт.

Для автора marketplace это, наоборот, **исходники** — здесь описаны имя каталога, перечень плагинов и адреса-источники. Разбор полей `marketplace.json`, форматов `source` и закрепления версий нужен только ему, и он — в части 6.

## Часть 4. Управление плагинами и маркетплейсами

### Команда `/plugin` и `settings.json`

Управление плагинами и маркетплейсами идёт через группу команд `/plugin ...` — они захардкожены в Claude Code и не являются пользовательскими слэш-командами. Под капотом меняются две вещи:

- **внутренние папки Claude Code** — клоны marketplace в `~/.claude/plugins/marketplaces/` и runtime-копии плагинов в `~/.claude/plugins/cache/` (трогать руками не надо);
- **`settings.json`** — общий конфиг Claude Code (отвечает за множество настроек; в части плагинов хранит, какие marketplace подключены и какие плагины включены).

`settings.json` — единственная часть «под капотом», предназначенная для ручной правки (как `.gitconfig` или `.vscode/settings.json`). Манифесты `plugin.json` и `marketplace.json` из частей 2 и 3 — наоборот, исходники автора пакета: править их пользователю смысла нет.

>Общее правило: управлять плагинами и маркетплейсами надо через команды `/plugin ...`. Ручной правкой `settings.json` задача решается только в юзкейсах 4–5 (включить или выключить уже установленный плагин).


### Области видимости (scope) `settings.json`

**Scope** — это область видимости настроек, то есть круг сессий, к которым применяется конкретный файл `settings.json`.

Куда писать настройки — определяется двумя независимыми вопросами:

1. **Для кого они?** Только тебе или расходятся по людям.
2. **Насколько привязаны к контексту?** Один конкретный репозиторий или межпроектные.

Эти две оси дают четыре варианта:

| | Привязан к одному репо | Универсальный (межпроектный) |
|---|---|---|
| **Только себе** | `<repo>/.claude/settings.local.json` — scope `local` (под `.gitignore`, за пределы твоей машины не уходит) | `~/.claude/settings.json` — scope `user` (все твои сессии на этой машине) |
| **Расходится по людям** | `<repo>/.claude/settings.json` — scope `project` (под git, едет с репо ко всем участникам) | `managed-settings.json` в системной директории — scope `managed`, разворачивает IT-админ через MDM |

Подкоманды `install`, `uninstall`, `enable`, `disable` принимают флаг `--scope user|project|local` — он определяет, в какой из трёх «обычных» `settings.json` записывается изменение. **По умолчанию scope — `user` (т.е. правка идёт в `~/.claude/settings.json`)**, даже если команда запущена внутри репозитория. Чтобы записать в project или local, флаг нужно задать явно. Scope `managed` через флаг не указывается — это IT-уровень.

Если одно и то же значение фигурирует сразу в нескольких scope, приоритет такой: **managed > local > project > user**. То есть IT-политика перекрывает всё, далее более узкая запись перекрывает более широкую.

### Юзкейс 1. Подписаться на marketplace и поставить из него первый плагин

**Контекст.** Ты получил ссылку на чужой marketplace (или адрес репозитория с твоим собственным) и хочешь поставить оттуда хотя бы один плагин.

**Команды.**

```
/plugin marketplace add owner/repo                        # подключить marketplace
/plugin install plugin-name@marketplace-name              # поставить из него плагин
```

В `owner/repo` можно дать GitHub-сокращение, git URL или локальный путь — см. справочник по `source` в части 6.

**`/plugin marketplace add owner/repo` — подключить marketplace.**

На диске под капотом:

- В `~/.claude/plugins/marketplaces/<marketplace-name>/` появляется клон каталога (для git/URL-источников; локальные `source: directory` не клонируются — Claude Code читает прямо из исходной папки).
- В `~/.claude/plugins/known_marketplaces.json` — новая запись о подключённом marketplace.

В `settings.json` — добавляется запись в `extraKnownMarketplaces` (список подключённых сторонних marketplace):

```json
"extraKnownMarketplaces": {
  "marketplace-name": {
    "source": { "source": "github", "repo": "owner/repo" }
  }
}
```

Слово **«extra»** в `extraKnownMarketplaces` значит «сверх дефолтных»: Claude Code знает про официальные marketplace Anthropic из коробки, поэтому записи в `enabledPlugins` вида `something@anthropic-agent-skills` без соответствующего блока в `extraKnownMarketplaces` — нормально, marketplace встроенный.

**`/plugin install plugin-name@marketplace-name` — поставить плагин.**

На диске под капотом:

- В `~/.claude/plugins/cache/<marketplace-name>/<plugin-name>/<version>/` появляется копия плагина.
- В `~/.claude/plugins/installed_plugins.json` — запись об установке.

В `settings.json` — добавляется запись в `enabledPlugins` (какие плагины активны):

```json
"enabledPlugins": {
  "plugin-name@marketplace-name": true
}
```

> **Можно ли обойтись правкой `settings.json`:** не полностью. Объявить marketplace через `extraKnownMarketplaces` — это документированный путь (Anthropic явно описывает его в гайде по [team marketplaces](https://code.claude.com/docs/en/discover-plugins)). А вот запись в `enabledPlugins` без `/plugin install` саму установку не выполнит: **файлов плагина в `cache/` нет, реестр `installed_plugins.json` про него не знает**, а `enabledPlugins` — это переключатель «включён / выключен» для уже установленного плагина, а не команда «обеспечить установку».

### Юзкейс 2. Поставить ещё один плагин из уже подключённого marketplace

**Контекст.** Marketplace уже добавлен, и оттуда хочется поставить второй (третий, ...) плагин.

**Команды.**

```
/plugin install another-plugin@marketplace-name           # поставить ещё один плагин из уже подключённого marketplace
```

**`/plugin install another-plugin@marketplace-name` — поставить ещё один плагин.**

На диске под капотом:

- В `~/.claude/plugins/cache/<marketplace-name>/another-plugin/<version>/` появляется копия плагина.
- В `~/.claude/plugins/installed_plugins.json` — новая запись об установке.
- Папку marketplace в `marketplaces/` и реестр `known_marketplaces.json` команда не трогает — marketplace уже подключён.

В `settings.json` — добавляется запись в `enabledPlugins`:

```json
"enabledPlugins": {
  "another-plugin@marketplace-name": true
}
```

> **Можно ли обойтись правкой `settings.json`:** нет. Это `install`, и аргумент про переключатель из юзкейса 1 действует — без команды файлы плагина в `cache/` не появятся, а реестр `installed_plugins.json` руками править не стоит.

### Юзкейс 3. Обновить плагины

**Контекст.** Ты или автор marketplace выпустил новые версии плагинов, и хочется их получить.

Обновление двухслойное: сначала обновляется **каталог** (выясняется, какие версии сейчас доступны), потом — **установленная копия** конкретного плагина.

**Команды.**

```
/plugin marketplace update marketplace-name               # узнать про новые версии в каталоге
/plugin update plugin-name                                # перезабрать файлы установленной копии
/reload-plugins                                           # применить в текущей сессии
```

**`/plugin marketplace update <name>` — узнать про новые версии в каталоге.**

На диске под капотом — в `~/.claude/plugins/marketplaces/<name>/` подтягиваются последние коммиты каталога (`git pull`), обновляется `marketplace.json`.

В `settings.json` — ничего.

**`/plugin update <plugin>` — перезабрать файлы установленной копии.**

На диске под капотом:

- В `~/.claude/plugins/cache/<marketplace>/<plugin>/<новая-версия>/` появляется новая папка рядом со старой (старые версии держатся ~7 дней — grace period для уже идущих сессий).
- В `~/.claude/plugins/installed_plugins.json` обновляются `installPath`, `version`, `gitCommitSha` соответствующей записи.

В `settings.json` — ничего.

**`/reload-plugins` — применить в текущей сессии.**

На диске под капотом — ничего, это переключение runtime, на диск не пишет.

В `settings.json` — ничего.

Слой подписки и активации (`extraKnownMarketplaces`, `enabledPlugins`) обновлением не затрагивается — это работа с версиями и runtime-копиями, в `settings.json` она не отражается.

> **Можно ли обойтись правкой `settings.json`:** нет. `settings.json` хранит «что включено» и «какие marketplace подключены», но не работает с версиями и runtime-копиями плагинов. Обновление — это всегда либо команда, либо авто-обновление (юзкейс 8).

### Юзкейс 4. Временно выключить плагин, не удаляя

**Контекст.** Плагин мешает (например, конфликтует с другим скиллом), но удалять не хочется — он ещё пригодится.

**Команды.**

```
/plugin disable plugin-name                               # выключить плагин, не удаляя
```

**`/plugin disable plugin-name` — выключить плагин, не удаляя.**

На диске под капотом — ничего. Файлы плагина остаются в `cache/`, реестры `known_marketplaces.json` и `installed_plugins.json` не трогаются — плагин по-прежнему числится установленным.

В `settings.json` — в `enabledPlugins` запись плагина ставится в `false` (или удаляется — результат тот же):

```json
"enabledPlugins": {
  "plugin-name@marketplace-name": false
}
```

> **Можно ли обойтись правкой `settings.json`:** да, полностью эквивалентно. Поставь `false` в нужном уровне `settings.json` либо удали запись — скиллы и hooks перестают загружаться при следующем старте.
>
> Если плагин включён на более узком уровне (например, в local), записать `false` на более широком (project, user) не поможет — более узкий перекрывает. Выключать нужно на том же уровне, где плагин включён, либо на ещё более узком.

### Юзкейс 5. Снова включить выключенный плагин

**Контекст.** Плагин уже установлен (файлы на диске лежат), но в `enabledPlugins` стоит `false` или записи нет.

**Команды.**

```
/plugin enable plugin-name                                # снова включить выключенный плагин
```

**`/plugin enable plugin-name` — снова включить выключенный плагин.**

На диске под капотом — ничего, файлы плагина и так лежат в `cache/`, реестры не трогаются.

В `settings.json` — в `enabledPlugins` запись плагина ставится в `true` (если её не было — добавляется):

```json
"enabledPlugins": {
  "plugin-name@marketplace-name": true
}
```

> **Можно ли обойтись правкой `settings.json`:** да, полностью эквивалентно. В отличие от юзкейса 1 файлы плагина уже на диске, поэтому правка `enabledPlugins` работает без подводных камней.

### Юзкейс 6. Удалить плагин (marketplace оставить)

**Контекст.** Плагин больше не нужен, но marketplace продолжает быть полезным — оттуда стоит другое.

**Команды.**

```
/plugin uninstall plugin-name                             # удалить плагин (marketplace оставить)
```

Алиасы: `remove`, `rm`.

**`/plugin uninstall plugin-name` — удалить плагин.**

На диске под капотом:

- Папка плагина `~/.claude/plugins/cache/<marketplace>/<plugin>/` удаляется (вместе со всеми версиями).
- В `~/.claude/plugins/installed_plugins.json` запись об установке исчезает.
- Marketplace и его клон в `marketplaces/` остаются на месте — их трогает только юзкейс 7.

В `settings.json` — запись плагина исчезает из `enabledPlugins`.

> **Можно ли обойтись правкой `settings.json`:** нет. Удаление записи из `enabledPlugins` эквивалентно `/plugin disable`, а не `uninstall` — файлы плагина останутся в `cache/`, а запись — в `installed_plugins.json`. Чтобы получить полный аналог команды, пришлось бы вручную трогать внутренние реестры в `~/.claude/plugins/` (см. часть 6), а формат этих файлов CLI вправе менять. Используй команду.

### Юзкейс 7. Удалить marketplace целиком

**Контекст.** Вся подписка больше не нужна — например, автор закрыл проект, или ты переходишь на форк.

**Команды.**

```
/plugin marketplace remove marketplace-name               # удалить marketplace целиком (каскадно — и плагины из него)
```

Команда деструктивная: документация явно предупреждает, что *«Removing a marketplace also uninstalls any plugins you installed from it. To refresh a marketplace without losing installed plugins, use `claude plugin marketplace update` instead»* ([plugin-marketplaces reference](https://code.claude.com/docs/en/plugin-marketplaces#plugin-marketplace-remove)).

**`/plugin marketplace remove marketplace-name` — удалить marketplace целиком.**

На диске под капотом:

- Клон marketplace в `~/.claude/plugins/marketplaces/<name>/` удаляется.
- В `~/.claude/plugins/known_marketplaces.json` запись marketplace исчезает.
- **Каскадно** удаляются все установленные из этого marketplace плагины: их папки в `~/.claude/plugins/cache/<name>/<plugin>/` и соответствующие записи в `installed_plugins.json` — как если бы для каждого плагина выполнили юзкейс 6.

В `settings.json` — запись marketplace исчезает из `extraKnownMarketplaces`; записи плагинов из этого marketplace в `enabledPlugins` уходят вместе с самими плагинами (по симметрии с юзкейсом 6).

Встроенные marketplace (вроде `anthropic-agent-skills`) в `extraKnownMarketplaces` не лежат — удалить такой через `/plugin marketplace remove` нельзя.

> **Можно ли обойтись правкой `settings.json`:** нет — она даёт **другой результат**. Если просто удалить запись из `extraKnownMarketplaces`, marketplace перестанет считаться подключённым, **но установленные из него плагины останутся в `cache/` и в `installed_plugins.json`**, а их записи в `enabledPlugins` будут ссылаться на «несуществующий» marketplace — CLI положит такие плагины во вкладку **Errors**.
>
> Это полезный сценарий, если ты хочешь «отписаться, но плагины не терять». А вот настоящий аналог `/plugin marketplace remove` руками не собирается — пришлось бы вручную чистить `cache/` и подкапотные реестры, формат которых CLI вправе менять. Используй команду.

### Юзкейс 8. Настроить авто-обновления

**Контекст.** Не хочется каждый раз делать `marketplace update` + `plugin update` руками.

По умолчанию для официальных marketplace Anthropic авто-обновление **включено**, для сторонних (включая твои собственные) — **выключено**. Когда авто-обновление включено, Claude Code при старте сам делает `marketplace update` + `plugin update` и показывает уведомление с предложением запустить `/reload-plugins`.

**Команды.** Управляются через shell environment variables (кладутся в `~/.zshrc` / `~/.bashrc` или указываются перед командой):

```bash
export DISABLE_AUTOUPDATER=1                              # отключить авто-обновления Claude Code и плагинов
export FORCE_AUTOUPDATE_PLUGINS=1                         # принудительно включить авто-обновление плагинов даже при DISABLE_AUTOUPDATER=1
```

Полезная комбинация «обновляй плагины, но не сам CLI» — задать обе переменные одновременно.

**`DISABLE_AUTOUPDATER=1` / `FORCE_AUTOUPDATE_PLUGINS=1` — настроить авто-обновления.**

На диске под капотом — ничего напрямую от настройки переменной. Реальные изменения в `marketplaces/` и `cache/` происходят при следующем срабатывании авто-обновления (как в юзкейсе 3).

В `settings.json` — ничего; этот рукав в `settings.json` не вынесен, через него управлять авто-обновлениями нельзя.

> **Можно ли обойтись правкой `settings.json`:** нет, и не потому что она «не эквивалентна» — а потому что `settings.json` про авто-обновления вообще ничего не знает. Управление — только через переменные окружения выше.

## Часть 5. Создаем своё. Декомпозиция: что в чём держать

Поскольку у нас три уровня — скилл, плагин и marketplace — нужно понять, где какие границы. Идём снизу вверх: сначала решаем про скиллы, потом про плагины, потом про marketplace.

### Шаг 1. Скилл — упаковывать в плагин или оставить «голым»?

Сначала развилка:

- **Если в комплекте со скиллом идут hooks, MCP-серверы или slash-команды** — нужен плагин, распространяемый через marketplace. Альтернатив нет. Каждый из этих компонентов по отдельности плагина не требует (hooks конфигурируются в `settings.json`, MCP-серверы — в `.mcp.json` / `settings.json`, команды кладутся в `~/.claude/commands/`), но связать их со скиллом в один версионированный пакет можно только так. Дальше — шаги 2 и 3.
- **Если всего этого рядом со скиллом нет** — действует матрица из части 1:

| | Привязан к одному репо | Универсальный (межпроектный) |
|---|---|---|
| **Только себе** | Оставляем голым: `<repo>/.claude/skills/<skill>/SKILL.md` (через `.gitignore`, если репо общий) | Оставляем голым: `~/.claude/skills/<skill>/SKILL.md` |
| **Расходится по людям** | Оставляем голым: `<repo>/.claude/skills/<skill>/SKILL.md` под git — едет вместе с репо при клоне | **Плагин** — дальше шаги 2 и 3 |

Три клетки из четырёх — голый скилл. Плагин нужен только в правом-нижнем углу: межпроектный скилл, который должен расходиться по людям. Там уже не обойтись без имени автора, номера версии (то есть плагина) и канала обновлений (то есть маркетплейса).

### Шаг 2. Несколько скиллов: один плагин или разные?

Когда скилл уже решено упаковывать, вопрос — компоновать ли его с другими.

| Сигнал | Решение |
|---|---|
| Скиллы одной роли, делят общий workflow, ставятся вместе | Один плагин |
| Один автор, общий цикл релизов | Один плагин |
| Бессмысленно ставить по одному (без соседей пользы мало) | Один плагин |
| Скиллы независимы, разные аудитории | Разные плагины |
| Разные циклы релизов и темп изменений | Разные плагины |

**Один скилл на плагин — это нормально?** Да, вполне. Накладные расходы — только тонкий `plugin.json`. Так делают, когда:

- Скилл значимый, имеет собственную ценность и аудиторию.
- Хочется отдельной версии и отдельного канала обновления.
- В комплекте идут hooks, команды или MCP-сервер — это тянет за собой плагин.

Не делай этого «из правильности». Если у тебя пять связанных скиллов одной команды — это один плагин, не пять.

### Шаг 3. Плагины: один marketplace или несколько?

Marketplace — единица **подписки**. Подписавшись на marketplace, пользователь получает доступ ко всему набору плагинов и обновления к ним.

| Сигнал | Решение |
|---|---|
| Все плагины от одной команды/бренда, общая аудитория | Один marketplace |
| Один владелец и одна политика обновлений | Один marketplace |
| Аудитории не пересекаются (внутренние/внешние, разные роли) | Разные marketplace |
| Разные политики доверия и обновлений | Разные marketplace |
| Несвязанные продукты | Разные marketplace |

Главная идея: **marketplace — это «кому подписываться»**. Если внешний разработчик не должен видеть внутренние плагины, они не должны жить в одном каталоге, даже если технически могли бы.

### Юзкейс: компания на 50 человек, несколько департаментов, ~100–200 плагинов

В этом масштабе наивная схема «всё в одном плагине» или «один marketplace на компанию» уже трещит. Разумная декомпозиция:

1. **Marketplace — по аудитории/департаменту**, не «один на компанию» и не «по плагину». Например:
   - `acme-engineering` — для разработчиков (code review, релизы, тестовые скиллы).
   - `acme-pm` — для PM-ов (брифы, статус-апдейты, дорожные карты).
   - `acme-design`, `acme-ops`, `acme-data`...

   Каждый сотрудник подписывается только на нужные. Дизайнер не таскает в контексте описания скиллов SRE.

2. **Плагины внутри marketplace — по роли + workflow.** Не один гигантский `acme-engineering`, а несколько: `code-review`, `incident-response`, `release-tooling`. Это даёт независимый версионинг и возможность поставить только часть.

3. **Один скилл на плагин — только когда обоснованно** (отдельная аудитория, отдельный цикл, hooks/MCP в комплекте). Иначе склеивай в наборы.

4. **Часть скиллов держи проектными**, а не в плагинах. Скилл, полезный только внутри одного репозитория, не тащи в marketplace — он только шумит. Положи в `<repo>/.claude/skills/`, его увидят те, кто работает в этом репо.

Грубый ориентир для 100–200 единиц: **3–7 marketplace по департаментам, 5–15 плагинов в каждом, 1–10 скиллов в плагине** — на этот объём такая раскладка ложится естественно.

## Часть 6. Создаем своё. Разработка плагинов и маркетплейсов

Если ты пишешь плагин для себя или для команды, **не нужен** GitHub-аккаунт и публикация. Достаточно локальной папки. И именно с этого уровня важно знать, что у плагинов под капотом — какие папки и реестры Claude Code держит на диске, чтобы понимать, что произойдёт при `install`, `update`, `uninstall`.

### Локальный marketplace как dev-среда

Создаёшь папку с такой структурой:

```
my-marketplace/
├── .claude-plugin/
│   └── marketplace.json          # с одним или несколькими плагинами в plugins[]
└── plugins/
    └── my-plugin/
        ├── .claude-plugin/
        │   └── plugin.json
        └── skills/
            └── my-skill/
                └── SKILL.md
```

Подключаешь как локальную папку:

```
/plugin marketplace add /Users/dima/Dev/my-marketplace
/plugin install my-plugin@my-marketplace
```

Дальше пишешь скиллы, и при изменениях:

```
/plugin marketplace update my-marketplace
/plugin update my-plugin
/reload-plugins
```

В зависимости от настроек auto-update, иногда хватает только первого шага — но безопаснее делать все три.

### Манифест `marketplace.json` под капотом

Для того, кто marketplace **использует**, эти детали неважны — всё делает CLI. Но если ты **сам собираешь marketplace**, нужно понимать, что внутри `marketplace.json`.

#### Поля `marketplace.json`

| Поле | Назначение | Используется в логике? |
|---|---|---|
| `name` | ID реестра, используется в командах вида `plugin@marketplace` | Да |
| `version` | Версия самого манифеста (не плагинов внутри) | Косметика, не критична |
| `description` | Текст в UI при добавлении marketplace | UX |
| `owner` | Информация об авторе/команде | Справочно, технически не проверяется |
| `plugins[]` | **Список плагинов в каталоге — единственное функционально критичное поле** | Да |

Внутри `plugins[]` каждая запись содержит:

| Поле | Назначение |
|---|---|
| `name` | Имя плагина — по нему `/plugin install` ищет |
| `source` | Где лежит плагин (см. ниже) |
| `description` | Описание в UI |
| `category` | Группировка для фильтра (`productivity`, `development`...) |

#### Параметр `source`: семантика и форматы

Параметр `source` — это адрес-источник. Он есть и у marketplace, и у плагина внутри `marketplace.json`. Семантика этих двух адресов разная, перечень форматов — частично пересекающийся.

##### Семантика

| Адрес чего | Где записан | Что CLI находит по адресу | Пользователь правит руками? |
|---|---|---|---|
| **Marketplace** | В `settings.json`, в поле `extraKnownMarketplaces` | Файл `marketplace.json` — реестр доступных плагинов. В папках и git-источниках лежит в подкаталоге `.claude-plugin/`; HTTP(S) URL может указывать прямо на файл | Да. Альтернатива команде `/plugin marketplace add` |
| **Плагин** | В `marketplace.json` соответствующего marketplace, в `plugins[].source` | Папку плагина с `.claude-plugin/plugin.json` и содержимым — `skills/`, `commands/`, `hooks/` | Нет. Записывается автором marketplace при сборке каталога |

Marketplace — это **реестр** (перечень доступных плагинов), плагин — это **пакет** (папка с исполняемым содержимым). CLI сначала идёт за реестром, потом, по записи в реестре, — за пакетом.

##### Форматы

| Тип | Как записан | Marketplace | Plugin |
|---|---|---|---|
| GitHub-репо | строка `owner/repo` или объект `{"source": "github", "repo": "owner/repo"}` | ✓ | ✓ |
| Git URL (https / ssh) | строка `https://...` или объект `{"source": "url", "url": "..."}` | ✓ | ✓ |
| Локальная папка | путь к папке | ✓ | ✓ |
| Прямой URL на `marketplace.json` | `https://example.com/marketplace.json` | ✓ | — |
| Git subdir в монорепо | `{"source": "git-subdir", "url": "...", "path": "tools/plugin"}` | — | ✓ |
| npm-пакет | `{"source": "npm", "package": "@org/plugin"}` | — | ✓ |

Примеры записи `source` для плагина в `plugins[]`:

```json
{ "source": "./plugins/local" }                                      // относительный путь
{ "source": { "source": "github", "repo": "owner/repo" } }           // GitHub
{ "source": { "source": "url", "url": "https://..." } }              // прямой URL
{ "source": { "source": "git-subdir", "url": "...", "path": "..." } } // подпапка git-репо
{ "source": { "source": "npm", "package": "@org/plugin" } }          // npm-пакет
{ "source": { "source": "directory", "path": "/abs/path" } }         // абсолютный локальный путь
```

**Разные плагины в одном marketplace могут иметь разные типы `source`.** Это позволяет, например, корпоративному реестру миксовать «свои» плагины (локальные пути) и «одобренные сторонние» (GitHub-ссылки на конкретные коммиты).

То есть **marketplace — это просто содержимое `marketplace.json`**, способ доставки вторичен. Git и GitHub — конвенция, а не требование.

#### Закрепление за веткой, тегом или коммитом

Для git-based источников (`github`, `git-subdir`) по умолчанию берётся **default branch** репозитория (обычно `main` или `master`) и её последний коммит. Чтобы зафиксировать конкретную точку, в объект `source` добавляют:

| Поле | Что закрепляет |
|---|---|
| `ref` | Имя ветки или тега (например, `"v2.0.0"`, `"stable"`) |
| `sha` | Конкретный коммит (40 символов хеша) |
| `version` | Явная версия плагина — Claude Code сравнивает её при `/plugin marketplace update` и пропускает обновление, если значение не изменилось |

Примеры с закреплением:

```json
{
  "name": "my-plugin",
  "source": {
    "source": "github",
    "repo": "owner/repo",
    "ref": "v2.0.0"
  }
}
```

```json
{
  "name": "another-plugin",
  "source": {
    "source": "git-subdir",
    "url": "https://github.com/owner/monorepo.git",
    "path": "packages/another-plugin",
    "sha": "a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0"
  }
}
```

**Как работает обновление плагина.** В Claude Code обновление двухшаговое (см. часть 4): сначала `/plugin marketplace update` перечитывает `marketplace.json` (там лежат `source` каждого плагина), затем `/plugin update <plugin>` реально перезабирает файлы плагина с указанного источника. Что происходит на каждом шаге, зависит от того, как закреплён `source`:

- **`ref` — ветка, `version`/`sha` не заданы.** `marketplace.json` со временем не меняется (всё та же ветка), поэтому из `/plugin marketplace update` о новых коммитах узнать нельзя — это выясняется только во время `/plugin update`, когда Claude Code тянет HEAD ветки и обнаруживает изменения.
- **Задан `sha` или `version`.** Когда автор marketplace меняет значение в `marketplace.json`, после `/plugin marketplace update` Claude Code видит новый pin и может показать «доступно обновление» во вкладке `/plugin` — файлы плагина при этом ещё не скачиваются. Реально pull произойдёт только на следующем шаге, через `/plugin update`. Это работает как `apt update` + `apt upgrade`: индекс отдельно от установки.

### Где плагины живут на диске

Всё, что относится к плагинам, распределено по **трём слоям**: реестр источников, runtime-копии и состояние «включено / выключено». Каждый слой — своя папка и своя задача.

#### Общая картина

```
~/.claude/
├── plugins/
│   ├── marketplaces/                       ← клоны внешних git/URL-источников (локальные marketplace не клонируются)
│   │   └── <name>/
│   │       └── .claude-plugin/marketplace.json
│   ├── cache/                              ← runtime-копии установленных плагинов
│   │   └── <marketplace>/
│   │       └── <plugin>/
│   │           ├── 1.0.0/
│   │           ├── 1.1.0/
│   │           └── 1.2.0/                  ← текущая
│   ├── known_marketplaces.json             ← реестр: какие marketplace добавлены
│   └── installed_plugins.json              ← реестр зарегистрированных установок (cache/ может содержать orphan-версии после update/uninstall)
└── settings.json                           ← состояние: что включено
                                              (поле enabledPlugins, см. часть 4)

<repo>/.claude/settings.json                ← то же поле, но scope узче
<repo>/.claude/settings.local.json            (включения уровня проекта/локали)
```

Проверить можно одной командой:

```bash
ls ~/.claude/plugins/
# marketplaces/  cache/
```

#### Реестр marketplace'ов и их клоны

```
~/.claude/plugins/known_marketplaces.json    ← реестр всех добавленных
~/.claude/plugins/marketplaces/<name>/       ← клоны внешних git/URL-источников
```

**Реестр** `known_marketplaces.json` — источник истины: какие marketplace'ы добавлены. Для каждого хранятся имя, тип источника (`github`, `directory`, ...) и `installLocation` — путь к папке с содержимым marketplace'а.

**Папка `marketplaces/`** хранит **только клоны внешних источников** (`source: github` и т. п.). Локальные marketplace'ы с `source: directory` сюда не клонируются: их `installLocation` в реестре указывает прямо на исходную папку, и Claude Code читает `marketplace.json` оттуда.

Главный файл в каждом marketplace — `.claude-plugin/marketplace.json`, в нём — список доступных плагинов. Это аналог `apt`-списка пакетов: что в принципе доступно к установке. Из marketplace'а плагины **не исполняются**.

#### Реестр установок и их runtime-кэш

```
~/.claude/plugins/installed_plugins.json                    ← реестр зарегистрированных установок
~/.claude/plugins/cache/<marketplace>/<plugin>/<version>/   ← физические копии плагинов
```

**Реестр** `installed_plugins.json` — источник истины: какие плагины зарегистрированы как установленные. Для каждой записи хранятся `scope`, `version`, `installPath` (путь к конкретной папке в `cache/`), `installedAt`, `lastUpdated`, `gitCommitSha`.

**Папка `cache/`** — место, откуда плагин реально работает. Содержимое — физическая копия (не симлинк, не ссылка), сделанная в момент установки. Внутри версии — ровно та же структура, что в части 2:

```
cache/imaga/imaga-skills/1.2.0/
├── .claude-plugin/
│   └── plugin.json
├── skills/
├── commands/
└── hooks/
```

**Одновременно могут лежать несколько версий одного плагина.** При `/plugin update` или `/plugin uninstall` старая версия не стирается сразу: она помечается как **orphaned** и удаляется автоматически примерно через 7 дней (grace period). Это нужно, чтобы уже идущие сессии, загрузившие предыдущую версию в память, могли доработать без ошибок. Glob и Grep встроенных инструментов Claude Code такие orphan-папки в выдачу не включают — старый код не «торчит» в результатах поиска.

То есть orphans в `cache/` — это про **старые версии живых плагинов**, а не про плагины от удалённых marketplace. `/plugin marketplace remove` каскадно удаляет и сам marketplace, и все установленные из него плагины (см. юзкейс 7), так что после него в `cache/` от этого marketplace ничего не остаётся.

> **Про метку «← текущая» в дереве выше.** Обычно она совпадает с последней установленной версией, но активную выбирает **не сравнение semver-номеров папок**: запись плагина в `installed_plugins.json` содержит явное поле `installPath`, указывающее на конкретную папку версии в `cache/`. Именно эта запись и фиксирует, какая копия зарегистрирована как установленная.
>
> *Semver* (Semantic Versioning) — схема нумерации `MAJOR.MINOR.PATCH`: например, `1.2.0` старше `1.1.5` и младше `1.3.0`. Стандарт большинства пакетных систем — npm, cargo, RubyGems и т. д.

#### Где хранится «включено / выключено»

В самих папках плагинов этой информации нет. Состояние живёт в `settings.json`, в поле `enabledPlugins` (детально — в части 4):

- `~/.claude/settings.json` — глобально для пользователя.
- `<repo>/.claude/settings.json` и `.claude/settings.local.json` — для конкретного репозитория. Это то, что переключают флаги `--scope project|local` у `/plugin enable`.

Файлы самого плагина при этом всегда лежат в одном и том же `~/.claude/plugins/cache/`. **Scope управляет не местом хранения, а тем, где записан флаг «использовать».**

#### Поток: как marketplace становится runtime'ом

```
/plugin marketplace add  ──>  marketplaces/<name>/             (git clone каталога)
/plugin install <plugin> ──>  cache/<m>/<p>/<v>/               (копия плагина)
/plugin enable <plugin>  ──>  settings.json: enabledPlugins    (флаг включения)
```

Три каталога — три фазы: **узнать → установить → активировать**. Большинство операций обратимы независимо: `disable` не удаляет cache; `uninstall` не удаляет marketplace. Главное исключение — **`/plugin marketplace remove` каскадирует вниз**: убирает и marketplace, и все установленные из него плагины (документация явно предупреждает об этом, см. юзкейс 7). Это сделано осознанно — без marketplace плагины потеряли бы канал обновлений, и в `cache/` зависал бы мусор без хозяина.

В каждой фазе под капотом — **папка с содержимым** и **JSON-реестр**, который перечисляет, что в этой фазе зарегистрировано:

| Фаза | Папка с содержимым | Реестр |
|---|---|---|
| узнать | `marketplaces/<name>/` | `known_marketplaces.json` |
| установить | `cache/<m>/<p>/<v>/` | `installed_plugins.json` |
| активировать | — | `settings.json: enabledPlugins` |

Реестры — это «список зарегистрированных», папки — «что у этой записи лежит на диске». На фазе активации физической папки нет: включить плагин — это просто записать его имя в `enabledPlugins`.

#### Зачем cache вообще существует

Кажется, что для локального плагина (`source: directory`) можно было бы исполнять прямо из исходной папки. Кэш существует, потому что:

- **слепок на время сессии** — изменения файла на диске не меняют поведение скилла в полёте;
- **параллельные версии** — несколько версий лежат рядом, откат — это переключение указателя;
- **граница доверия** — установка явная, согласие на hooks и permissions дано один раз;
- **унификация источников** — git, npm, directory, URL после установки выглядят одинаково для loader'а.

Поэтому даже `directory` source копируется в `cache/`, а не читается напрямую из исходной папки.

### Когда хочется работать с чьим-то плагином

Сценарий: ты склонировал чужой репозиторий с marketplace, и хочешь:

- Дорабатывать конкретный плагин под себя.
- Иметь возможность делиться правками (минимально — backup в облако).
- Уметь подтягивать апдейты автора, но не быть к ним привязанным.

**Рекомендуемая схема:**

1. **Форкнуть на GitHub** — чтобы свои изменения куда-то пушить.
2. **Перенастроить remotes:** `origin` → твой fork, `upstream` → оригинал автора.
3. **Подключить локальную папку как marketplace** (directory source) — для быстрого dev-цикла без commit/push на каждое изменение.

Команды для перенастройки удалённых репозиториев:

```bash
cd /path/to/cloned-repo
git remote rename origin upstream
git remote add origin git@github.com:<твой-username>/<repo>.git
git push -u origin main
```

Дальше:

```bash
# Свои изменения
git add ... && git commit -m "..." && git push origin main

# Подтянуть апдейты автора селективно
git fetch upstream
git log --oneline upstream/main ^main      # что нового
git merge upstream/main                    # либо cherry-pick по коммитам
```

Опциональная страховка — отключить push к upstream, чтобы случайный `git push upstream` не падал с непонятной ошибкой авторизации:

```bash
git remote set-url --push upstream DISABLED
```

### Почему directory source, а не git source для своей разработки

Directory source даёт **мгновенный dev-цикл**: правишь файл — `marketplace update` — видишь изменения. Git source требует commit + push + pull для каждой итерации, что бессмысленно, пока ты на одной машине.

Git source нужен только тогда, когда хочешь, чтобы **другие** ставили твою версию через `/plugin marketplace add <твой-username>/<repo>`.

### Можно ли подключить плагин в обход marketplace

Для **разовой проверки** — да, через CLI-флаги:

```bash
claude --plugin-dir /path/to/plugin           # папка
claude --plugin-url https://.../plugin.zip    # архив
```

Это загружает плагин **только на одну сессию**, в `enabledPlugins` ничего не пишется. Удобно для теста плагина, который ты ещё не добавил в свой marketplace.

Для **постоянной установки** marketplace необходим — это единственный поддерживаемый путь.

## Приложение. Команды — это заготовленные промпт-шаблоны

Слэш-команда — это **заготовленный промпт-шаблон**: markdown-файл с инструкцией для модели, который разворачивается в чат, когда пользователь явно набирает `/<имя>`. По содержанию команда близка к скиллу — там тоже текст «что делать модели в этом случае». Различия — в способе активации и в структуре:

| | Скилл | Команда |
|---|---|---|
| Авто-активация моделью по `description` | Да | Нет |
| Явный вызов пользователем через слэш | Да, через `/<skill-name>` (если `user-invocable: true`, что по умолчанию) | Да, через `/<command-name>` — единственный способ |
| Структура | Папка с `SKILL.md` + опциональные скрипты, ассеты, доп. документы | Просто markdown-файл с промптом (или папка с `COMMAND.md`) |

Скилл — более универсальный механизм: модель может вспомнить про него сама, а пользователь — вызвать руками. Команда — упрощённый формат **только для явного вызова**: без авто-активации и без богатой структуры со скриптами и ассетами.

Дистрибуция у команд устроена так же, как у скиллов, — три точки:

- Личные: `~/.claude/commands/<name>.md`
- Проектные: `<repo>/.claude/commands/<name>.md`
- Из плагинов: `<plugin>/commands/<name>.md`

Когда что выбирать:

- **Скилл** — если хочется, чтобы модель сама вспоминала про него в подходящий момент («когда работаешь с PDF-файлами, используй вот этот workflow»).
- **Команда** — если пользователь и так знает, когда её вызывать, и не нужно ни авто-активации, ни вспомогательных файлов («сделай ревью PR по моему шаблону»).

## Шпаргалки

### Жизненный цикл плагина

```
[автор] публикует marketplace
   ↓
/plugin marketplace add <source>          → marketplaces/<name>/
   ↓
/plugin install <plugin>@<marketplace>    → cache/<marketplace>/<plugin>/<version>/
   ↓                                          + enabledPlugins["<plugin>@<marketplace>"] = true
/plugin disable / enable / uninstall      → меняет enabledPlugins или удаляет cache
   ↓
/plugin marketplace update                → обновляет marketplaces/<name>/ (каталог)
/plugin update <plugin>                   → обновляет cache/...../<новая-версия>/
/reload-plugins                           → применяет в текущей сессии
```

Расшифровка папок справа — в части 6.

### Где что лежит (для разбора и разработки)

| Путь | Что внутри |
|---|---|
| `~/.claude/plugins/marketplaces/<name>/` | Зеркало marketplace (клон git или ссылка на directory) |
| `~/.claude/plugins/cache/<marketplace>/<plugin>/<version>/` | Runtime-копия плагина |
| `~/.claude/settings.json` | `enabledPlugins`, `extraKnownMarketplaces` |

### Минимальные манифесты

`marketplace.json`:

```json
{
  "name": "my-marketplace",
  "owner": { "name": "Me" },
  "plugins": [
    { "name": "my-plugin", "source": "./plugins/my-plugin" }
  ]
}
```

`plugin.json`:

```json
{
  "name": "my-plugin",
  "version": "1.0.0",
  "description": "...",
  "author": { "name": "Me" }
}
```

`SKILL.md`:

```markdown
---
name: my-skill
description: Когда использовать этот скилл
---

# My Skill

Инструкции...
```

### Разделение ответственности — итог

| Уровень | Отвечает на вопрос | Единица чего |
|---|---|---|
| Marketplace | Откуда брать обновления? | Подписки |
| Plugin | Что я ставлю и чему доверяю? | Установки |
| Skill | Когда применить это поведение? | Активации в runtime |

Структурно это иерархия: marketplace содержит плагины, плагин содержит скиллы. Но **семантика каждого уровня самостоятельная** — это не просто матрёшка. Marketplace отвечает за подписку и канал доставки, плагин — за установку и доверие (в нём может вообще не быть скиллов — только hooks или MCP), скилл — за активацию поведения в runtime.
