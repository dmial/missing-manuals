# Куда паковать тулы агента

Тул для агента упаковывается одним из четырёх способов: **in-process registry**, **CLI**, **локальный MCP**, **remote MCP**. Выбор определяется не вкусом и не «универсальностью», а одним вопросом — **кто потребляет тул** — с двумя уточнениями поверх. Правило по умолчанию: бери самый низкий уровень, который решает задачу. «Более универсальный» вариант — это цена, а не достоинство.

Эта заметка — **карта выбора**, а не объяснение MCP: механика протокола (хост, клиент, сервер, транспорты, JSON-RPC) живёт в [mcp-protocol.md](mcp-protocol.md), а HTTP/REST — в [api-protocols.md](api-protocols.md). Здесь позиция другая — архитектор, который решает, *куда положить тул*, написав своего агента или взяв чужого.

Карта расширяется двумя оговорками, которые типичные разборы упускают. Первая — **вырожденный «нулевой» вариант**: не паковать тул совсем, а положить описание API в скилл и пустить агента его же generic-тулом (curl/fetch); сравнение с MCP — §10. Вторая — **вторая ось** выбора: не «куда положить тул», а «как агент его зовёт» — прямым typed-вызовом или написав код против тулов; она ортогональна четырём вариантам — §11.

---

## 1. Один вопрос решает почти всё: кто потребитель тула

Тул существует, чтобы его вызывали. Единственный вопрос, переупорядочивающий всё остальное: **кто его вызывает — твой собственный agent loop или чужой?**

- **Пишешь своего агента** (владеешь loop'ом) → потребитель = твой loop. Тулу некуда «ехать»: она живёт в том же процессе. Это in-process registry, и пока потребитель один, всё остальное — оверхед.
- **Берёшь чужого агента** (Claude Code, Cursor, облачный агент) → loop не твой. Потребитель по определению внешний, и тул надо упаковать так, чтобы внешний loop до него дотянулся. Вот здесь и начинаются CLI и MCP.

«Свой агент vs чужой» — это и есть «потребитель = мой loop или нет». Всё дерево висит на этом вопросе.

## 2. Диаграмма выбора

От далёкого варианта (максимум машинерии, максимум развязки) к близкому (минимум):

```
Q1 ─ Исполнитель тула УДАЛЁННЫЙ? ───────────── да ──▶ ① remote MCP
     (чужой датацентр; «код не присылай,             тул и его данные живут у тебя;
      дай эндпоинт»; облачный агент как loop)         потребитель ходит по URL+токен, ничего не ставит
        │
        нет  (всё локально)
        │
Q2 ─ Тулы нужны НЕЗАВИСИМЫМИ, ──────────────── нет ─▶ ② in-process registry
     переподключаемыми между агентами?               тулы — функции в том же процессе, что loop;
     (= потребитель не только мой loop)               ноль протокола, вызов типизирован
        │
        да
        │
Q3 ─ Нужен ТИПИЗИРОВАННЫЙ вызов? (пожелание) ── да ──▶ ④ локальный MCP
        │                                              └ гейт: потребитель говорит в MCP из коробки
        нет
        │
        ▼
     ③ CLI
     └ гейт: у потребителя есть shell
```

Q1 и Q2 — последовательные развилки. Q3 устроен иначе: это **пожелание**, на которое сверху накладываются два **гейта-ограничения** (см. §4). Приписочки на листах ③ и ④ — это и есть гейты: если на выбранном листе условие не выполняется, тебя выбрасывает на соседний.

## 3. Четыре варианта по отдельности

**① in-process registry.** Тулы — обычные функции, зарегистрированные в loop'е; loop отдаёт их схемы модели и сам исполняет вызовы (паттерн ToolRegistry, [mcp-protocol.md §8](mcp-protocol.md)). Ноль сетевого транспорта; вызов типизирован — модель шлёт структурный объект аргументов, рантайм валидирует его до исполнения. Потребитель — твой и только твой loop. Самый дешёвый, быстрый, детерминированный. **Дефолт, пока не появился второй потребитель.**

Важно: модульность (общее ядро + разные наборы тулов под разные задачи) **не требует** ни CLI, ни MCP — это обычный шов на уровне кода (`get_tools()` у каждого набора). Протокольная граница не нужна, чтобы переключать наборы внутри своего же loop'а.

**② CLI.** Тул — исполняемая команда; потребляющий агент зовёт её строкой через свой shell-тул (ровно как `bash` зовёт любую команду в терминале). Тулы независимы и переиспользуемы любым агентом, у которого есть shell. Цена: вызов **stringly** — модель сама форматирует командную строку (кавычки, флаги) и парсит свободный stdout; и потребителю нужен и shell, и установленный бинарь. Зато CLI даёт то, чего нет у остальных, — **композицию**: пайпы, цепочки, комбинации с любым другим инструментом в одном действии.

**③ локальный MCP.** Тул живёт в отдельном процессе-сервере; потребитель — готовый MCP-клиент — *спавнит* его (запускает дочерним процессом) и общается по протоколу через stdio. Вызов типизирован, потребителю **не нужен shell**, тулы авто-обнаруживаются через handshake «перечисли свои tools» — это **zero wiring**: подключение к совместимому клиенту это строчка конфига, а не код (см. §6). Цена — отдельный процесс и протокольная обвязка. Механика — [mcp-protocol.md](mcp-protocol.md).

**④ remote MCP.** То же, но сервер хостится у тебя, а потребитель ходит к нему по HTTP (URL + токен). Единственный вариант, когда исполнитель — в чужом датацентре: ему нечего установить и нечем запустить твой локальный бинарь (см. §7). Тул и его данные живут на твоей стороне; потребитель **не ставит ничего** и исполняет не твой код, а лишь шлёт запросы — как к любому SaaS API.

Тулы, написанные для ① или ②, дёшево превращаются в MCP-сервер позже: схемы аргументов, которые ты и так пишешь (JSON Schema), — это ровно то, что MCP отдаёт на discovery. Обёртка в сервер — тонкий адаптер транспорта, а не переписывание. Поэтому правильно держать слой тулов transport-agnostic, а MCP добавлять, когда появится внешний потребитель.

## 4. Пожелание против ограничения

Это главная методологическая ловушка выбора между ③ CLI и ④ локальным MCP (т.е. когда тулы уже решено делать независимыми). В одну развилку легко смешать два разных **по природе** входа:

- **Пожелание** — что ты *хочешь*. Здесь оно одно: нужен ли типизированный вызов. Да → тянет к MCP; нет (хочу композицию, shell-native) → к CLI.
- **Ограничение** — что среда *позволяет*. Их ровно два: есть ли у потребителя **shell** (нужен для CLI); говорит ли потребитель **в MCP из коробки** (нужно для дешёвого локального MCP).

Смешивать их — ошибка моделирования: ограничение нельзя «хотеть», оно либо есть, либо нет. Правильная связь — **пожелание выбирает, ограничения вето́ируют**:

| | потребитель говорит в MCP из коробки | НЕ говорит |
|---|---|---|
| **shell есть** | оба на столе → **пожелание решает** (typed→④ MCP, нет→③ CLI) | CLI легко, MCP = трение → **③ CLI** (④ только если typed критичен) |
| **shell нет** | CLI нельзя → **принудительно ④ MCP** | ни то, ни другое из коробки → строить MCP-клиент-интеграцию (худший угол) |

Пожелание активно ровно в одном углу — shell есть **и** MCP-клиент есть. В остальных трёх ограничение перекрывает желание: нет shell → выбрасывает в MCP, даже если хотел CLI; нет MCP-клиента, но есть shell → выбрасывает в CLI, даже если хотел typed.

Та же логика «потребитель — готовый MCP-клиент» — это **ограничение, а не пожелание**. Быть готовым клиентом — факт о потребителе, его не выбирают. Поэтому в дереве (§2) на листе ④ это приписка-гейт, а не вопрос-развилка.

## 5. Типизированный вызов — что это и реально ли это выбор

**Что это.** Вызов типизирован, когда тул описан схемой (JSON Schema), и из этого следует: модель отдаёт **структурный объект** аргументов (а не строку), рантайм **валидирует** его до исполнения (не тот тип, нет обязательного поля, левый enum → отказ и просьба переделать, ещё до запуска кода), и обратно приходит **структура**, а не сырой текст. Один и тот же вызов:

```
Типизированно (MCP / in-process):
   модель →  { "name": "read_file", "input": { "path": "a.md", "lines": 50 } }
   рантайм проверил типы  →  вернул { "content": "...", "truncated": false }

Stringly (CLI / shell):
   модель →  tool read-file a.md --lines 50      ← строку, сама следит за кавычками/флагами
   обратно → сырой stdout, который модель парсит
   опечатка `--lines 50abc`  →  exit 2 + текст в stderr
```

**Когда нужен:** важна надёжность вызова (исчезают два класса отказов — кривой аргумент и кривой парсинг вывода); между тулами течёт структура (id, записи, списки); много параметров; задача скорится за точный формат вывода. **Когда не нужен (и shell даже лучше):** задача про композицию (одна shell-строка вместо пяти round-trip'ов); очень много тулов (грузить все схемы заранее дорого, через `--help` тянешь только нужное — у MCP та же отложенная загрузка даётся через tool search, §9); тулы — это стандартные утилиты, которые модель и так зовёт строкой.

**Реальный выбор или надуманный — честно.** Механизм реальный: типизированный function-calling и stringly-shell — разные интерфейсы с разными режимами отказа. Но как ось выбора он слабее, чем кажется:

1. **Это спектр, а не бинарь.** CLI подтягивается ближе к типизированному: `--json` на выходе, строгий парсер аргументов, структурные ошибки.
2. **Зазор надёжности сузился.** Современные модели неплохо и форматируют shell-строки, и парсят stdout; аргумент «typed надёжнее» был куда сильнее несколько лет назад.
3. **Часто он уже решён за тебя.** in-process registry *и так* типизирован. Если потребитель — твой loop, «нужен ли typed?» не выбор, а свойство, которое у тебя уже есть. Живым выбор становится только в ветке CLI-vs-MCP.

## 6. Что решает выбор MCP vs CLI: надёжность или охват

CLI и MCP различаются по двум осям, и какая из них решает — зависит от задачи, а не от тула.

- **Надёжность** (типизация, §5) — вес растёт, когда вызов должен срабатывать точно: задача скорится за поведение, между тулами течёт структура.
- **Охват** — вес растёт, когда один набор тулов надо воткнуть во много разных агентов. Здесь решает **zero wiring**: подключить MCP-тулы к совместимому клиенту — это строчка конфига, а не код. Клиент сам спрашивает «какие у тебя tools?» через handshake, получает схемы и показывает модели; ты не пишешь ни регистрации, ни маппинга интентов, ни парсинга вывода. У CLI этот слой собирается руками под каждого агента (каталог в AGENTS.MD, shell, парсинг) — это и есть «wiring», которого у MCP нет.

Важная оговорка: zero-wiring MCP держится на том, что **стандарт широко принят**, и работает только с агентами, которые говорят в MCP. Будь адопшн низким — преимущество испарилось бы, и CLI (shell понимает почти всё — это lowest common denominator) дал бы охват шире. Сегодня поддержка широкая, поэтому для «один набор тулов → много разных агентов» побеждает MCP. Но это **ставка на адопшн**, а не свойство протокола самого по себе.

Вывод: **ось выбора зависит от сценария.** Соревнование за точность → решает надёжность (typed). Универсальная интеграция с кем угодно → решает охват (zero wiring). Диаграмма (§2) одна, но вес пожелания и гейтов в Q3 смещается.

## 7. Co-location: почему у CLI нет «удалённого» режима

Структурная асимметрия, объясняющая Q1 в дереве:

- **CLI требует, чтобы код тула исполнялся внутри рантайма потребителя** — бинарь стоит у него, зовётся через его shell.
- **MCP позволяет коду тула жить за границей протокола** — потребитель не исполняет его инлайн, а *разговаривает* с ним (локальный subprocess по stdio или удалённый сервер по HTTP).

И CLI, и локальный MCP требуют **co-location**: исполнитель и тул на одной машине. Как только исполнитель оказывается в чужом датацентре, co-location нет — и остаётся только remote MCP.

**Пример.** Ты гоняешь задачу облачным агентом (он крутится на серверах вендора), а твои тулы и рабочие файлы — у тебя на VM.

- *CLI не работает*: у облачного агента shell есть, но это shell его контейнера, а не твоего. Он не видит ни твой бинарь, ни твои файлы — нет общей файловой системы, нет пути к твоей машине. Исполнять буквально нечем.
- *Локальный stdio MCP тоже не работает*: stdio предполагает, что клиент спавнит сервер дочерним процессом на той же машине. Облачный агент не может породить процесс на твоей VM.
- *Remote HTTP MCP — единственное*: ты поднимаешь сервер на VM с публичным адресом, агент ходит по HTTPS. Ноль установки на его стороне. Следствие: за границу протокола уезжает не только бинарь, но и его **данные** — рабочее состояние должно жить там же, где сервер.

Поэтому «отдать тулы удалённому потребителю без установки» умеет только MCP. Чтобы то же сделать на CLI, пришлось бы обернуть бинари в REST API и заставить потребителя написать клиента — то есть переизобрести нестандартный MCP. Стандарт выигрывает тем, что клиент потребителя **уже умеет** в него ходить (N+M интеграций вместо N×M).

## 8. CLI: где живёт описание тулов

Если выбрал ② CLI, описание раскладывается на два слоя с разными задачами:

- **`--help` каждой подкоманды — контракт «как вызвать».** Что делает, какие флаги/аргументы, что возвращает. Прямой эквивалент `inputSchema` у MCP-тула. В Python (argparse/click/typer) пишется один раз — в докстринге и определении аргументов — и утекает в `--help` само.
- **Каталог, который агент читает заранее — «что вообще есть».** Верхний `--help` (список подкоманд) плюс строчка в AGENTS.MD / системном промте: «тулы доступны через команду X; список — `X --help`, детали — `X <cmd> --help`». Этот слой у MCP даётся бесплатно handshake'ом; в CLI его надо собрать руками, иначе агент про тулы просто не узнает.

Разница MCP и CLI здесь — в доставке, не в содержании: MCP **пушит** описания в контекст модели заранее, CLI заставляет агента **притянуть** их через `--help`. Pull экономит токены (читаешь help только нужной тулы), но требует направить агента к каталогу — это и есть зазор, которого у MCP нет.

Оговорка: «MCP пушит всё» — поведение **по умолчанию**, а не потолок протокола. `defer_loading` и tool search дают MCP такой же pull, что у CLI (§9), — так что этот зазор на современном клиенте сужается.

## 9. Контекст-налог MCP: push по умолчанию, не по природе

**На дефолтном клиенте MCP грузит определения всех тулов в контекст наперёд — это реальный, названный налог; но это свойство конфигурации, а не потолок протокола.** Тот же MCP умеет ленивую подгрузку, что и pull у CLI (§8) или скилла.

По умолчанию большинство MCP-клиентов кладут все определения тулов прямо в контекст. При нескольких серверах это **«tool definition bloat»**: связка GitHub + Slack + Sentry + Grafana + Splunk съедает ~55k токенов на одних схемах *до* начала работы, а точность выбора тула заметно падает после 30–50 доступных тулов. В пределе — сотни тысяч токенов только на метаданные.

Лечится двумя механизмами, и оба не требуют отказа от MCP:

- **Tool search + `defer_loading`** (general availability — вышло из беты — с февраля 2026). Тулы с `defer_loading: true` **не попадают в system-prompt prefix**; модель видит только search-тул и 3–5 неотложенных «горячих» тулов, остальное ищет по запросу (regex или BM25) и подгружает только нужное. Экономия ~85%, prompt caching сохраняется. Для MCP — `mcp_toolset` с `defer_loading`.
- **Code execution** — модель пишет код против тулов, определения читаются по требованию (§11).

Вывод: ось «push vs pull» — это **«включена ли отложенная загрузка»**, а не «MCP против скилла». На современном клиенте (Claude Code сам дефёрит MCP-тулы) разрыв в контекст-эффективности между обёрткой в MCP и описанием API в скилле в основном закрывается. Поэтому «скилл чище по контексту» — устаревший довод, а не свойство протокола.

## 10. Скилл с описанием API vs обёртка в MCP

**Для одного агента, ходящего в API с авторизацией, дефолт — скилл; MCP выигрывает не из-за типизации, а из-за сокрытия токена и суженной аудируемой поверхности — и только когда эти свойства нужны.**

Что сравнивается. *Скилл* пакует не тул, а **контекст**: описание ручек, авторизации, паттернов; сами вызовы делает generic HTTP-тул агента (curl/fetch) — это stringly и pull (детали ручки подтягиваются по необходимости, §8). *MCP* пакует **typed-обёртку**: каждая ручка (курированно) — тул с JSON Schema, авторизация внутри сервера. Скилл — не пятый сосед четырёх вариантов §2: он пакует инструкции, а не исполняемый тул; это вырожденное «не паковать вообще».

Почему дефолт — скилл: один потребитель, ноль инфраструктуры, самый низкий уровень (§1).

Что меняет «вынести реализацию MCP за скобки». Допущение убирает **главный** довод за скилл (zero infra). Но всё прочее — **рантайм-свойства**, не стоимость постройки, и они остаются. Среди них перевес MCP даёт не типизация:

- **Сокрытие токена.** Авторизация на сервере; агент секрет не видит и не держит. В скилле агент сам формирует `Authorization` — токен в его контексте.
- **Суженная аудируемая поверхность.** MCP выставляет только обёрнутое; дискретный typed-вызов легко гейтить, логировать, ставить под подтверждение.
- *Типизация* (§5) — третий, слабый бонус, не причина: зазор надёжности узкий и убывающий.

Что скилл сохраняет даже при бесплатном MCP: **композицию** (сырой вызов пайпится с jq и другими тулами в одном действии, §3) и **полную поверхность** (агент дотянется до любой ручки и скомбинирует ad hoc, тогда как MCP даёт лишь обёрнутый срез).

Итог: перевес MCP — для пишущих/чувствительных/структурных нагрузок (writes, деньги, недоверенный агент, данные текут между тулами). Скилл конкурентоспособен для исследовательских/композиционных. Решает **характер работы с API**, а не типизация сама по себе.

## 11. Вторая ось: как агент зовёт тул — direct vs code-API

**Поверх «куда положить тул» есть ортогональная ось — «как агент его зовёт»: прямым typed-вызовом или написав код против тулов (code-API). Code-API выигрывает только под конкретную боль — много тулов плюс тяжёлый chaining — ценой песочницы и более слабого пер-вызовного контроля.**

**Код-API** — интерфейс, к которому обращаешься, **написав код** (вызов функции, импорт модуля), а не послав протокольное сообщение (tool-call JSON, HTTP-запрос). Слово «код» уточняет форму интерфейса, не его суть. Это **не альтернатива MCP на том же слое**: code-API — слой потребления поверх тех же тулов (в «code execution with MCP» тулы по-прежнему из MCP, просто модель пишет против них код). Реальные системы гибридны: 3–5 горячих тулов прямыми вызовами, хвост — через код.

Спектр из трёх ступеней:

| | прямой MCP | прямой MCP + tool search (§9) | code-API (code execution) |
|---|---|---|---|
| Тулы в контексте | все схемы наперёд | каталог + 3–5 горячих | один `execute_code` |
| Bloat **определений** | есть при масштабе | вылечен | вылечен |
| Промежуточные данные через контекст | текут | текут | **не текут** (в песочнице) |
| Вызов | typed, валидируется | typed | произвольный код |
| Нужна песочница | нет | нет | **да** |
| Пер-вызовный аудит/гейтинг | простой | простой | сложнее |

Где исполняется код: в **песочнице** — изолированном код-рантайме (интерпретатор Python/TS), **не в субагенте**. Субагент — это другая LLM со своим контекстом; песочница — детерминированный исполнитель без LLM внутри. Именно поэтому промежуточные данные (отфильтровал 10k строк, вернул 5) остаются в переменных рантайма и не текут через контекст — будь там субагент, всё снова шло бы через токены.

Code-API не безусловно лучше: песочница — это инфраструктура и поверхность безопасности (исполняешь код, написанный моделью); пер-вызовный гейтинг слабее; на простом одиночном вызове code-gen дороже одного `tool_use`; упирается в способность модели писать корректный код. Anthropic называет традиционный путь лучшим при <10 тулов, которые все используются часто.

Эвристика: начни с прямого вызова; упёрся в bloat определений → включи tool search; боль не в определениях, а в том, что между тулами течёт много промежуточных данных при тяжёлом chaining → переходи на code-API. Лезь вправо по спектру под конкретную боль, не авансом.

## 12. Частые ошибки

- **«REST API + документация = то же, что MCP».** Механизм тот же (описал тулы → агент вызывает), но MCP — *стандарт*. Ценность не в механизме, а в том, что клиент потребителя уже умеет в него ходить: N+M интеграций вместо N×M. Прямая аналогия — LSP, с которого MCP и срисован. См. [mcp-protocol.md](mcp-protocol.md).
- **«CLI = раздать утилиты» путают с «CLI запускает агента».** Это разные вещи под одним словом «CLI»: бинарь может *быть самим агентом* (внутри LLM + loop + тулы — его и запускают), а может быть *пассивным тулом*, который зовёт чужой loop. Раздаёшь наружу — второе.
- **«Не мой loop → значит MCP».** Нет: CLI тоже обслуживает не-мой-loop (любой агент с shell зовёт твой бинарь). «Не мой loop» отделяет только in-process от остального. MCP от CLI различает другое — *должен ли код исполниться в рантайме потребителя* (CLI) *или может стоять за протоколом* (MCP).
- **«MCP универсальнее → всегда лучше».** На двух осях (локально-или-удалённо; без shell) MCP правда доминирует. Но универсальность — это **цена**: ради optionality, которую можешь не использовать, ты платишь сервером и протоколом. И есть ось, где CLI универсальнее, — композиция. Бери универсальность, когда список потребителей её требует, а не авансом.
- **«MCP — только для удалённого хостинга».** Нет: локальный MCP (stdio) нужен для готовых клиентов (Claude Desktop, Cursor) даже без всякой сети — ради авто-discovery и работы без shell.
- **«MCP по дизайну забивает контекст».** Нет: это поведение дефолтного клиента, а не протокола. `defer_loading` + tool search и code execution дают MCP ленивую подгрузку (§9).
- **«Скилл vs MCP решает типизация».** Слабый довод (§5). Если вынести реализацию за скобки, перевес MCP даёт сокрытие токена и суженная аудируемая поверхность, а не typed-вызов (§10).
- **«Code-API — альтернатива MCP».** Нет: это слой потребления поверх тех же тулов — ортогональная ось «как зовёт», а не «куда положить» (§11).
- **«Code execution исполняется в субагенте».** Нет: в песочнице — код-рантайме без LLM внутри. Субагент — другая LLM; песочница «считает», субагент «думает» (§11).
