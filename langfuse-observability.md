# Observability агентов (на примере Langfuse)

**Observability агента — это способность реконструировать снаружи, что агент делал, почему принимал решения и где сломался, не влезая в его код.** Применительно к агентам это значит записывать каждый LLM-вызов (вход, выход, токены, стоимость) и каждый tool-вызов (имя, аргументы, результат), связывать их в иерархию и показывать UI-дерево. **Langfuse** — open-source, self-hostable инструмент этого класса; ниже разбирается его модель данных на сквозном примере — sequential tool-use агенте (линейный цикл «запрос к модели → инструмент → запрос → …»).

Сквозной пример здесь — агент PAC: автономный агент, решающий бенчмарк-задачи через последовательность вызовов инструментов. Все его термины (run, task, trial, turn, tool call) вводятся по ходу как словарь примера, а не как общие понятия. Этот файл смотрит на агента **снаружи** — когда он уже работает, и нужно понять, что он делает, почему так решает и где сломался. Файл-компаньон [agentic-loop.md](agentic-loop.md) смотрит на того же агента **изнутри** — как устроен его цикл. Смежная тема — измерение качества: [agent-evals.md](agent-evals.md).

## 1. Зачем нужна observability

**Без observability агентный цикл невозможно отлаживать: виден только финальный скор (например, 0), но не его причина.** Агент неправильно понял инструкцию? Выбрал не тот tool? Получил кривой результат и не заметил? На каком из 23 шагов сломался? Без записи внутренних шагов ответа нет.

Агент — чёрный ящик с большим числом внутренних движений. За одну задачу он может сделать 20 LLM-вызовов, каждый потребляет токены, возвращает текст, принимает решение о следующем шаге. Между LLM-вызовами агент дёргает инструменты — читает файлы, ходит по HTTP, меняет workspace. На выходе — ответ, правильный или нет. Observability вскрывает этот ящик: каждый LLM-вызов и tool-вызов записывается, связывается в иерархию и отображается деревом.

Langfuse специализирован именно под LLM-системы (не general-purpose observability), self-hostable (данные остаются у тебя) и имеет богатую модель данных под разные паттерны агентов.

## 2. Модель данных: контейнеры и observation'ы

**Модель данных Langfuse — это контейнеры группировки (Trace, Session) плюс элементы работы внутри trace'а (observation'ы).**

### Контейнеры группировки

**Trace** — верхнеуровневый логический контейнер «одной логической операции», которую наблюдают как единое целое. У trace есть имя, timestamp, input, output, tags, metadata, session_id, user_id. В главной таблице Langfuse UI каждому trace соответствует одна строка. Собственного времени работы у trace нет — start/end он выводит из вложенных observation'ов.

**Session** — необязательная группировка связанных trace'ов по общему `session_id`. Диалог из 10 сообщений в чат-боте — это 10 отдельных trace'ов с одним session_id. Session создаётся неявно через session_id, но в Langfuse это полноценный агрегат: отдельная страница сессии в UI с воспроизведением всей последовательности взаимодействий (session replay) и session-level scores (оценки уровня сессии).

### Элементы работы внутри trace'а: Observation и его типы

**Observation** — базовая сущность шага внутри trace'а, дискриминируемая полем-типом (`type`). Общие поля: `id`, `parent_id` (для дерева), `name`, `start_time`, `end_time`, `input`, `output`, `metadata`, `level`. Чтобы создать observation, указывают конкретный тип.

Типов observation в Langfuse десять: `event`, `span`, `generation`, `agent`, `tool`, `chain`, `retriever`, `evaluator`, `embedding`, `guardrail` (см. docs Langfuse, Observation Types). В сквозном примере используются четыре из них — они равноправные специализации одной сущности Observation, различаемые полем `type`, а не подклассы друг друга:

- **Span** — универсальный «интервал работы с началом и концом». Семантически нейтрален; годится для группировки и обозначения фаз: «фаза исследования workspace», «retry-блок», «вся задача целиком» (root).

- **Generation** — один LLM round-trip (один HTTP-вызов к Anthropic/OpenAI). Добавляет к общим полям специфические: `model`, `model_parameters`, `usage_details` (токены), `cost_details`, `completion_start_time`, а также `prompt` — объект управляемого промпта из Prompt Management (на уровне ingestion/REST-API то же связывание зовётся `prompt_name`). В UI рендерится с блоком токенов и стоимости.

- **Tool** — один tool-вызов. Input — аргументы, output — результат. В UI — с tool-иконкой.

- **Event** — мгновенное событие в одной точке времени. У него нет `end_time`, только timestamp. Применяется редко — пометить момент без длительности.

Остальные шесть типов (`agent`, `chain`, `retriever`, `evaluator`, `embedding`, `guardrail`) — такие же равноправные специализации Observation для своих сценариев; выбор четырёх выше — особенность конкретного примера, а не полный список.

## 3. Две ортогональные оси вложенности

**В Langfuse две ортогональные оси вложенности: типовая иерархия (какой это вид observation) и runtime-дерево (где он в parent-child структуре). Их путают, потому что обе называются «вложенностью».**

### Ось 1: типовая иерархия (тип vs сущность)

Статическая структура, заданная в схеме данных и в SDK. Отвечает на вопрос «к какому семейству принадлежит эта штука» — это «генетика».

```
Observation                        ← базовая сущность, дискриминируется полем type
├── Span          (type="span")
├── Generation    (type="generation")
├── Tool          (type="tool")
├── Event         (type="event")
└── … (agent, chain, retriever, evaluator, embedding, guardrail)
```

Все типы — равноправные специализации: общие поля наследуются от Observation, каждый добавляет свои.

### Ось 2: runtime-дерево (parent-child)

Динамическая структура, строящаяся в runtime через `parent_id`. Отвечает на вопрос «где в данной конкретной структуре этот инстанс находится» — это «генеалогия».

```
Trace "task t042"
└── [root] Span "task t042"
    ├── Generation "turn-0"
    ├── Tool "list_files"
    ├── Generation "turn-1"
    ├── Tool "read_file"
    ├── Generation "turn-2"
    └── Tool "answer"
```

### Следствие: типовая общность не определяет runtime-позицию

Оси независимы. Один и тот же тип может занимать любую позицию в дереве:

- Generation **может быть** корнем trace'а (без parent) — one-shot классификация.
- Generation **может быть** внутри Span'а — стандартный случай в агентном цикле.
- Generation **может быть** внутри Tool'а — если tool сам зовёт LLM (например, `summarize_document`).
- Span **может быть** внутри другого Span'а — вложенные фазы задачи.

Langfuse не накладывает ограничений на то, какой тип может быть родителем какого. Типовая иерархия отвечает «что это за вещь», runtime-дерево — «где она находится»; ни одно измерение не подчиняется другому.

## 4. Trace vs Span

**Trace — логический контейнер без собственного времени работы; Span — observation с конкретными start/end внутри trace'а. Они сливаются визуально, потому что SDK создаёт trace и его корневой span одним вызовом.**

**Trace — это контейнер.** Есть идентичность (trace_id, имя, session_id, user_id, tags, metadata), но нет собственного времени работы — start/end выводятся из вложенных observation'ов. Trace — логическая обёртка, под которой живут один или более observation'ов.

**Span — это observation конкретного типа.** У него есть реальные start/end и конкретная работа. Span всегда находится внутри trace'а (или другого observation'а).

### Почему они «сливаются»

В Langfuse SDK нет отдельного метода `create_trace()`. Когда создаётся root-observation (на клиенте, без родителя), SDK одновременно создаёт и trace, и первый (корневой) span внутри него. Поэтому новичку кажется, что trace и root span — одно и то же. Это не так: trace — контейнер, root span — первая работа внутри него, которая по совпадению часто представляет trace целиком.

Чтобы input/output корневого span'а отображались как поля строки trace в главной таблице, их «протягивают» на уровень trace. Исторически это делал метод `set_trace_io()`, но в актуальном SDK он помечен deprecated. Актуальный способ — задавать input/output прямо на корневом observation (через `input=`/`output=` при создании или через `update()`); для trace-уровневых атрибутов есть отдельный механизм (см. §8).

### У trace'а может быть больше одного корневого observation'а

Trace не обязан иметь ровно один root: например, HTTP-запрос может породить несколько параллельных веток, и все они — top-level observation'ы одного trace без общего родителя. Это легитимная структура.

Паттерн «один root span на trace» — выбор инструментирующего. Для задачи с линейным циклом шагов он даёт самое читаемое дерево, но это выбор, а не ограничение Langfuse.

## 5. Маппинг концепций агента на примитивы Langfuse

**Маппинг концепций агента на примитивы Langfuse — это инженерное решение инструментирующего, а не навязанная или единственно верная схема.** У Langfuse нет понятий run/task/trial/turn/tool call — только примитивы из §2.

Sequential tool-use агент (сквозной пример, PAC) оперирует своими терминами:

- **run** — прогон бенчмарка целиком;
- **task** — одна задача;
- **trial** — серверное представление задачи в рамках конкретного run'а (то, что бенчмарк-харнес — серверная среда, выдающая задачу и принимающая ответ, — заводит на свою сторону);
- **turn** — один LLM-вызов в agent-loop'е;
- **tool call** — один вызов инструмента.

Один разумный способ положить это на примитивы Langfuse:

| Концепция примера | Примитив Langfuse | Почему |
|---|---|---|
| **task** (trial) | **trace** (с root `span`) | Задача — одна логическая единица работы с чётким началом и концом. Trace — естественный контейнер. |
| **turn** (LLM round-trip) | **generation** внутри root span'а | Один turn = один LLM API-вызов. Generation моделирует это с автоподсчётом токенов и стоимости. |
| **tool call** | **tool** внутри root span'а | Прямой маппинг. |
| **run** | **session_id** на trace'ах | Session — нативный механизм группировки trace'ов. Каждый trace одного прогона получает `session_id = run_id`; в UI прогоны видны в разделе Sessions. |
| **account / identity** | **user_id** на trace'ах | Разделение trace'ов по пользователям/аккаунтам. |
| **benchmark name, model name, api-key prefix** | **tags** на trace'ах | Ортогональные разрезы для быстрой фильтрации в UI. |
| **прочие тех. детали** | **metadata** на trace'ах и observation'ах | Всё, что не укладывается в остальное: harness_url, trial_id, git commit, stop_reason. |

Langfuse не навязывает этот маппинг — возможны и другие. Критерий выбора уровня агрегации простой: **один trace = одна логическая операция.** Отсюда и обоснование `trace = task`, и почему крайности плохи:

- **trace = run** — тысячи observation'ов в одном дереве (десятки задач × десятки LLM-вызовов): UI не справится, нельзя сравнить одну задачу между прогонами (она в разных тяжёлых trace'ах), и сам trace перестаёт быть «одной операцией».
- **trace = один LLM-вызов, task = session** — session не поддерживает иерархию observation'ов, и структура задачи целиком теряется.

`trace = task` укладывается ровно в «одну логическую операцию» — отсюда выбор.

## 6. Turn — концепция агента, не Langfuse

**Turn — понятие агента, не Langfuse: один turn = один round-trip к LLM API (от отправки запроса до получения ответа со `stop_reason`).** В сквозном примере turn маппится на один generation.

Ключевая тонкость: **tool-вызовы живут между turn'ами, а не внутри.** Выполнение инструмента — работа клиента, не часть LLM-вызова:

```
turn N start         → отправляем модели контекст
turn N end           → получаем ответ со stop_reason=tool_use
[между turn'ами]     выполняем tool, получаем результат
turn N+1 start       → отправляем результат tool'а + новый контекст
turn N+1 end         → получаем ответ со stop_reason=end_turn
```

Поэтому tool-observation'ы и generation-observation'ы лежат **сиблингами** под root span'ом задачи, упорядоченные по времени:

```
task t042 [span]
├── turn-0 [generation, stop_reason=tool_use]
├── list_files [tool]
├── turn-1 [generation, stop_reason=tool_use]
├── read_file [tool]
├── turn-2 [generation, stop_reason=end_turn]
```

### Почему у Langfuse нет понятия «turn» и «agent loop»

Langfuse намеренно низкоуровневый: даёт примитив `generation` и позволяет собрать из него любой паттерн. Встроенного понятия «agent loop» или «последовательность turn'ов до end_turn» нет, потому что единого паттерна не существует:

- **one-shot classifier** — один generation, ноль инструментов, нет цикла;
- **sequential tool-use** (наш пример) — линейный цикл generation → tool → generation → … → end_turn;
- **multi-agent** — несколько независимых циклов в одном trace'е.

(Другие паттерны — RAG, planning, ReAct, Reflexion, tree-of-thoughts — собираются из тех же примитивов; их устройство — тема [agentic-loop.md](agentic-loop.md).) Захардкоженный «agent loop» как примитив заточил бы Langfuse под один стиль. Поэтому Langfuse — строительный материал, а agent loop — то, как ты этот материал укладываешь.

## 7. Stop reason — концепция LLM API, не Langfuse и не агента

**`stop_reason` — поле ответа LLM API, объясняющее, почему модель прекратила генерацию. Это контракт провайдера (Anthropic Messages API; у OpenAI аналог — `finish_reason`), а не понятие Langfuse и не понятие агента.**

Значения Anthropic:

| Значение | Что значит |
|---|---|
| `end_turn` | Модель завершила ответ естественно. |
| `tool_use` | Выдан `tool_use` блок, модель ждёт результата инструмента. |
| `max_tokens` | Упёрлись в лимит токенов вывода, ответ обрезан. |
| `stop_sequence` | Вывод совпал со стоп-последовательностью из запроса. |
| `refusal` | Отказ по политике безопасности (возвращается как обычный HTTP 200). |
| `pause_turn` | Модель приостановилась на длительной серверной операции (например, web search). |
| `model_context_window_exceeded` | Ответ заполнил контекстное окно модели. |

(Источник значений: docs Anthropic, handling stop reasons.)

### Модель сама про `stop_reason` не знает

Модель — авторегрессионный генератор токенов; она не «знает», что в HTTP-ответе есть поле `stop_reason`. `stop_reason` — ярлык, навешиваемый inference-сервером **извне** по наблюдаемым сигналам: модель эмитировала end-of-turn токен? достигли max_tokens? совпала stop_sequence? появился tool_use блок? Сервер ловит сигнал, останавливает генерацию и кладёт в ответ соответствующее значение. В обучении модель училась эмитировать правильные сигналы остановки, но оперирует токенами, а не полем `stop_reason`.

### Где `stop_reason` обычно хранят

Один из способов — класть `stop_reason` в `metadata` соответствующего generation'а (так делает реализация observer'а в сквозном примере). Открыв generation в UI, видишь `metadata.stop_reason: "tool_use"` или `"end_turn"`. Это позволяет, например, отфильтровать все задачи, завершившиеся по `max_tokens` (агент не успел закончить) — через metadata-фильтр.

## 8. Механизмы группировки

**У Langfuse несколько независимых первоклассных способов группировать trace'ы — они работают одновременно, каждый для своей задачи.**

| Механизм | Что это | Как заполняется | Когда использовать |
|---|---|---|---|
| **trace_id** | Уникальный id trace'а | Auto-generated | Точный идентификатор одного trace'а |
| **session_id** | Строка, объединяющая trace'ы в сессию | trace-атрибут (см. ниже) | Логические сессии: беседа чата, прогон бенчмарка, сессия отладки |
| **user_id** | Идентификатор пользователя/аккаунта | trace-атрибут | Атрибуция «чья это работа»: пользователь, аккаунт, разработчик |
| **environment** | Среда исполнения | Из SDK-конфигурации | Разделение `prod` / `staging` / `dev` |
| **tags** | Массив коротких строк-меток | trace-атрибут | Плоские ортогональные разрезы: бенчмарк, модель, эксперимент |
| **metadata** | Свободный JSON-словарь | Параметр observation'а | Технические детали, ссылки, id из внешних систем |
| **level** | Severity (`DEBUG`/`DEFAULT`/`WARNING`/`ERROR`) | На observation'е | Визуальная пометка проблемных observation'ов |
| **scores** | Оценки качества | Через API скоринга (`score_trace`) | Метрики: accuracy, human rating |

Trace-уровневые атрибуты (`user_id`, `session_id`, `tags`, `metadata`, `version`, имя trace) в SDK v3 задаются не прямыми kwargs создания observation'а, а через механизм распространения атрибутов `propagate_attributes(...)` (контекст-менеджер, навешивающий атрибуты на все observation'ы блока) либо обновлением корневого observation. Сам вызов создания observation принимает `name`, `as_type`, `input`, `output`, `metadata`, `version`, `level`, `status_message` (плюс generation-поля). (Сверьтесь с docs Langfuse, SDK instrumentation.)

### Tags vs metadata

- **Tags** — плоский многомерный namespace коротких строк (`model:haiku-4.5`, `bench:pac1-prod`). Langfuse автоматически собирает уникальные значения tag'ов и показывает их как фильтр-чипсы. Идеально для быстрой визуальной фильтрации одним кликом.
- **Metadata** — иерархический JSON-словарь: вложенные объекты, массивы, числа. Для точных технических данных и извлечения. Фильтрация возможна (`metadata.run_id = "run-xxx"`), но менее визуальна, чем клик по tag-чипсу.

Практическое правило: видеть и фильтровать одним кликом → tag; хранить точные технические данные → metadata. У одного trace'а могут быть оба.

### Пример заполнения полей (иллюстративно)

```
Trace "task t042"
├── session_id  = "run-XXXXXXXX"                       ← run_id прогона
├── user_id     = "user@example.com"                   ← владелец аккаунта
├── environment = "default"
├── tags        = [
│                   "bench:bitgn/pac1-prod",
│                   "model:claude-haiku-4-5",
│                   "key:abcd"
│                 ]
├── metadata    = {
│                   "trial_id": "vm-03nm...",
│                   "harness_url": "https://vm-...example.com",
│                   "commit": "a608a9c",
│                   "task_id": "t042",
│                   "benchmark_id": "bitgn/pac1-prod"
│                 }
└── ...
```

## 9. Диагностика по UI

**Несколько полей trace'а читаются как быстрые диагностические сигналы прямо из таблицы UI, без захода внутрь.**

### Пустой output — сигнал падения

Input корневого observation'а ставится при его создании, а output — только при успешном завершении агентного цикла (в завершающем callback'е). Если агент упал до завершения — исключение, ошибка провайдера, timeout — output остаётся пустым при заполненном input.

**Пустой output = trace не закрылся корректно.** Удобный визуальный индикатор: задачи с пустым output подозрительны. Одно поле, видно моментально в таблице, без захода в metadata.

### Observation level — severity, не качество

Поле `level` (`DEBUG`/`DEFAULT`/`WARNING`/`ERROR`) — это severity-флаг, не оценка качества. Если уровень явно не выставляется, всё уходит в `DEFAULT`. Зелёные бейджи означают лишь отсутствие Python-исключения внутри цикла, **не** правильность решения задачи.

### Качество — это scores, отдельное поле

Для фактической правильности есть отдельное поле — **scores** (записываются через API скоринга: accuracy, human rating и т.п.). На странице trace'а — секция `Scores`. Для блайнд-прогонов — режима, где скоры намеренно скрыты, чтобы участник не подгонял агента под ответы, — `score=None` и секция пуста; это нормально.

### Найти все trace'ы одного прогона

Пока `session_id = run_id` не задан, все trace'ы одного прогона ищутся через metadata-фильтр в UI:

1. В панели фильтров над таблицей traces выбери `metadata`.
2. Ключ: `run_id`.
3. Оператор: `=` или `contains`.
4. Значение: полный id, например `run-XXXXXXXX`.

Работает только для trace'ов, созданных после того, как observer начал писать `run_id` в metadata. После перехода на `session_id` появляется раздел Sessions, где каждый прогон — отдельная строка: клик показывает все trace'ы прогона без фильтров.

## 10. Langfuse в экосистеме

**Langfuse выигрывает там, где нужны open-source + self-hosting (данные у тебя) и богатая модель данных под агентные сценарии; альтернативы сильны в своих нишах.**

- **LangSmith** (от LangChain) — закрытый, LangChain-first; self-host только на Enterprise (BYOC), не open-source. Observability при этом framework-agnostic: трейсит OpenAI SDK, Anthropic SDK, LlamaIndex, кастомный код, поддерживает OpenTelemetry-ingest.
- **W&B Weave** — часть платформы Weights & Biases; удобен командам, уже живущим на W&B. Тяжеловат, если приходишь только ради LLM-observability.
- **Arize Phoenix** — open-source, self-hostable, на OpenTelemetry; изначально силён в RAG/eval, но general-purpose (агенты, чат-боты, копайлоты).
- **Helicone** — proxy-based: перехватывает HTTP-запросы к провайдерам без изменений в коде. Проще ставится, но даёт более плоскую картину — не видит tool-вызовов и структуры агентного цикла.
- **TruLens** — фокус на evaluation и human feedback, observability вторична.

Langfuse подходит, когда нужно: open-source + self-hosting; богатая модель данных под агентные сценарии; активное развитие и сообщество; гибкое моделирование своей системы через примитивы.

## 11. Шпаргалка

Краткий указатель ключевых различений с отсылкой к главам:

1. **Langfuse — набор строительных блоков** (trace, типы observation, механизмы группировки), а не готовый фреймворк «agent observability». Что на что мапить — решает инструментирующий. → §5
2. **Trace = одна логическая операция.** В примере — одна задача. → §5
3. **Observation — базовая сущность с полем-типом.** Span, Generation, Tool, Event и ещё шесть типов — равноправные специализации, не подклассы друг друга. → §2
4. **Типовая иерархия ⊥ runtime-дерево.** Тип не определяет позицию в parent-child структуре. → §3
5. **Trace ≠ Span.** Trace — контейнер без своего времени; root span — первая работа внутри, создаётся тем же вызовом. → §4
6. **Turn — концепция агента**, не Langfuse: один turn = один LLM round-trip = одна generation; tool-вызовы между turn'ами. → §6
7. **`stop_reason` — концепция LLM API**, не Langfuse и не агента: ярлык от inference-сервера по сигналам модели. → §7
8. **Группировки независимы.** session_id, user_id, environment, tags, metadata работают одновременно. → §8
9. **Tags vs metadata:** tags — быстрая визуальная фильтрация (плоский namespace); metadata — точные тех. данные (иерархический JSON). → §8
10. **Пустой output = trace не закрылся.** Level — severity, не качество; качество — в scores. → §9
