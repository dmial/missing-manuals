# MCP: Model Context Protocol

Заметки о MCP — стандартном протоколе для подключения инструментов к AI-агентам. Построен на базе JSON-RPC (см. [api-protocols.md](api-protocols.md)).

---

## 1. Зачем MCP

LLM-агенту нужно взаимодействовать с внешним миром: читать почту, смотреть календарь, работать с файлами. Классический подход — писать HTTP-клиент для каждого API. MCP — альтернатива: стандарт, позволяющий LLM вызывать инструменты (tools) через унифицированный интерфейс. Вместо написания HTTP-клиента для каждого API, MCP-сервер описывает свои возможности, а LLM сам выбирает и вызывает нужные.

---

## 2. Главный инсайт: LLM не знает про MCP

MCP — это протокол **между хостом и сервером**, а не между моделью и сервером:

```
┌─────────┐     tool_use/tool_result      ┌──────────┐     JSON-RPC      ┌────────────┐
│   LLM   │  ◄──────────────────────────►  │   Хост   │  ◄─────────────►  │ MCP-сервер │
│         │   (стандартный API модели)      │(Claude   │   (stdio/SSE)    │(Read AI,   │
│         │   модель НЕ знает про MCP      │ Code)    │                   │ GitHub...) │
└─────────┘                                └──────────┘                   └────────────┘
```

Хост (например, Claude Code) делает всю работу:

1. Спрашивает MCP-сервер: «какие у тебя tools?»
2. Получает список с названиями и JSON Schema параметров
3. Превращает их в **обычные tool definitions** для модели
4. Когда модель делает tool_call — транслирует его в JSON-RPC вызов к MCP-серверу
5. Ответ сервера конвертирует обратно в tool_result для модели

**Модель думает, что вызывает обычные tools.** Она не знает, что за ними стоит MCP-сервер, REST API, база данных или файл на диске. Поэтому даже старая модель, работавшая до изобретения MCP, прекрасно бы с ним работала — при условии, что хост умеет конвертировать MCP tools в формат tool definitions этой модели.

## 3. Архитектура: хост, клиент, сервер

```
┌─────────────────────────────────────────┐
│                 Хост                     │
│  (Claude Code / Claude Desktop / IDE)    │
│                                          │
│  ┌─────────────┐    ┌─────────────────┐  │
│  │  LLM-слой   │    │  MCP-клиент(ы)  │  │
│  │  (вызов     │◄──►│  (библиотека,   │  │
│  │   Claude    │    │   поддерживает  │  │
│  │   API)      │    │   JSON-RPC)     │  │
│  └─────────────┘    └────────┬────────┘  │
│                              │            │
└──────────────────────────────┼────────────┘
                               │ stdio / HTTP
                    ┌──────────▼──────────┐
                    │    MCP-серверы       │
                    │  (Read AI, GitHub…)  │
                    └─────────────────────┘
```

- **Хост** — приложение целиком (Claude Code)
- **MCP-клиент** — компонент внутри хоста, умеющий общаться по JSON-RPC
- **MCP-сервер** — внешний процесс/сервис, предоставляющий tools

Один хост содержит **несколько инстансов** MCP-клиента — по одному на каждый подключённый сервер. Код класса один, но каждый инстанс поддерживает своё соединение (stdio pipe или HTTP). Как три вкладки браузера — один код `fetch`, но три независимых HTTP-соединения.

MCP-клиент **не специфичен для сервера**. Протокол стандартизирован: один и тот же клиент внутри Claude Code общается с Read AI, GitHub и любым другим MCP-сервером. Anthropic предоставляет референсные SDK (Python: `mcp`, TypeScript: `@modelcontextprotocol/sdk`), но можно написать с нуля по спецификации.

## 4. Транспорты MCP

MCP = JSON-RPC, завёрнутый в один из двух транспортов:

| Транспорт | Когда | Как работает |
|---|---|---|
| **stdio** | MCP-сервер локальный | Хост запускает процесс, пишет JSON в stdin, читает из stdout |
| **Streamable HTTP** | Удалённый сервер | Клиент шлёт POST, сервер отвечает JSON или SSE-потоком |

### Эволюция транспорта

**Старый (2024, deprecated):** два эндпоинта — `GET /sse` для приёма событий, `POST /message` для отправки запросов.

**Новый (2025, Streamable HTTP):** один эндпоинт — `POST /mcp`, сервер сам решает, ответить обычным JSON или SSE-потоком.

---

## 5. Что возвращает MCP-сервер

JSON-RPC сообщения. Полезная нагрузка внутри — массив блоков `content`:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [
      { "type": "text", "text": "У вас 3 непрочитанных письма..." },
      { "type": "image", "data": "/9j/4AAQ...", "mimeType": "image/jpeg" }
    ]
  }
}
```

Типы блоков:
- `text` — текст (самый частый)
- `image` — base64-картинка (см. [api-protocols.md](api-protocols.md), раздел 1.5 про base64)
- `resource` — ссылка на MCP-ресурс

## 6. Когда REST API, когда MCP

Ключевой вопрос: **кто принимает решение, какой инструмент вызвать?**

### REST API — когда логика фиксирована

Ты точно знаешь шаги и пишешь их в коде:

```python
# Ежедневный отчёт: шаги известны заранее
async def daily_report():
    meetings = await readai.get("/meetings", params={"date": yesterday})
    for m in meetings:
        transcript = await readai.get(f"/meetings/{m['id']}/transcript")
        summary = await claude.summarize(transcript)
        await slack.post(channel, summary)
```

Скрипты, cron-задачи, ETL-пайплайны, интеграции — REST API напрямую.

### MCP — когда решение принимает LLM

Множество возможных запросов слишком велико для `if/else`:

```
Пользователь: "Что решили по бюджету на вчерашнем созвоне с финансами?"

LLM должна сама:
1. Найти вчерашние встречи → list_meetings
2. Отфильтровать по теме "финансы"
3. Достать транскрипт → get_meeting_by_id
4. Найти обсуждение бюджета
5. Сформулировать ответ
```

Чат-боты, ассистенты, агенты — MCP (или tool_use).

### Гибридный вариант — самый частый

В одном приложении оба подхода:

| Компонент | Подход |
|---|---|
| Cron-задача (ежедневный отчёт) | REST API напрямую |
| Чат-интерфейс (вопросы пользователя) | LLM + MCP |
| Дашборд (статистика) | REST API напрямую |

## 7. Кто пишет MCP-серверы

Если поставщик дал только REST API — MCP-сервер пишешь **ты** (или используешь написанный сообществом):

| Кто написал | Пример |
|---|---|
| **Поставщик** | Read AI, GitHub, Stripe — дают готовый MCP-сервер |
| **Сообщество** | npm/pip пакеты вроде `mcp-server-postgres` |
| **Ты** | Если API есть, а MCP-сервер никто не написал |

Написать MCP-сервер — это обёртка над REST API:

```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("my-readai")

@mcp.tool()
async def list_meetings(date: str) -> str:
    """List meetings for a given date."""
    resp = await httpx.get(
        "https://api.read.ai/meetings",
        params={"date": date},
        headers={"Authorization": f"Bearer {TOKEN}"}
    )
    return resp.text
```

## 8. Когда MCP — оверкилл

### Пример: ToolRegistry вместо MCP

Если ты пишешь агента, который использует Claude API и набор tools известен заранее, можно обойтись без MCP. Пример — паттерн ToolRegistry:

```
Что делает MCP-клиент + сервер:          Что делает ToolRegistry:

Сервер хранит определения tools           get_definitions() → JSON schemas
Сервер содержит реализации tools          execute(name, args) → результат
Клиент запрашивает список tools           get_definitions() — тот же вызов
Клиент шлёт JSON-RPC серверу              execute() — прямой вызов функции
Клиент возвращает результат LLM           Возвращает результат LLM
```

Те же шаги, но без JSON-RPC и сетевого транспорта — потому что клиент и сервер живут в одном процессе. Замена mock → production API — замена одного модуля.

### Когда MCP оправдан

| Сценарий | MCP нужен? |
|---|---|
| Один агент, один набор tools | Нет — ToolRegistry достаточно |
| 10 агентов используют одни tools | Да — один сервер, 10 клиентов |
| Tools подключаются в рантайме из разных источников | Да — динамическое обнаружение |
| Ты хочешь дать свои tools другим | Да — стандартный протокол |

MCP решает проблему **интероперабельности**: один сервер — много клиентов, или один клиент — много серверов. Если у тебя один клиент и один сервер — MCP как USB-хаб для одного устройства.

CLI/shell — третий способ упаковки тулов, не рассмотренный здесь. Полная карта выбора (in-process / CLI / локальный MCP / remote MCP) с разделением «пожелание vs ограничение» — в [tool-packaging.md](tool-packaging.md).

Даже если поставщик предложит MCP-сервер, а у тебя уже работает прямая интеграция через ToolRegistry — **не обязательно переключаться**. Свой адаптер даёт больше контроля: логирование, кэширование, retry, подмена отдельных tools. На конкурсе важна скорость и контроль, а не красота архитектуры.

## 9. Практический кейс: почему MCP не заменяет прямой API в скриптах

### Ситуация

Есть проект — система подготовки коммерческих предложений. Контекст для каждого предложения собирается из нескольких источников: CRM (Attio), транскрипты встреч (Read.ai), переписка с клиентом (Gmail). Скрипт `sync.py` автоматически синхронизирует эти данные в markdown-файлы.

Задача: добавить синхронизацию email. В Claude Code уже подключён Gmail MCP-сервер — через него можно искать и читать почту прямо в чате. Возникает логичный вопрос: **можно ли использовать это же MCP-подключение из `sync.py`?**

### Короткий ответ: нет

### Почему: MCP-подключение привязано к хосту, а не к тебе

Схема того, что происходит, когда ты читаешь почту через Claude Code:

```
┌──────────────────────────────────────────────────────────────────┐
│                    Claude Code (хост)                            │
│                                                                  │
│  ┌──────────┐     tool_use        ┌──────────────────────────┐  │
│  │  Claude   │ ◄────────────────► │  MCP-клиент              │  │
│  │  (LLM)   │  "gmail_search"    │  (встроенный в хост)     │  │
│  └──────────┘                     └────────────┬─────────────┘  │
│                                                │                 │
│                     Твой OAuth-токен Gmail      │                 │
│                     хранится здесь, на          │                 │
│                     платформе Anthropic ────────┤                 │
│                                                │                 │
└────────────────────────────────────────────────┼─────────────────┘
                                                 │ JSON-RPC
                                      ┌──────────▼──────────┐
                                      │  Gmail MCP-сервер   │
                                      │  (Anthropic-managed) │
                                      │  → Gmail API        │
                                      └─────────────────────┘
```

А вот что пытается сделать `sync.py`:

```
┌──────────────────────────────────────────────────────────────────┐
│                    Claude Code (хост)                            │
│                                                                  │
│  ┌──────────┐                     ┌──────────────────────────┐  │
│  │  Claude   │                    │  MCP-клиент              │  │
│  │  (LLM)   │                    │  + OAuth-токен            │  │
│  └──────────┘                     └──────────────────────────┘  │
│                                          ▲                       │
└──────────────────────────────────────────┼───────────────────────┘
                                           │
                                           ✗ нет доступа
                                           │
                              ┌────────────┴───────────┐
                              │  sync.py               │
                              │  (отдельный процесс)   │
                              │  Хочет читать Gmail,   │
                              │  но не может добраться  │
                              │  до MCP-клиента хоста   │
                              └────────────────────────┘
```

Три уровня изоляции, каждый из которых блокирует доступ:

**1. OAuth-токен недоступен.** Когда ты подключил Gmail в Claude, ты авторизовал **приложение Anthropic** в Google. Токен сохранился на серверах Anthropic, привязанный к твоему Claude-аккаунту. Он не лежит у тебя на диске — `sync.py` не может его взять.

**2. MCP-сервер недоступен.** Gmail MCP-сервер — Anthropic-managed (видно по префиксу `claude_ai` в именах tools). Он работает в инфраструктуре Anthropic, а не как локальный процесс на твоей машине. У него нет адреса, к которому `sync.py` мог бы подключиться.

**3. Разные продукты.** Claude Code (подписка) и Anthropic API (developer API с ключом) — разные системы. Даже если `sync.py` вызовет Claude через Anthropic API, это будет **другой контекст** — без твоих MCP-интеграций, без OAuth-токенов, без подключённых серверов.

### Аналогия

Ты авторизовал Gmail в приложении Notion на телефоне. Notion может показывать тебе письма. Но ты не можешь из своего Python-скрипта «попросить Notion» прочитать почту за тебя — Notion не предоставляет такого API, а OAuth-токен хранится внутри Notion.

MCP-подключение в Claude Code — то же самое. Это интеграция внутри конкретного приложения, а не системный сервис, к которому может обратиться любой процесс.

### Что делать

| Задача | Решение |
|---|---|
| Разовая потребность (посмотреть переписку перед созвоном) | Claude Code + Gmail MCP — работает прямо сейчас |
| Автоматическая синхронизация (cron, скрипт) | Прямой Gmail API в `sync.py` — своё OAuth-приложение через Google Cloud Console |
| Интерактивный агент | Claude API + tool_use с Gmail API (ты контролируешь токены) |

### А если взять публичный MCP-сервер?

У Attio и Read.ai есть публичные MCP-серверы — их можно подключить в Claude Code, указав свой API-ключ. Google мог бы сделать такой же для Gmail. Тогда `sync.py` мог бы подключиться к нему как MCP-клиент.

Но `sync.py` уже работает с Attio и Read.ai **напрямую через REST API**, а не через их MCP-серверы:

| Сервис | Claude Code (интерактивно) | sync.py (автоматизация) |
|---|---|---|
| Attio | MCP-сервер | REST API напрямую |
| Read.ai | MCP-сервер | REST API напрямую |
| Gmail | MCP (Anthropic-managed) | → тоже REST API напрямую |

Для Gmail логично сделать то же самое. Вот почему — сравнение кода:

**Вариант A: REST API напрямую (3 строки)**

```python
# sync.py — читаем письма от контакта
gmail = build("gmail", "v1", credentials=creds)

messages = gmail.users().messages().list(
    userId="me",
    q=f"from:{contact_email} OR to:{contact_email}"
).execute()

for msg_id in messages.get("messages", []):
    msg = gmail.users().messages().get(
        userId="me", id=msg_id["id"], format="full"
    ).execute()
    write_email_file(lead_dir, msg)
```

**Вариант B: через MCP-сервер**

MCP-сервер может быть локальным (stdio — дочерний процесс) или удалённым (Streamable HTTP — сервер, хостящийся у Google). Для нашего случая реалистичнее удалённый — если Google хостит MCP-сервер сам:

```python
from mcp import ClientSession
from mcp.client.streamable_http import streamablehttp_client

# Подключиться к удалённому MCP-серверу Google (не дочерний процесс)
async with streamablehttp_client(
    "https://mcp.googleapis.com/gmail",   # гипотетический URL
    headers={"Authorization": f"Bearer {token}"}
) as (read, write, _):
    async with ClientSession(read, write) as session:
        await session.initialize()

        # Вызвать tool через JSON-RPC
        result = await session.call_tool(
            "gmail_search_messages",
            {"q": f"from:{contact_email} OR to:{contact_email}"}
        )
        # result.content — список TextContent, нужно парсить

        for msg_summary in parse_mcp_response(result):
            detail = await session.call_tool(
                "gmail_read_message",
                {"messageId": msg_summary["id"]}
            )
            write_email_file(lead_dir, parse_mcp_response(detail))
```

Что происходит в варианте B под капотом:

```
sync.py                       MCP-сервер Google              Gmail API
  │                              (удалённый)                   (внутри)
  │── POST /mcp ──────────────►│                               │
  │   JSON-RPC: initialize     │                               │
  │◄── ready ─────────────────│                               │
  │── POST /mcp ──────────────►│                               │
  │   JSON-RPC: tools/call     │─── внутренний вызов ────────►│
  │   {gmail_search_messages}  │◄── результат ───────────────│
  │◄── {content: [{text:...}]}│                               │
  │                            │                               │
  │  ... и так для каждого     │                               │
  │      письма                │                               │
```

А в варианте A:

```
sync.py                                               Google
  │                                                      │
  │─── GET /gmail/v1/messages?q=from:... ──────────────►│
  │    Authorization: Bearer                             │
  │◄── JSON ───────────────────────────────────────────│
```

**Итог:** даже с удалённым MCP-сервером (без дочернего процесса) остаётся лишний сетевой hop и JSON-RPC обёртка — ради того же обращения к Gmail API, которое можно сделать напрямую. Плюс OAuth-настройку MCP не отменяет: токен всё равно нужен, просто передаётся через MCP-сервер, а не напрямую.

MCP полезен, когда **LLM решает**, какой tool вызвать. В скрипте с фиксированной логикой («возьми письма от контактов этого лида») LLM не нужен, а значит не нужен и MCP.

### Вывод

MCP — это протокол для AI-агентов внутри хоста, а не универсальная шина, к которой можно подключиться откуда угодно. Если тебе нужен доступ к Gmail из собственного скрипта — ты делаешь **то же самое, что MCP-сервер делает под капотом**: регистрируешь OAuth-приложение, получаешь токен, вызываешь Gmail API. Просто без MCP-обёртки, потому что она тут не нужна.
